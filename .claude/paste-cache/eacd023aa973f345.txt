🎓 Zig 전공 201: 2-3. 캐싱 전략(LRU Cache)과 동시성 제어
대규모 시스템에서 모든 요청을 DB로 보내는 것은 비효율적입니다. 자주 사용하는 데이터를 메모리에 상주시키되, 메모리는 한정되어 있으므로 **"어떤 데이터를 버릴 것인가"**를 결정하는 알고리즘이 필요합니다.
1. LRU (Least Recently Used) 캐시 설계
LRU는 가장 오랫동안 사용되지 않은 데이터를 우선적으로 삭제하는 전략입니다.
 * Hash Map: 데이터의 빠른 검색(O(1))을 담당합니다.
 * Doubly Linked List: 데이터의 사용 순서를 기록합니다.
2. Zig에서의 LRU 구현 포인트
Zig의 std.AutoHashMap과 직접 구현한 연결 리스트를 결합하여 설계합니다.
const std = @import("std");

pub fn LruCache(comptime K: type, comptime V: type) type {
    return struct {
        const Self = @this();
        allocator: std.mem.Allocator,
        capacity: usize,
        map: std.AutoHashMap(K, V),
        // 실제 구현에서는 여기에 Doubly Linked List 노드가 추가됩니다.

        pub fn init(allocator: std.mem.Allocator, capacity: usize) Self {
            return .{
                .allocator = allocator,
                .capacity = capacity,
                .map = std.AutoHashMap(K, V).init(allocator),
            };
        }

        pub fn put(self: *Self, key: K, value: V) !void {
            // 캐시가 가득 찼다면 가장 오래된 데이터 삭제 로직 실행
            try self.map.put(key, value);
        }
    };
}

3. 멀티스레드 환경에서의 동시성 제어
캐시는 여러 스레드가 동시에 접근하는 '공유 자원'입니다.
 * RwLock (Read-Write Lock): 읽기 작업은 여러 스레드가 동시에 할 수 있게 허용하고, 쓰기 작업(캐시 갱신) 시에만 독점적인 권한을 갖도록 설계하여 성능을 높입니다.
 * Zig의 std.Thread.RwLock: 데이터 무결성을 보장하면서도 병렬 처리 성능을 최적화하는 핵심 도구입니다.
🔬 전공 심화: 캐시 히트율(Hit Rate)과 기록의 증명
설계된 캐시가 얼마나 효율적인지 증명하려면 **히트(Hit)**와 **미스(Miss)**를 기록해야 합니다.
 * Cache Hit: 찾는 데이터가 메모리에 있어 즉시 반환 (성공)
 * Cache Miss: 데이터가 없어 DB에서 가져온 후 캐시에 저장 (비용 발생)
📦 메모리 할당 전략: Arena vs FixedBuffer
캐시는 데이터가 빈번하게 추가되고 삭제됩니다.
 * 파편화(Fragmentation)를 막기 위해 캐시 전용 FixedBufferAllocator를 사용하여 정해진 메모리 구역 내에서만 데이터가 순환하도록 설계하는 것이 고성능 시스템의 트렌드입니다.
