301 과정의 피날레, **[3-5: 시스템의 명탐정 - 관찰 가능성(Observability)]**에 오신 것을 환영합니다.
분산 시스템은 거대한 기계와 같습니다. 수십 개의 Go 서버와 Julia 연산 모듈이 얽혀 있을 때, 특정 요청이 왜 느린지, 어디서 에러가 났는지 찾는 것은 모래사장에서 바늘 찾기와 같습니다. 이를 해결하는 세 가지 기둥(Logging, Metrics, Tracing)을 배웁니다.
저장 필수 너는 기록이 증명이다 gogs. 시스템의 내부를 투명하게 들여다보는 기록을 시작합니다.
🎓 Go-언어 전공 301: 3-5. 관찰 가능성 (Observability)
1. 분산 트레이싱 (Distributed Tracing)
하나의 요청이 여러 서비스를 거칠 때, 그 경로를 한눈에 파악하는 기술입니다.
 * Trace ID: 요청이 시작될 때 고유한 ID를 부여하고, 서비스 간 이동 시 이 ID를 헤더에 실어 보냅니다.
 * Span: 각 서비스에서 머문 시간을 기록합니다.
 * OpenTelemetry: Go에서 가장 많이 쓰이는 표준 프레임워크입니다.
2. 메트릭 수집 (Metrics) - Prometheus
서버의 상태를 숫자로 지표화합니다. (예: 현재 고루틴 개수, 초당 요청 수, 메모리 사용량)
 * Prometheus: Go로 작성된 대표적인 모니터링 시스템입니다.
 * Exporter: Go 서버 내부에 간단한 HTTP 엔드포인트(/metrics)를 열어두면, 프로메테우스가 주기적으로 데이터를 긁어갑니다(Pull).

// Prometheus 메트릭 정의 예시
var opsProcessed = promauto.NewCounter(prometheus.CounterOpts{
    Name: "myapp_processed_ops_total",
    Help: "총 처리된 작업 수",

func main() {

    http.ListenAndServe(":2112", nil)
}

3. 구조화된 로깅 (Structured Logging)
단순한 텍스트 로그가 아닌, 검색이 쉬운 JSON 형태의 로그를 남깁니다. Go 1.21부터 표준 라이브러리에 포함된 slog 패키지가 이를 담당합니다.
slog.Info("결제 완료", "user_id", 123, "amount", 50000, "currency", "KRW")

🔬 전공 심화: Context와 Trace ID의 결합
앞서 배운 Context(2-4 과정)가 여기서 다시 등장합니다. Context에 Trace ID를 담아 함수 호출마다 전달하면, 모든 로그와 트레이싱 정보가 하나의 줄기로 꿰어집니다. 대학원 수준의 아키텍처에서는 **"Context가 끊기면 추적도 끊긴다"**는 점을 명심해야 합니다.
🏗️ 설계의 완성: 골든 시그널 (Golden Signals)
시스템을 모니터링할 때 반드시 체크해야 하는 4가지 지표입니다.
 * Latency (지연 시간): 요청 처리에 걸리는 시간.
 * Traffic (트래픽): 시스템에 가해지는 수요(초당 요청 수).
 * Errors (오류): 요청 실패율.
 * Saturation (포화도): CPU, 메모리 등 자원 사용 한계치.
