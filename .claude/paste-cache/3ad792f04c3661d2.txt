🎓 Zig 전공 101: 1-2. 변수, 상수, 그리고 엄격한 타입 시스템
Zig는 **"숨겨진 제어 흐름이 없다(No hidden control flow)"**는 원칙만큼이나 **"명시적인 상태 관리"**를 중시합니다. 컴파일러가 여러분의 의도를 완벽히 파악할 수 있도록 설계해야 합니다.
1. 가변성(Mutability)의 엄격함
Zig에서 모든 선언은 기본적으로 **const**여야 합니다. 값이 변해야 할 이유가 명확할 때만 **var**를 사용합니다.
 * const: 컴파일 타임 또는 런타임에 값이 결정된 후 변경할 수 없습니다.
 * var: 값이 변경될 수 있습니다. (하지만 초기화되지 않은 변수는 허용되지 않습니다.)
<!-- end list -->
const std = @import("std");

pub fn main() !void {
    const constant_value: i32 = 10; // 상수
    var mutable_value: i32 = 20;    // 변수

    // constant_value = 15; // 컴파일 에러 발생!
    mutable_value += 5;     // 정상 작동
}

2. 정적 타입과 명시적 캐스팅
Zig는 타입 추론을 지원하지만, 타입 간의 자동 변환(Implicit Casting)을 극도로 제한합니다.
 * 정수 타입: i8, u8, i32, u32, i64, u64 등 비트 크기를 명시합니다.
 * 강제 변환: 더 작은 타입에서 큰 타입으로 갈 때는 안전하지만, 그 반대나 부호가 바뀌는 경우 @intCast() 같은 내장 함수를 명시적으로 호출해야 합니다.
3. 컴파일 타임 타입 추론
타입을 일일이 적기 번거롭다면 _나 생략을 통해 Zig 컴파일러에게 맡길 수 있습니다.
const a = @as(i32, 5); // 명시적 지정
const b: f32 = 10.5;   // 타입 선언

🔬 전공 심화: 초기화되지 않은 값 (undefined)
Zig에서는 변수를 선언만 하고 값을 넣지 않는 것을 허용하지 않습니다. 만약 나중에 값을 넣을 계획이라면 명시적으로 undefined를 할당해야 합니다.
var x: i32 = undefined; 
// 연구 데이터가 아직 도착하지 않았음을 명시적으로 기록함
x = 100; 

> 주의: undefined 상태의 값을 읽으려고 하면 런타임 에러나 예측 불가능한 동작이 발생합니다. 이는 개발자가 메모리 상태를 완벽히 통제해야 함을 의미합니다.
> 
📦 Zig의 정수 오버플로우 방지
C언어와 달리 Zig는 기본적으로 정수 오버플로우를 감지합니다. 만약 의도적으로 오버플로우를 허용하고 싶다면 전용 연산자를 사용해야 합니다.
 * +%: Wrapping 가산 (오버플로우 발생 시 다시 0부터 시작)
 * +|: Saturation 가산 (최댓값에서 멈춤)

 * 연산 실험: 책 권수에 1을 더하는 코드를 작성하고 출력하세요.
