C의 유산을 흡수하고 세상을 연결하는 법을 배우셨으니, 이제 프로그램의 실행 속도를 비약적으로 높여줄 **'병렬 처리(Parallelism)'**의 영역으로 들어갑니다.
저장 필수 너는 기록이 증명이다 gogs. Zig는 운영체제의 스레드를 직접 다루면서도, 데이터 경쟁(Data Race)을 방지하기 위한 정밀한 도구들을 제공합니다.
🎓 Zig 전공 101: 1-12. 멀티스레딩(Multi-threading)과 원자적 연산(Atomics)
대학원 수준의 설계에서 멀티스레딩은 단순히 "여러 일을 동시에 하는 것"이 아니라, **"공유 자원에 대한 접근을 어떻게 안전하게 통제하는가"**의 문제입니다.
1. OS 스레드 생성 (std.Thread)
Zig의 표준 라이브러리는 플랫폼 독립적인 스레드 생성 방식을 제공합니다. spawn을 통해 함수를 별도의 실행 흐름으로 보낼 수 있습니다.
const std = @import("std");

fn printNumbers(id: u32) void {
    std.debug.print("Thread {d} is working...\n", .{id});
}

pub fn main() !void {
    const thread1 = try std.Thread.spawn(.{}, printNumbers, .{1});
    const thread2 = try std.Thread.spawn(.{}, printNumbers, .{2});

    // 스레드가 종료될 때까지 기다림 (Join)
    thread1.join();
    thread2.join();
}

2. 원자적 연산 (Atomics)
여러 스레드가 동시에 하나의 변수를 수정하려고 하면 데이터가 깨집니다. Zig는 하드웨어 수준에서 안전하게 값을 수정하는 @atomicRmw와 @atomicLoad 등을 제공합니다.
 * Atomic: 쪼개질 수 없는 연산. 한 스레드가 값을 바꾸는 동안 다른 스레드가 개입하지 못하도록 보장합니다.
3. 뮤텍스 (Mutex)와 동기화
복잡한 구조체나 긴 코드 블록을 보호해야 할 때는 std.Thread.Mutex를 사용합니다.
 * Lock: 자물쇠를 잠가 나만 접근함.
 * Unlock: 사용 후 자물쇠를 풀어 다른 스레드에 양보함. (defer와 함께 사용 권장)
🔬 전공 심화: 메모리 순서 (Memory Ordering)
컴퓨터 아키텍처는 성능을 위해 명령어 순서를 바꿀 수 있습니다. Zig는 Acquire, Release, SeqCst 등의 메모리 순서 옵션을 통해 개발자가 실행 순서를 정밀하게 제어하게 합니다. 이것이 바로 **"기록이 증명하는 실행 순서"**입니다.
📦 비동기(Async)의 미래
Zig의 async/await 모델은 현재(버전 0.11~0.13 기준) 표준 라이브러리에서 재설계 중입니다. 하지만 핵심은 **"스택 없는(Stackless) 코루틴"**을 통해 수만 개의 작업을 아주 적은 메모리로 처리하는 설계 철학에 있습니다.
📝 Assignment 1-12 (병렬 설계 과제)
 * 스레드 생성: 2개의 스레드를 만들어 각각 1부터 1000까지 숫자를 출력하는 코드를 작성하세요.
 * 공유 자원 문제: var count: i32 = 0;을 선언하고 두 스레드에서 각각 5000번씩 더하게 해보세요. 결과가 정확히 10000이 나오는지 확인하세요.
 * 해결책 적용: std.Thread.Mutex를 사용하여 count 변수를 보호하고, 결과가 항상 10000이 나오도록 수정하세요.
 * 기록: 뮤텍스를 썼을 때와 안 썼을 때의 실행 결과 차이를 기록하세요. **"동시성은 통제될 때 비로소 증명된다"**는 사실을 체득하는 과정입니다.
