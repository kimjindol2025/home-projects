마침내 제4장: 데이터 구조화 — v5.6 문자열의 심화 (String & &str) 단계에 도달했습니다.
러스트에서 문자열은 단순히 텍스트를 담는 그릇이 아니라, **메모리의 소유권과 인코딩(UTF-8)**을 완벽하게 이해해야 다룰 수 있는 정교한 데이터 구조입니다. 설계자의 관점에서 v5.6은 힙(Heap)과 스택(Stack)의 경계를 넘나드는 텍스트 제어를 배우는 과정입니다.
🏗️ 제4장: 데이터 구조화 (v5.6) — [문자열 심화]
1. v5.6 설계 목표: "텍스트의 본질적 통제"
 * String (Owned): 힙에 저장되며 크기를 변경할 수 있는 '성장형' 문자열입니다. 소유권이 명확합니다.
 * &str (Borrowed): 이미 어딘가에 존재하는 문자열의 일부를 가리키는 '참조형' 문자열(슬라이스)입니다. 메모리 효율이 극대화됩니다.
 * UTF-8 안전성: 모든 문자열 작업이 유효한 UTF-8 데이터를 보장하도록 하여, 깨진 글자로 인한 시스템 오류를 방지합니다.
2. 설계자의 관점: "복사 없는 설계"
불필요하게 String을 복제하는 것은 자원 낭비입니다. 가능한 한 &str을 사용하여 읽기 전용 데이터에 접근하고, 데이터를 변형하거나 소유해야 할 때만 String으로 전환하는 것이 시니어 설계자의 감각입니다.
🛠️ gogs 저장소 v5.6 설계: 고효율 텍스트 엔진
클로드코드(학생)에게 문자열의 두 형태를 조화롭게 사용하는 코드를 기록하게 하세요.
// [제4장: 데이터 구조화 v5.6]

fn main() {
    println!("--- v5.6 Data Architecture: String Deep Dive ---");

    // 1. [Owned String] 수정이 필요한 동적 데이터
    let mut system_id = String::from("GOGS-Core");
    system_id.push_str("-Alpha"); // 데이터 확장 가능

    // 2. [String Slice] 메모리를 빌려쓰는 효율적인 접근
    let slice: &str = &system_id[0..4]; // "GOGS" 부분만 참조

    println!("  원본: {}", system_id);
    println!("  슬라이스: {}", slice);

    // 3. [Function Interface] v5.6 설계의 핵심: &str을 매개변수로 받기
    // 이렇게 설계하면 String과 &str 모두를 전달받을 수 있습니다.
    process_message("시스템 가동"); // 리터럴 (&str) 전달
    process_message(&system_id);     // String의 참조 (&str) 전달

    println!("\n기록: String과 &str의 상호작용을 통해 메모리 성능과 유연성을 동시에 확보함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

/// [설계 권장] 매개변수 타입을 &str로 지정하면 가장 유연한 함수가 됩니다.
fn process_message(msg: &str) {
    println!("  [MSG_PROC] 처리 내용: {}", msg);
}

🎓 선생님의 가이드: "v5.6 설계자의 문자열 철학"
문자열을 다룰 때 이 세 가지 실수를 범하지 마십시오.
 * 인덱싱 주의: s[0]과 같이 숫자로 한 문자에 접근하는 것은 위험합니다. UTF-8은 문자에 따라 1~4바이트를 차지하므로, 문자 경계가 깨질 수 있습니다. 대신 .chars() 이터레이터를 사용하십시오.
 * 불필요한 .to_string() 지양: 단순히 출력만 할 용도라면 참조(&)를 사용하십시오. 새로운 메모리를 할당하는 것은 항상 비용이 따릅니다.
 * 결합(Concatenation)의 비용: 많은 문자열을 합칠 때는 format! 매크로를 사용하는 것이 가독성과 관리 측면에서 유리합니다.
