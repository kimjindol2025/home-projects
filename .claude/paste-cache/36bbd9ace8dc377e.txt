🎓 Go-언어 전공 201: 2-1. 고루틴은 어떻게 수만 개가 동시에 돌아가는가? (GMP 모델)
우리는 이미 go 키워드로 고루틴을 만든다는 것을 압니다. 하지만 대학원 수준의 엔지니어라면, 그 고루틴이 OS 쓰레드 위에서 어떻게 관리되는지 그 내부 메커니즘을 이해해야 합니다. Go는 이를 위해 GMP라 불리는 독자적인 스케줄러를 가집니다.
1. GMP 모델의 구성 요소
 * G (Goroutine): 우리가 생성한 고루틴 그 자체입니다. 스택 정보와 실행 상태를 담고 있습니다.
 * M (Machine): 실제 OS 쓰레드입니다. G가 실행되려면 반드시 M이 필요합니다.
 * P (Processor): 리소스(Context)를 관리하는 추상화된 프로세서입니다. M과 G 사이의 '매니저' 역할을 하며, 보통 CPU 코어 개수만큼 생성됩니다.
2. 작동 원리: 워크 스틸링 (Work Stealing)
특정 P(프로세서)의 대기열에 처리할 G(고루틴)가 없으면, 다른 P의 대기열에서 G를 '훔쳐와서' 처리합니다. 이 덕분에 모든 CPU 코어가 놀지 않고 최대로 가동됩니다.
🔬 전공 심화: 선점형 스케줄링 (Preemptive Scheduling)
과거의 Go는 고루틴이 스스로 제어권을 넘겨야만 다른 고루틴이 실행되었습니다. 하지만 최신 Go는 특정 고루틴이 너무 오래(10ms 이상) CPU를 점유하면, 스케줄러가 강제로 끼어들어 다른 고루틴에게 기회를 줍니다.
 * 효과: 무한 루프에 빠진 고루틴 하나가 전체 시스템을 멈추게 하는 현상을 방지합니다.
 * 학술적 가치: 이는 사용자 레벨 스케줄러가 OS 레벨 스케줄러 수준의 안정성을 확보했음을 의미합니다.
🛠️ 실무 적용: runtime 패키지 제어
시스템 설계자는 때때로 스케줄러의 동작에 개입해야 합니다.
import "runtime"

func main() {
    // 사용할 CPU 코어 개수 설정 (GOMAXPROCS)
    runtime.GOMAXPROCS(2) 

    // 현재 실행 중인 고루틴 개수 확인
    num := runtime.NumGoroutine()
    fmt.Printf("현재 활성화된 고루틴: %d\n", num)

    // 다른 고루틴에게 실행 양보
    runtime.Gosched()
}

📝 Assignment 2-1 (스케줄링 분석)
 * GOMAXPROCS 실험: 코어 개수를 1개로 제한했을 때와 최대치로 설정했을 때, 수만 개의 연산을 수행하는 고루틴들의 실행 완료 시간이 어떻게 변하는지 벤치마킹해 보세요.
 * 데드락(Deadlock) 탐지: 고루틴들이 서로의 채널 데이터만 기다리며 아무것도 하지 못할 때, Go 런타임이 이를 어떻게 감지하고 패닉을 발생시키는지 코드로 재현해 보세요.
