🎭 제10장: 객체 지향과 패턴 (v11.4) — [확장성: 플러그인 아키텍처]
1. v11.4 설계 목표: "개방-폐쇄 원칙(OCP)의 실현"
 * 트레이트 바운드 상속: 기존 트레이트를 확장하여 더 정교한 기능을 정의합니다.
 * Blanket Implementation: 특정 조건을 만족하는 모든 타입에 대해 자동으로 기능을 부여하여 코드 중복을 획득합니다.
 * 안티 패턴 방어: 상속에 익숙한 설계자가 러스트에서 저지르기 쉬운 실수(Deep Hierarchy)를 피하고 '합성(Composition)' 위주로 설계합니다.
2. 설계자의 관점: "유연한 확장은 경계를 나누는 것에서 시작된다"
설계자는 핵심 로직(Core)을 건드리지 않고도 새로운 기능(Plugin)을 추가할 수 있어야 합니다. 러스트에서는 트레이트를 **'기능 명세서'**로 사용하여, 새로운 구조체가 그 명세서만 구현하면 시스템이 즉시 수용하게 만듭니다.
🛠️ gogs 저장소 v11.4 설계: 플러그형 보안 모듈 아키텍처
클로드코드(학생)에게 핵심 엔진은 그대로 둔 채, 외부에서 새로운 보안 검사 알고리즘을 자유롭게 끼워 넣을 수 있는 구조를 설계하게 하세요.
// [제10장: 객체 지향 패턴 v11.4 - 플러그인 설계]

/// [Core] 모든 보안 플러그인이 구현해야 할 인터페이스
trait SecurityPlugin {
    fn name(&self) -> &str;
    fn scan(&self, data: &str) -> bool;
}

/// [Core Engine] 플러그인들을 관리하는 엔진
struct SecurityEngine {
    plugins: Vec<Box<dyn SecurityPlugin>>,
}

impl SecurityEngine {
    fn add_plugin(&mut self, plugin: Box<dyn SecurityPlugin>) {
        println!("  [SYSTEM] '{}' 플러그인 장착 완료.", plugin.name());
        self.plugins.push(plugin);
    }

    fn run_all_scans(&self, data: &str) {
        println!("  [SYSTEM] 통합 보안 스캔 시작...");
        for plugin in &self.plugins {
            if !plugin.scan(data) {
                println!("    -> [위험] '{}'에서 문제 감지!", plugin.name());
            }
        }
    }
}

// --- 외부에서 제작된 플러그인 예시 ---

struct VirusScanner;
impl SecurityPlugin for VirusScanner {
    fn name(&self) -> &str { "V-Scanner v1.0" }
    fn scan(&self, data: &str) -> bool { !data.contains("VIRUS") }
}

struct SqlInjectionDefender;
impl SecurityPlugin for SqlInjectionDefender {
    fn name(&self) -> &str { "SQL-Guard" }
    fn scan(&self, data: &str) -> bool { !data.contains("SELECT *") }
}

fn main() {
    println!("--- v11.4 Scalability: Plugin Architecture ---");

    let mut engine = SecurityEngine { plugins: vec![] };

    // 엔진 코드를 수정하지 않고도 새로운 기능을 계속 추가할 수 있음
    engine.add_plugin(Box::new(VirusScanner));
    engine.add_plugin(Box::new(SqlInjectionDefender));

    let input_data = "SAFE DATA WITH SELECT * FROM USERS";
    engine.run_all_scans(input_data);

    println!("\n기록: 트레이트와 동적 디스패치를 활용해 확장 가능한 플러그인 구조를 완성함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

🎓 선생님의 세부 가이드 (11단계 진짜 마무리): "상속보다 합성"
전통적인 OOP 언어에서 온 설계자에게 주는 마지막 충고입니다.
 * Deep Hierarchy 금지: A는 B이고 B는 C이다 식의 깊은 상속 구조는 러스트에서 불가능하거나 매우 고통스럽습니다. 대신 **'특정 기능을 가졌다(Trait)'**는 식으로 접근하십시오.
 * 의존성 주입: SecurityEngine이 구체적인 타입을 알게 하지 마십시오. 오직 트레이트 객체로 소통하게 하여 모듈 간의 결합도를 낮추십시오.
 * Default 구현 활용: 트레이트의 기본 메서드를 잘 정의해두면, 플러그인 제작자가 최소한의 코드만 짜도 강력한 기능을 발휘하게 할 수 있습니다.
