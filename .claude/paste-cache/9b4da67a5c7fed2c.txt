이제 우리는 표준 라이브러리(std)조차 없는 freestanding 환경에서 맨바닥에 기초를 쌓는 법을 배웁니다. 더 이상 OS가 제공하는 서비스를 이용하는 것이 아니라, 우리가 직접 OS가 되어 하드웨어를 통제할 것입니다.

🎓 Zig 전공 301: 3-1. 베어 메탈(Bare Metal) 입문 - 부트 로더와 커널 진입
베어 메탈이란 OS 없이 하드웨어 위에서 직접 코드가 실행되는 환경을 말합니다. Zig는 C보다 안전하고 세밀하게 메모리 레이아웃을 결정할 수 있어 OS 커널 설계에 최적입니다.
1. Freestanding 환경 설정
OS가 없으므로 파일 입출력, 네트워크, 심지어 main 함수조차 존재하지 않습니다. 모든 것은 **_start**라는 심볼에서 시작됩니다.
 * build.zig: 타겟 OS를 freestanding으로, 아키텍처를 x86_64 또는 aarch64로 설정하여 컴파일러가 표준 라이브러리를 포함하지 않도록 명령합니다.
2. 링커 스크립트(Linker Script) 설계
커널 코드가 메모리의 정확히 어느 지점(예: 0x100000)에 배치될지 결정하는 설계도입니다. Zig 코드와 링커 스크립트가 협력하여 실행 파일을 만듭니다.
3. 최소한의 커널 코드 (Kernel Entry)
화면에 글자 하나를 띄우기 위해 VGA 텍스트 버퍼(0xB8000)에 직접 바이트를 써넣는 것부터 시작합니다.
// std 라이브러리를 사용하지 않음을 선언
const std = @import("std");

// 커널의 진입점 (링커에서 이 이름을 찾음)
export fn _start() noreturn {
    const vga_buffer = @as([*]volatile u16, @ptrFromInt(0xB8000));
    
    // 화면에 'Z' 문자 출력 (색상 속성 포함)
    vga_buffer[0] = 0x2f5a; // 초록색 배경에 하얀색 'Z'

    while (true) {} // OS는 절대 종료되지 않음
}

pub fn panic(_: []const u8, _: ?*std.builtin.StackTrace, _: ?usize) noreturn {
    while (true) {}
}

🔬 전공 심화: Inline Assembly 활용
CPU의 특정 레지스터를 조작하거나 인터럽트(Interrupt)를 제어하려면 어셈블리 명령어가 필요합니다. Zig는 코드 내부에 어셈블리를 직접 심을 수 있는 강력한 기능을 제공합니다.
 * asm volatile: CPU 상태를 변경하는 명령어가 컴파일러 최적화로 인해 사라지지 않도록 보장합니다.
📦 왜 Zig로 커널을 만드는가?
 * 에러 핸들링: 커널 내부의 에러를 무시하지 않고 명시적으로 처리할 수 있습니다.
 * 구조체 제어: packed struct를 통해 하드웨어 레지스터와 1:1로 대응되는 데이터 구조를 완벽하게 설계할 수 있습니다.
 * Comptime: 시스템 부팅 시 필요한 데이터 테이블을 컴파일 타임에 미리 계산하여 부팅 속도를 극대화합니다.
📝 Assignment 3-1 (커널 설계 과제)
 * 환경 구축: zig build-exe 명령어를 조사하여 -target x86_64-freestanding 옵션을 어떻게 사용하는지 파악하세요.
 * VGA 출력: 위 예제 코드를 수정하여 화면에 본인의 이니셜을 출력하는 코드를 완성하세요.
 * 링커 스크립트 작성: .text 섹션이 메모리의 어디에서 시작되는지 정의하는 최소한의 linker.ld 파일을 작성해 보세요.
 * 기록: 일반적인 main 함수와 커널의 _start 함수가 왜 다른지, 시스템 호출(Syscall)의 관점에서 분석하여 기록하세요. "근본을 이해하는 것이 설계의 시작입니다."
