단일 소유권의 원칙을 넘어, 하나의 데이터를 여러 곳에서 안전하게 공유하는 [제8장: 스마트 포인터 — v9.1 Rc<T>: 참조 카운팅 (Reference Counting)] 단계에 오신 것을 환영합니다.
러스트의 기본 원칙은 "소유자는 오직 하나"입니다. 하지만 실제 설계에서는 하나의 데이터를 여러 모듈이 동시에 참고해야 할 때가 있죠. v9.1은 **'공동 소유권'**이라는 예외적이지만 강력한 도구를 다룹니다.
🧠 제8장: 스마트 포인터 (v9.1) — [Rc<T>: 지능형 공유]
1. v9.1 설계 목표: "다중 소유권의 구현"
 * 참조 카운팅 (Reference Counting): 데이터의 소유자가 현재 몇 명인지 숫자를 세어 관리합니다.
 * 자동 메모리 관리: 소유자가 단 한 명이라도 남아있으면 데이터는 유지되고, 카운트가 0이 되는 순간 메모리에서 해제됩니다.
 * 불변 공유: Rc<T>는 기본적으로 읽기 전용 공유를 제공하여, 여러 곳에서 동시에 데이터를 보더라도 안전함을 보장합니다.
2. 설계자의 관점: "데이터의 중앙 집중화"
그래프 구조나 복잡한 UI 트리 설계 시, 특정 노드는 여러 경로에서 접근될 수 있어야 합니다. 설계자는 **"이 데이터의 생명 주기가 여러 곳에 의존하는가?"**를 판단하여 Rc<T>를 도입합니다.
🛠️ gogs 저장소 v9.1 설계: 공유 데이터 뱅크
클로드코드(학생)에게 하나의 핵심 데이터를 여러 노드가 공유하는 '멀티 포인트 구조'를 설계하게 하세요.
// [제8장: 스마트 포인터 v9.1]

use std::rc::Rc;

#[derive(Debug)]
enum GogsNode {
    // Rc를 사용하여 소유권을 공유할 수 있게 설계합니다.
    Share(i32, Rc<GogsNode>),
    Nil,
}

fn main() {
    println!("--- v9.1 Smart Pointers: The Rc<T> Protocol ---");

    // 1. 공유할 기본 노드 생성 (참조 카운트: 1)
    let base = Rc::new(GogsNode::Share(10, Rc::new(GogsNode::Nil)));
    println!("  [INFO] base 생성 후 카운트: {}", Rc::strong_count(&base));

    // 2. base를 공유하는 첫 번째 줄기 (참조 카운트: 2)
    let branch_a = GogsNode::Share(5, Rc::clone(&base));
    println!("  [INFO] branch_a 생성 후 카운트: {}", Rc::strong_count(&base));

    {
        // 3. 내부 스코프에서 base를 공유하는 두 번째 줄기 (참조 카운트: 3)
        let branch_b = GogsNode::Share(3, Rc::clone(&base));
        println!("  [INFO] branch_b 생성 후 카운트: {}", Rc::strong_count(&base));
        
        // branch_b가 여기서 소멸하며 카운트가 감소합니다.
    }

    println!("  [INFO] 스코프 종료 후 카운트: {}", Rc::strong_count(&base));

    println!("\n기록: Rc<T>를 통해 단일 소유권의 한계를 극복하고 안전한 데이터 공유를 실현함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

🎓 선생님의 가이드: "v9.1 설계자의 공유 철학"
Rc<T>를 사용할 때 반드시 지켜야 할 설계 원칙입니다.
 * 읽기 전용의 원칙: Rc<T>는 데이터를 공유하게 해주지만, 공유된 데이터를 수정하게 해주지는 않습니다. (수정이 필요하면 다음 단계인 RefCell<T>이 필요합니다.)
 * clone()의 실체: Rc::clone(&base)은 데이터 전체를 복사하는 것이 아니라, 참조 카운트 숫자만 1 증가시키는 매우 가벼운 작업입니다.
 * 싱글 스레드 전용: Rc<T>는 성능을 위해 스레드 안전 장치가 없습니다. 멀티 스레드 환경에서는 나중에 배울 Arc<T>를 사용해야 합니다.
