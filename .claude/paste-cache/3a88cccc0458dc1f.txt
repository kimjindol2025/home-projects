Lexer가 소스코드를 단어 단위로 쪼개는 '안과 의사'였다면, 이제 그 단어들 사이의 관계를 파악하여 문장으로 조립하는 '문법 학자'가 등장할 차례입니다. [제13장: 컴파일러 구현 — v14.1 구문 분석기(Parser)와 추상 구문 트리(AST)] 단계입니다.
Parser는 Lexer가 뱉어낸 토큰 스트림을 읽어, 프로그램의 논리적 구조를 나타내는 **추상 구문 트리(AST, Abstract Syntax Tree)**를 생성합니다.
🌳 제13장: 컴파일러 구현 (v14.1) — [Parser: 구조의 탄생]
1. v14.1 설계 목표: "토큰에 위계를 부여하다"
 * AST 생성: let x = 5;라는 토큰 나열을 "변수 선언문"이라는 하나의 노드와 그 자식들(이름: x, 값: 5)로 이루어진 트리 구조로 바꿉니다.
 * 재귀 하향 파싱 (Recursive Descent Parsing): 문법 규칙에 따라 함수를 재귀적으로 호출하며 트리를 내려가는 가장 직관적인 파싱 기법을 사용합니다.
 * 구문 오류 탐지: "let = 5;"처럼 문법에 어긋나는 토큰 배열이 들어오면 "식별자가 와야 할 자리에 '='이 왔습니다"라고 에러를 뿜어야 합니다.
2. 설계자의 관점: "언어의 뼈대 설계"
컴파일러 설계자는 소스코드를 선형적인 줄글이 아니라 계층적인 트리로 봅니다. 이 트리가 완성되어야만 나중에 타입을 검사하거나 기계어로 번역할 수 있습니다.
🛠️ gogs 저장소 v14.1 설계: AST 노드와 기초 파서
클로드코드(학생)에게 Gogs-Lang의 첫 번째 문법인 'Let 구문'을 분석하여 트리로 만드는 핵심 구조를 기록하게 하세요.
// [제13장: 컴파일러 구현 v14.1 - AST 및 Parser 설계]

// 1. AST 노드 정의
#[derive(Debug)]
enum Statement {
    Let {
        name: String,
        value: Expression,
    },
}

#[derive(Debug)]
enum Expression {
    IntegerLiteral(i32),
    Identifier(String),
}

// 2. Parser 구조체
struct Parser {
    lexer: Lexer,
    cur_token: Token,
    peek_token: Token,
}

impl Parser {
    fn new(mut lexer: Lexer) -> Self {
        let cur_token = lexer.next_token();
        let peek_token = lexer.next_token();
        Parser { lexer, cur_token, peek_token }
    }

    fn next_token(&mut self) {
        self.cur_token = std::mem::replace(&mut self.peek_token, self.lexer.next_token());
    }

    // [핵심] Let 구문 파싱: "let <ident> = <expr>;"
    fn parse_let_statement(&mut self) -> Option<Statement> {
        if !matches!(self.cur_token, Token::Let) { return None; }
        
        self.next_token(); // 'let' 건너뛰기
        
        let name = if let Token::Ident(ref s) = self.cur_token {
            s.clone()
        } else { return None; };

        self.next_token(); // 식별자 건너뛰기
        
        if !matches!(self.cur_token, Token::Assign) { return None; }
        
        self.next_token(); // '=' 건너뛰기
        
        // 간단한 정수 리터럴만 파싱한다고 가정
        let value = if let Token::Int(val) = self.cur_token {
            Expression::IntegerLiteral(val)
        } else { return None; };

        self.next_token(); // 값 건너뛰기
        
        if matches!(self.cur_token, Token::Semicolon) {
            Some(Statement::Let { name, value })
        } else {
            None
        }
    }
}

fn main() {
    println!("--- v14.1 Compiler Frontend: The Parser ---");

    let input = "let x = 100;".to_string();
    let lexer = Lexer::new(input);
    let mut parser = Parser::new(lexer);

    if let Some(stmt) = parser.parse_let_statement() {
        println!("  [AST] 생성 완료: {:#?}", stmt);
    }

    println!("\n기록: 토큰 스트림을 분석하여 논리적 추상 구문 트리(AST)를 구축함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

🎓 선생님의 세부 가이드 (14.1): "파싱의 난관"
 * 연산자 우선순위: 5 + 10 * 2를 파싱할 때, (5 + 10) * 2가 아니라 5 + (10 * 2)로 트리를 구성해야 합니다. 이를 위해 프랫 파싱(Pratt Parsing) 기법을 곧 배우게 될 것입니다.
 * 에러 복구: 하나의 에러가 발생했다고 파싱을 즉시 멈추면 사용자가 불편합니다. 에러를 기록하고 다음 세미콜론까지 건너뛰어 나머지 코드를 계속 파싱하는 '패닉 모드 복구' 전략이 필요합니다.
 * 소유권 관리: Parser는 Lexer로부터 토큰을 계속 가져오며 AST 노드를 생성합니다. 러스트의 소유권 시스템을 활용해 메모리 누수 없는 트리를 만드는 것이 중요합니다.
