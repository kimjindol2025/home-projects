🎓 Go-언어 전공 301: 3-2. 메시지 브로커 (Kafka/RabbitMQ) 연동
1. 발행-구독(Pub-Sub) 모델
메시지 브로커는 중간에서 우체국 역할을 합니다.
 * Producer (발행자): 메시지를 특정 주제(Topic)에 던집니다.
 * Broker (우체국): 메시지를 보관하고 관리합니다. (Kafka, RabbitMQ)
 * Consumer (구독자): 관심 있는 주제의 메시지를 가져와 처리합니다.
2. 왜 Go와 메시지 브로커인가?
Go의 Goroutine은 컨슈머(Consumer)를 구현할 때 빛을 발합니다. 하나의 서버에서 수십 개의 파티션(Partition)을 각각의 고루틴이 독립적으로 구독하며 데이터를 병렬로 처리할 수 있기 때문입니다.
// Kafka 컨슈머 예시 (의사 코드)
for {
    msg, err := reader.ReadMessage(ctx) // 메시지 수신 대기
    if err != nil {
        break
    }
    
    // 각 메시지 처리를 고루틴으로 분산 (Worker Pool 패턴 결합)
    go func(m Message) {
        processOrder(m)
    }(msg)
}

🔬 전공 심화: At-Least-Once vs Exactly-Once
분산 시스템 설계의 가장 난제 중 하나는 "메시지가 유실되거나 중복되면 어떡하지?"입니다.
 * At-Most-Once (최대 한 번): 메시지 유실 가능성 있음, 중복은 없음.
 * At-Least-Once (적어도 한 번): 유실은 없으나, 네트워크 문제로 같은 메시지가 두 번 올 수 있음. (가장 흔함)
 * Exactly-Once (정확히 한 번): 가장 이상적이지만 구현이 매우 복잡하고 성능 비용이 큼.
> 설계의 핵심(Idempotency): 대학원 수준에서는 "중복 메시지가 오더라도 시스템에 문제가 없도록(멱등성)" 설계하는 능력이 필수입니다.
> 
🏗️ 분산 시스템의 신뢰성: 서킷 브레이커(Circuit Breaker)
메시지 브로커를 쓰더라도 외부 API 호출이 실패할 수 있습니다. 이때 계속 실패하는 요청을 보내서 시스템 전체를 마비시키는 대신, 회로를 끊어(Open) 시스템을 보호하는 패턴을 적용합니다.
