당신은 Rust 컴파일러 팀 출신 + 15년차 시스템 언어 설계 전문가입니다. Rust 1.0(2015)부터 Rust 1.80\~1.90대(2025\~2026)까지의 실제 발전 과정을 모두 알고 있으며, Rust가 어떤 순서로 어떤 기능을 안정화했는지 정확히 기억합니다.

지금 내가 만든 언어(또는 프로토타입)의 버전별 구현 상태를 Rust와 비교해서 **v1 \~ v6 전 버전을 전수검사**해 주세요.
Rust 기준으로 "이 시점에서 이미 안정화된 기능", "아직 불안정하거나 nightly였던 기능", "절대 넣지 말아야 했던 과도 설계" 등을 냉정하게 평가합니다.

현재 내 언어 버전별 요약 (여기 실제로 당신이 채워넣으세요 - 예시):
- v1: 변수, 기본 타입(int/str/bool), 사칙연산, if/while, print → tree-walking 인터프리터
- v2: 함수 정의 + return, 기본 스코프(지역/전역), 리스트 기본
- v3: dict + for-each, 간단 예외(try/except 기본), ref counting GC
- v4: 모듈 import 기본, 파일 입출력, 문자열 메서드 일부
- v5: 클로저 + lambda, 기본 클래스/메서드, 패닉! 비슷한 unwind
- v6: async/await 시도 구현, 제네릭 기본, borrow checker 비슷한 소유권 검사 시작

Rust를 기준 삼아 아래 구조로 **매우 구체적이고 냉정하게** 분석해 주세요.
각 버전(v1\~v6)마다 Rust의 해당 시기(1.0\~1.10대, 1.20대 등)와 비교합니다.

<필수 분석 구조 - 이 순서 그대로>

1. Rust 역사적 타임라인 요약 (v1\~v6에 대응되는 Rust 시기 매핑)
   - Rust 1.0 \~ 1.10대: 어떤 기능이 안정화됐나? (borrow checker, lifetime, impl Trait 등)
   - Rust 1.15 \~ 1.30대: 2018 edition 전후 큰 변화
   - Rust 1.40대 이후: async 완성, const generics 등

2. 버전별 부족한 점 Top 5 (Rust가 이미 갖춘/필수로 여기는 기능 중 없는 것)
   - 예: "v3에서 borrow checker / lifetime 없음 → Rust 1.0부터 핵심이었음"
   - "v5에서 panic unwind + Result<T,E> 패턴 미지원 → Rust 초기부터 안전성의 핵심"
   - 실무 영향: "이게 없으면 어떤 버그/고통이 발생하는지" 구체 사례

3. 버전별 과한 점 / 오버엔지니어링 Top 3 (Rust가 의도적으로 늦게 넣거나 아예 안 넣은 것)
   - 예: "v5에서 async/await 먼저 넣음 → Rust는 1.39(2019)까지도 nightly, 안정화 매우 신중"
   - "v6에서 제네릭 + trait 강제 → Rust는 trait object dyn + impl Trait로 점진적 안정화"
   - 위험성: "이걸 너무 일찍 넣으면 생태계 파편화 / 컴파일 타임 폭발 / 사용자 혼란 유발 가능성"

4. 전체 v1\~v6 종합 평가
   - Rust의 발전 속도 대비 내 언어의 "현재 위치" (예: Rust 1.0 직전? 1.10대? 2018 edition 직전?)
   - 가장 급하게 채워야 할 부족 기능 TOP 5 (우선순위 순)
   - 가장 위험하거나 제거/후퇴 고려할 과한 기능 TOP 3
