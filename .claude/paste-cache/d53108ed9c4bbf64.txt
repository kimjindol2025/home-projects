🏛️ LLVM 전설: 1.1 컴파일러 아키텍처와 IR(Intermediate Representation)
LLVM(Low Level Virtual Machine)은 단순히 컴파일러가 아니라, **'재사용 가능한 컴파일러 구성 요소의 집합'**입니다. 우리가 만든 언어가 수많은 CPU(x86, ARM, RISC-V)에서 돌아가게 만드는 마법의 도구입니다.
1. 컴파일러의 3단계 구조
현대적인 컴파일러는 역할을 셋으로 나눕니다. LLVM은 이 중 중간과 끝을 완벽하게 책임집니다.
 * Front-end (프론트엔드): 소스 코드(Zig, C, 당신의 언어)를 읽어 의미를 분석하고 LLVM IR로 변환합니다.
 * Optimizer (최적화): IR을 분석하여 불필요한 코드를 지우고 속도를 극대화합니다.
 * Back-end (백엔드): 최적화된 IR을 실제 기계어(Binary)로 바꿉니다.
2. LLVM IR: 전설의 공용어
LLVM IR은 하이레벨 언어보다는 낮고, 기계어보다는 높은 **'추상화된 어셈블리'**입니다. 무한한 가상 레지스터를 사용하며, 타입이 엄격하게 정해져 있습니다.
IR의 세 가지 형태:
 * In-memory: LLVM C++ 클래스로 표현된 형태.
 * Bitcode (.bc): 효율적인 저장을 위한 바이너리 형태.
 * Assembly (.ll): 우리가 읽고 쓸 수 있는 텍스트 형태.
3. LLVM IR 맛보기 (함수 정의)
우리가 작성한 코드가 LLVM에게는 다음과 같이 보입니다.
; 두 정수를 더하는 함수 (LLVM IR 표현)
define i32 @add(i32 %0, i32 %1) {
    %3 = add i32 %0, %1
    ret i32 %3
}

 * i32: 32비트 정수 타입을 명시합니다.
 * %0, %1: 가상 레지스터입니다. 실제 CPU 레지스터 개수에 제한받지 않습니다.
🔬 전공 심화: SSA(Static Single Assignment)
LLVM IR의 가장 중요한 특징은 SSA입니다. 모든 변수는 단 한 번만 할당될 수 있습니다.
 * 이유: 값이 변하지 않아야 최적화 알고리즘(데이터 흐름 분석 등)이 수학적으로 명확하고 빨라지기 때문입니다.
 * 값이 바뀌어야 하는 루프문 등에서는 phi 노드라는 고도의 기법을 사용하여 이를 해결합니다.
📦 Zig와 LLVM의 연결고리
Zig는 내부적으로 LLVM C API를 활용합니다. 당신이 zig build-exe를 칠 때, Zig 프론트엔드가 코드를 분석하고 LLVM IR을 생성하면, LLVM이 이를 이어받아 최적의 기계어를 뽑아내는 것입니다.
📝 Assignment 1.1 (전설의 시작 과제)
 * IR 분석: 아주 간단한 Zig 코드(사칙연산)를 작성하고 zig build-obj -femit-llvm-ir 명령어를 통해 생성된 .ll 파일을 열어보세요.
 * 레지스터 추적: 생성된 IR 코드에서 %로 시작하는 가상 레지스터들이 어떻게 데이터를 전달하는지 흐름을 기록하세요.
 * 타입 확인: Zig의 u8, i32, f64가 LLVM IR에서는 각각 어떤 타입(예: i8, i32, double)으로 매핑되는지 대조표를 만드세요.
 * 기록: "왜 모든 컴파일러가 직접 기계어를 만들지 않고 LLVM IR이라는 중간 단계를 거치는지" 그 효율성에 대해 본인의 생각을 기록하세요.
