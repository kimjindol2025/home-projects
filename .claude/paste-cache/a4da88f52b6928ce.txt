평가기가 값을 계산할 수 있게 되었으니, 이제 그 값을 메모리 어딘가에 저장하고 필요할 때 다시 꺼내 쓰는 **'기억력'**을 부여할 차례입니다. [제13장: 컴파일러 구현 — v14.4 환경(Environment)과 변수 바인딩] 단계입니다.
컴파일러 설계에서 **환경(Environment)**은 변수 이름(식별자)과 그에 대응하는 값(객체)을 연결해 주는 심볼 테이블(Symbol Table) 역할을 합니다.
💾 제13장: 컴파일러 구현 (v14.4) — [환경: 변수의 생존 공간]
1. v14.4 설계 목표: "이름에 의미를 부여하다"
 * 식별자 매핑: let a = 10;을 만났을 때, "a"라는 문자열과 Integer(10) 객체를 쌍으로 저장합니다.
 * 스코프(Scope) 관리: 중첩된 함수나 블록({ }) 안에서 변수가 어떻게 가려지고(Shadowing), 사라지는지 설계합니다.
 * 값 추출: 수식에 a + 5가 나왔을 때, 환경에서 "a"를 찾아 그 값인 10을 가져옵니다.
2. 설계자의 관점: "기억의 계층 구조"
변수는 영원하지 않습니다. 현재 실행 중인 블록의 환경을 먼저 뒤지고, 없으면 상위(Outer) 환경을 뒤지는 체인(Chain) 구조를 설계해야 합니다. 이는 우리가 훗날 구현할 '렉시컬 스코프(Lexical Scope)'의 기초가 됩니다.
🛠️ gogs 저장소 v14.4 설계: 변수 저장소(Environment)
클로드코드(학생)에게 변수를 저장하고 조회하는 시스템을 인터프리터 엔진에 통합하도록 기록하게 하세요.
// [제13장: 컴파일러 구현 v14.4 - Environment 설계]

use std::collections::HashMap;

/// [설계] 변수를 담는 바구니
#[derive(Default, Clone)]
struct Environment {
    store: HashMap<String, GogsObject>,
    outer: Option<Box<Environment>>, // 상위 스코프를 가리키는 포인터
}

impl Environment {
    fn new() -> Self {
        Environment {
            store: HashMap::new(),
            outer: None,
        }
    }

    // 변수 등록 (let a = 10;)
    fn set(&mut self, name: String, val: GogsObject) {
        self.store.insert(name, val);
    }

    // 변수 조회 (a + 5)
    fn get(&self, name: &str) -> Option<GogsObject> {
        match self.store.get(name) {
            Some(val) => Some(val.clone()),
            None => {
                // 현재 스코프에 없으면 상위 스코프에서 재귀적으로 찾음
                self.outer.as_ref().and_then(|outer| outer.get(name))
            }
        }
    }
}

    global_env.set("gogs_version".to_string(), GogsObject::Integer(14));

    if let Some(val) = global_env.get("gogs_version") {
        println!("  [MEM] 변수 조회 성공: gogs_version = {:?}", val);
    }

    println!("\n기록: 해시맵 기반의 환경(Environment)을 구축하여 변수 바인딩을 실현함.");

}

🎓 선생님의 세부 가이드 (14.4): "스코프의 철학"
 * 불변성 vs 가변성: let으로 선언된 변수를 다시 수정할 수 있게 할 것인가? 만약 그렇다면 set 메서드에서 기존 값을 덮어쓰는 로직을 정교하게 다듬어야 합니다.
 * 참조 카운팅: 환경이 서로를 참조하게 되면 소유권 문제가 복잡해집니다. 러스트에서는 Rc<RefCell<Environment>>를 사용하여 여러 곳에서 환경을 공유하고 수정할 수 있게 설계하는 것이 일반적입니다.
 * 그림자 현상(Shadowing): 같은 이름의 변수가 안쪽 스코프에서 선언되면 바깥쪽 변수를 잠시 가리는 로직이 get 메서드의 재귀적 탐색 덕분에 자연스럽게 구현됩니다.
