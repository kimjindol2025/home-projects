🎓 Go-언어 전공 301: 3-1. gRPC와 효율적인 데이터 통신
지금까지는 HTTP/JSON 기반의 REST API를 다뤘습니다. 하지만 수천 개의 서비스가 연결된 분산 시스템(MSA)에서는 JSON의 텍스트 데이터는 너무 무겁고 느립니다. 구글이 만든 gRPC는 이를 해결하기 위한 정밀한 통신 규격입니다.
1. Protocol Buffers (Protobuf)
데이터를 주고받을 때 텍스트(JSON)가 아닌 바이너리(이진 데이터) 형식을 사용합니다.
 * 훨씬 작고(약 3~10배), 훨씬 빠릅니다.
 * .proto 파일에 데이터 구조를 엄격하게 정의하여 서버와 클라이언트 간의 '계약'을 맺습니다.
<!-- end list -->
// user.proto
syntax = "proto3";

service UserService {
  rpc GetUser (UserRequest) returns (UserResponse);
}

message UserRequest {
  int32 id = 1;
}

message UserResponse {
  string name = 1;
  string email = 2;
}

2. gRPC의 4가지 통신 방식
대학원 수준에서는 단순한 요청-응답을 넘어 스트리밍을 이해해야 합니다.
 * Unary: 일반적인 1:1 요청/응답.
 * Server Streaming: 클라이언트가 요청하면 서버가 지속적으로 데이터를 보냄 (예: 실시간 주가 알림).
 * Client Streaming: 클라이언트가 대량의 데이터를 계속 보내고 서버가 한 번 응답 (예: 대용량 파일 업로드).
 * Bi-directional Streaming: 양방향 실시간 대화.
🔬 전공 심화: 인터셉터(Interceptors)와 미들웨어
분산 시스템에서는 모든 API 호출마다 인증, 로그 기록, 모니터링이 필요합니다. gRPC는 이를 인터셉터라는 개념으로 처리합니다.
 * Client Interceptor: 서버로 요청을 보내기 전/후에 동작. (예: 인증 토큰 주입)
 * Server Interceptor: 요청을 받아 처리하기 전/후에 동작. (예: 실행 시간 로깅)
🏗️ Go와 gRPC의 결합이 강력한 이유
Go의 고루틴은 gRPC의 스트리밍 구조와 완벽하게 결합됩니다. 하나의 gRPC 연결 안에서 수천 개의 스트림을 고루틴으로 가볍게 처리할 수 있기 때문입니다. 이는 HTTP/2 프로토콜의 멀티플렉싱(Multiplexing) 기술 덕분입니다.
