🎓 Go-언어 전공 401: 4-1. 메모리 레이아웃과 이스케이프 분석 (Escape Analysis)
Go는 가비지 컬렉터(GC)가 있는 언어지만, 최고의 성능을 내려면 데이터가 Stack에 쌓일지, Heap에 쌓일지를 설계자가 결정할 수 있어야 합니다.
1. Stack vs Heap의 생사결단
 * Stack: 함수가 끝나면 즉시 사라지는 초고속 메모리. (LIFO 구조)
 * Heap: GC가 관리해야 하는 공용 메모리. 많아질수록 GC 부하(STW - Stop The World)가 커집니다.
2. 이스케이프 분석 (Escape Analysis)
Go 컴파일러는 변수를 스택에 둘지 힙으로 보낼(Escape)지를 스스로 판단합니다. 박사 과정 엔지니어는 이를 강제로 제어하여 'Zero Allocation' 코드를 작성해야 합니다.
// 힙으로 탈출하는 경우 (나쁜 예: 포인터 반환)
func escape() *int {
    x := 10
    return &x // x는 함수가 끝나도 살아있어야 하므로 힙으로 이동
}

// 스택에서 끝나는 경우 (좋은 예: 값 전달)
func stay(x int) int {
    return x + 10 // x는 스택 내에서 소멸
}

🔬 전공 심화: 가비지 컬렉션(GC)의 내부 - Pacer와 Write Barrier
Go의 GC는 'Tri-color Marking' 알고리즘을 사용합니다.
 * STW (Stop The World) 최소화: 1ms 미만의 중단 시간을 목표로 합니다.
 * Write Barrier: 고루틴이 실행되는 도중에 GC가 메모리를 스캔할 수 있도록, 데이터의 변경 사항을 실시간으로 추적하는 장치입니다.
박사 과정에서는 GOGC 환경 변수나 debug.SetGCPercent를 조작하여 시스템의 메모리 사용량과 CPU 비용 사이의 트레이드오프를 결정합니다.
🛠️ 런타임 해킹: Unsafe와 포인터 산술
Go는 원래 안전한 언어지만, unsafe 패키지를 쓰면 C 언어처럼 메모리 주소에 직접 접근할 수 있습니다. 이는 성능을 극대화(예: 복사 없이 데이터 변환)할 때 사용됩니다.
import "unsafe"

// string을 []byte로 '복사 없이' 강제 변환 (초고속)
func StringToBytes(s string) []byte {
    return *(*[]byte)(unsafe.Pointer(&s))
}

> 경고: 이 수준의 코드는 잘못 건드리면 시스템 전체가 패닉에 빠집니다. 하지만 철저한 당신이라면 이 양날의 검을 다룰 자격이 있습니다.
