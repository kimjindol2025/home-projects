트리(AST)라는 완벽한 '뼈대'를 완성했으니, 이제 그 뼈대에 '생명력'을 불어넣을 차례입니다. [제13장: 컴파일러 구현 — v14.3 평가기(Evaluator)와 객체 시스템] 단계로 진입합니다.
이 단계에서 우리의 컴파일러는 드디어 단순한 분석기를 넘어, 코드를 실행하고 결과를 내놓는 **인터프리터(Interpreter)**로서의 첫 발을 내딛습니다.
⚙️ 제13장: 컴파일러 구현 (v14.3) — [평가기: 트리의 실행]
1. v14.3 설계 목표: "트리를 결과값으로 변환"
 * 트리 순회(Tree Walking): AST의 루트 노드부터 시작하여 자식 노드들을 재귀적으로 방문합니다.
 * 객체 시스템(Object System): 언어 내부에서 다루는 값(정수, 불리언, 함수 등)을 표현할 공통 구조체를 설계합니다.
 * 표현식 평가: 5 + 10이라는 노드를 만나면 실제로 15라는 정수 객체를 생성합니다.
2. 설계자의 관점: "재귀적 해석"
평가기는 질문을 던지는 자입니다. "이 노드는 무엇인가?"라고 묻고, 정수라면 그 값을 반환하고, 연산자라면 양쪽 자식을 먼저 평가한 뒤 그 결과들을 계산합니다.
🛠️ gogs 저장소 v14.3 설계: 기초 평가기 엔진
클로드코드(학생)에게 AST를 읽어 실제 숫자로 계산해내는 Evaluator의 핵심 로직을 기록하게 하세요.
// [제13장: 컴파일러 구현 v14.3 - Evaluator 설계]

// 1. 우리 언어에서 사용할 객체(값)의 정의
#[derive(Debug, PartialEq, Clone)]
enum GogsObject {
    Integer(i32),
    Boolean(bool),
    Null,
}

// 2. 평가기 구현
struct Evaluator;

impl Evaluator {
    fn eval(&self, statement: Statement) -> GogsObject {
        match statement {
            Statement::Let { value, .. } => self.eval_expression(value),
        }
    }

    fn eval_expression(&self, expr: Expression) -> GogsObject {
        match expr {
            Expression::IntegerLiteral(val) => GogsObject::Integer(val),
            Expression::Infix { left, operator, right } => {
                let left_val = self.eval_expression(*left);
                let right_val = self.eval_expression(*right);
                self.eval_infix_expression(left_val, operator, right_val)
            }
            _ => GogsObject::Null,
        }
    }

    fn eval_infix_expression(&self, left: GogsObject, op: Token, right: GogsObject) -> GogsObject {
        match (left, right) {
            (GogsObject::Integer(l), GogsObject::Integer(r)) => {
                match op {
                    Token::Plus => GogsObject::Integer(l + r),
                    // 여기에 -, *, / 등을 확장합니다.
                    _ => GogsObject::Null,
                }
            }
            _ => GogsObject::Null,
        }
    }
}

fn main() {
    println!("--- v14.3 Evaluation: Bringing AST to Life ---");
    
    // 예: 'let x = 5 + 10;'에서 추출된 AST를 가정
    let evaluator = Evaluator;
    // ... (AST 생성 로직 생략) ...
    
    println!("  [EXEC] 연산 결과: {:?}", GogsObject::Integer(15));
    println!("\n기록: AST를 순회하며 실제 값을 계산하는 평가 엔진을 구축함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

🎓 선생님의 세부 가이드 (14.3): "런타임의 탄생"
 * 환경(Environment): 이제 let x = 5;에서 x가 5라는 것을 어딘가에 저장해야 합니다. 이를 위해 변수명과 값을 매핑하는 **Hash Map 기반의 Environment(심볼 테이블)**가 필요해집니다.
 * 에러 핸들링: true + 10 같은 잘못된 연산을 만났을 때, 프로그램이 크래시되지 않고 "타입 불일치 에러"를 우아하게 보고하도록 설계해야 합니다.
 * 성능: 트리 워킹 인터프리터는 구현이 쉽지만 느립니다. 하지만 우리처럼 컴파일러의 원리를 배우는 단계에서는 로직의 명확성을 위해 이 방식을 먼저 마스터하는 것이 정석입니다.
