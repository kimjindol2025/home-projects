🎓 2단계: Go 오케스트레이터 (The Nerve Center) 설계
오케스트레이터는 외부에서 들어오는 거대한 데이터 스트림(Kafka, gRPC 등)을 받아, 1단계에서 만든 공유 메모리를 통해 Julia 워커들에게 가장 효율적으로 배분하는 역할을 합니다.
1. 지능형 워커 풀 (Dynamic Worker Pool)
단순한 루프가 아니라, Julia 프로세스의 상태를 실시간으로 모니터링하여 작업을 할당하는 '상태 인지형(State-aware)' 풀을 구축합니다.
 * Dispatcher 고루틴: 데이터를 수신하고 가용한 워커를 탐색합니다.
 * Worker 고루틴: 특정 Julia 인스턴스와 1:1로 매핑되어 공유 메모리에 데이터를 쓰고 연산 신호를 보냅니다.
2. Lock-Free 커뮤니케이션 (Ring Buffer)
Go 워커와 Julia 워커 사이의 통신에서 뮤텍스(Mutex)를 제거합니다. 대신 LMAX Disruptor 스타일의 Single-Producer, Single-Consumer(SPSC) 링 버퍼를 활용합니다.
// Go: Lock-Free 시퀀스 업데이트 예시
type RingBuffer struct {
    cursor uint64 // 원자적 연산으로 관리
    data   []Payload
}

func (rb *RingBuffer) Publish(val Payload) {
    next := atomic.AddUint64(&rb.cursor, 1)
    rb.data[next % capacity] = val
    // Julia에게 데이터 준비 완료 알림 (Atomic Flag)
}

🔬 전공 심화: 하드웨어 선호도 (CPU Affinity)
박사급 설계에서는 OS 스케줄러가 고루틴을 이리저리 옮기는 것조차 성능 손실로 간주합니다.
 * Runtime.LockOSThread(): 특정 고루틴을 OS 쓰레드에 고정합니다.
 * Taskset / CPU Pinning: 특정 Go 워커와 Julia 프로세스를 동일한 CPU 코어(혹은 NUMA 노드)에 고정하여 캐시 히트율(Cache Hit Rate)을 극대화합니다.
🏗️ 설계의 완성: 백프레셔(Backpressure) 제어
Julia가 연산 과부하에 걸리면 Go 오케스트레이터는 즉시 이를 감지해야 합니다.
 * Drop Strategy: 덜 중요한 데이터는 버립니다.
 * Buffering Strategy: 공유 메모리 내의 여유 공간에 임시 저장합니다.
 * Throttling: 데이터 소스(Producer)에게 전송 속도를 늦추라고 신호를 보냅니다.
