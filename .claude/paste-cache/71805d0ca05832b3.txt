 Dialect를 AI가 생성하게 만드는 것입니다.
1. 에이전트 아키텍처 (PPO/DQN 기반)
박사님의 에이전트는 현재의 IR 상태(State)를 입력받아 최적의 최적화 작업(Action)을 선택합니다.
 * State (S): IR의 특징 벡터 (루프 깊이, 연산 종류, 데이터 크기, 메모리 접근 패턴).
 * Action (A): 특정 루프에 대한 tile, interchange, vectorize 명령과 그 매개변수.
 * Reward (R): 1.2단계에서 측정한 실제 성능 향상치 (Latency_{old} / Latency_{new}).
2. Transform Dialect를 통한 동적 제어
AI는 직접 C++ 코드를 짜는 대신, 다음과 같은 MLIR 스크립트를 생성하여 컴파일러에 주입합니다.
// AI가 생성한 최적화 전략 (예시)
transform.sequence failures(propagate) {
^bb1(%arg1: !pdl.operation):
  %0 = transform.structured.match ops{["linalg.matmul"]} in %arg1
  // AI가 결정한 타일 크기 [32, 64, 32] 적용
  %1, %loops:3 = transform.structured.tile %0 [32, 64, 32]
  // AI가 결정한 벡터화 적용
  transform.structured.vectorize %1
}

[1.3 실전] 정책 경사(Policy Gradient) 최적화
박사님, 에이전트가 단순히 '빠른 것'만 찾는 게 아니라 **'안전한 것'**을 찾도록 학습시키는 것이 고도화된 전략입니다.
시나리오: 에이전트가 성능을 높이려고 타일 크기를 무리하게 키우다가 SRAM 용량을 초과해 버렸습니다.
[박사급 해결 전략]
 * Penalty Function: 비용 모델(1.1)을 활용하여 SRAM 한도를 초과하는 Action에 대해 강력한 **음의 보상(Negative Reward)**을 부여합니다.
 * Constrained RL: 하드웨어 제약 조건을 신경망의 마지막 레이어에 마스크(Mask)로 적용하여, 처음부터 불가능한 타일 크기는 선택조차 못 하게 설계합니다.
[1.3 실습] 연구자의 의사결정
박사님의 에이전트가 학습 중에 "성능은 좋은데 코드가 너무 복잡해져서 컴파일 시간이 100배 늘어나는" 전략을 찾아냈습니다. 이때 박사님은 어떻게 보상 함수를 수정하시겠습니까?
 * 성능이 최우선이므로 그대로 둔다.
 * 보상 함수에 Execution_Time뿐만 아니라 Compile_Time에 대한 페널티 항목을 추가하여 다중 목표 최적화(Multi-objective Optimization)를 수행한다.
정답: 2번입니다.
실제 산업계에서 쓰이는 컴파일러는 실행 성능과 컴파일 속도 사이의 균형(Trade-off)이 필수적입니다. 이를 수식으로 증명하는 것이 박사 논문의 현실성을 높여줍니다.
