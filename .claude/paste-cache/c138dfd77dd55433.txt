스마트 포인터 설계의 마지막 조각이자, 메모리 안전성의 마침표를 찍는 [제8장: 스마트 포인터 — v9.4 Weak<T>: 순환 참조의 고리 끊기 (Preventing Reference Cycles)] 단계입니다.
v9.3까지 우리는 강력한 공유 능력을 가졌지만, 그 대가로 **'메모리 누수(Memory Leak)'**라는 위험한 그림자를 마주하게 되었습니다. v9.4는 서로가 서로를 소유하여 영원히 메모리에서 사라지지 않는 '순환 참조' 문제를 해결하는 설계 기법을 다룹니다.
🧠 제8장: 스마트 포인터 (v9.4) — [Weak<T>: 소유하지 않는 참조]
1. v9.4 설계 목표: "비소유 참조의 구현"
 * 강한 참조(Strong Reference): Rc<T>가 제공하는 소유권이 있는 참조입니다. 카운트가 0이 되어야 데이터가 삭제됩니다.
 * 약한 참조(Weak Reference): Weak<T>가 제공하는, 소유권이 없는 참조입니다. 카운트가 올라가도 데이터의 삭제를 막지 않습니다.
 * 업그레이드 패턴: 약한 참조를 사용하기 전, 데이터가 아직 살아있는지 확인하고 잠시 강한 참조로 변환(upgrade)하는 안전한 접근 방식을 익힙니다.
2. 설계자의 관점: "부모와 자식의 관계 정립"
설계자는 데이터 간의 계층 구조를 명확히 해야 합니다. **"부모는 자식을 소유(Rc)하지만, 자식은 부모를 그저 바라볼 뿐(Weak)"**인 구조를 설계하여, 부모가 사라질 때 자식들도 함께 안전하게 메모리에서 해제되도록 만듭니다.
🛠️ gogs 저장소 v9.4 설계: 계층형 보안 노드 시스템
클로드코드(학생)에게 부모 노드와 자식 노드가 서로를 참조하면서도 메모리 누수가 발생하지 않는 '트리 구조'를 설계하게 하세요.
// [제8장: 스마트 포인터 v9.4]

use std::rc::{Rc, Weak};
use std::cell::RefCell;

#[derive(Debug)]
struct GogsNode {
    value: i32,
    // 부모는 자식을 소유함 (Rc)
    children: RefCell<Vec<Rc<GogsNode>>>,
    // 자식은 부모를 소유하지 않고 참조만 함 (Weak)
    parent: RefCell<Weak<GogsNode>>,
}

fn main() {
    println!("--- v9.4 Smart Pointers: Breaking Cycles with Weak<T> ---");

    // 1. 부모 노드 생성
    let leaf = Rc::new(GogsNode {
        value: 3,
        parent: RefCell::new(Weak::new()),
        children: RefCell::new(vec![]),
    });

    println!("  leaf의 강한 참조: {}, 약한 참조: {}", Rc::strong_count(&leaf), Rc::weak_count(&leaf));

    {
        let branch = Rc::new(GogsNode {
            value: 5,
            parent: RefCell::new(Weak::new()),
            children: RefCell::new(vec![Rc::clone(&leaf)]),
        });

        // 2. leaf의 부모를 branch로 설정 (Weak 참조)
        *leaf.parent.borrow_mut() = Rc::downgrade(&branch);

        println!("  branch 생성 후 - leaf 강한 참조: {}", Rc::strong_count(&leaf));
        println!("  branch 생성 후 - branch 강한 참조: {}, 약한 참조: {}", Rc::strong_count(&branch), Rc::weak_count(&branch));
    } 
    // 여기서 branch는 스코프를 벗어나며 소멸합니다. 
    // 만약 parent가 Rc였다면 branch는 결코 죽지 않았을 것입니다.

    println!("  branch 소멸 후 - leaf의 부모 존재 여부: {:?}", leaf.parent.borrow().upgrade());

    println!("\n기록: Weak<T>를 사용하여 순환 참조를 방지하고 메모리 자정 능력을 확보함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

스마트 포인터의 마지막 단계를 설계할 때의 지침입니다.
 * 소유권의 방향성: 데이터 흐름이 일방향(Top-down)인지 확인하십시오. 양방향 참조가 필요하다면 한쪽은 반드시 Weak<T>여야 합니다.
 * 존재의 불확실성: Weak<T>는 참조하는 대상이 이미 죽었을 가능성을 항상 내포합니다. 따라서 반드시 upgrade()를 호출하여 Option<Rc<T>> 결과를 안전하게 처리해야 합니다.
 * 메모리 누수 테스트: 복잡한 구조를 설계한 후에는 Rc::strong_count를 출력해 보며, 예상한 시점에 카운트가 0이 되는지 반드시 확인하십시오.
