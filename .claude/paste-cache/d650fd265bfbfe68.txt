박사 과정의 세 번째 관문, **5.3단계: 이기종 컴퓨팅의 심장, GPU 및 가속기 코드 생성(Codegen & GPU Dialect)**을 시작합니다.
"저장 필수 너는 기록이 증명이다 gogs" — 현대 컴퓨팅 연구의 핵심은 더 이상 CPU에만 머물지 않습니다. 박사 학위의 무게감은 CPU의 데이터를 어떻게 효율적으로 GPU나 NPU로 옮기고, 그곳의 수천 개 코어에 일을 나눠줄 것인가(Work-distribution)를 설계하는 능력에서 결정됩니다.
[MLIR 박사 코스 5.3] 경계를 넘는 데이터: "Host-Device Orchestration"
박사 수준의 연구자가 설계하는 가속기 컴파일러는 단순히 연산만 바꾸는 것이 아니라, **메모리의 위계(Memory Hierarchy)**를 완전히 장악해야 합니다.
1. 계층적 추상화 (Hierarchical Abstraction)
MLIR은 GPU 연산을 단계별로 깎아 내려갑니다.
 * gpu.launch: "이제부터 이 영역은 GPU에서 실행해!"라고 선언합니다.
 * gpu.block & gpu.thread: 수만 개의 스레드 중 각각의 스레드가 어떤 데이터를 처리할지 주소를 할당합니다.
 * 계층 구조: Grid > Block > Thread (이 구조를 설계하는 것이 성능의 핵심입니다.)
2. 메모리 최적화의 꽃: Shared Memory (공유 메모리)
박사 논문의 성능 그래프를 가르는 지점은 **'Global Memory'**로 가는 횟수를 줄이고, 각 블록의 빠른 **'Shared Memory'**를 얼마나 잘 활용하느냐에 있습니다.
 * Tiling + Promotion: 텐서 데이터를 작은 조각(Tile)으로 나눠서 공유 메모리로 승격(Promote)시킨 뒤 계산하는 패스를 설계해야 합니다.
3. 실제 흐름: GPU 로워링 파이프라인
 * High-level: linalg.matmul (행렬 곱셈)
 * GPU Mapping: gpu.launch 블록으로 감싸고 스레드 ID 할당.
 * To NVVM/ROCDL: NVIDIA(NVVM)나 AMD(ROCDL) 전용 저수준 Dialect로 변환.
 * PTX/Binary: 실제 하드웨어가 읽는 바이너리 생성.
[5.3 실전] 데이터 전송 병목 현상 분석
여러분이 설계한 AI 가속기 컴파일러에서 연산 속도는 매우 빠르지만, 전체 실행 시간은 단축되지 않았습니다. 박사 연구원으로서 어디를 점검하시겠습니까?
시나리오: "데이터를 CPU에서 GPU로 복사(Host-to-Device)하는 시간과 연산 시간이 분리되어 있음."
[박사급 해결 전략]
 * Asynchronous Transfer: 데이터 전송과 연산을 동시에 진행(Pipelining)하도록 IR 구조를 변경합니다.
 * Kernel Fusion: 여러 개의 작은 GPU 커널을 하나로 합쳐서 메모리 읽기/쓰기 횟수 자체를 줄입니다.
[5.3 실습] 설계자의 의사결정
GPU 최적화 패스를 짤 때, **'스레드 간의 데이터 간섭'**을 방지하면서 가장 빠른 속도를 내기 위해 반드시 확인해야 하는 개념은 무엇일까요?
 * Race Condition & Synchronization: 스레드들이 동시에 같은 메모리를 건드리지 않도록 gpu.barrier를 적절한 위치에 삽입했는가?
 * Print Debugging: 모든 스레드에 값을 출력해보고 있는가?
정답: 1번입니다.
병렬 프로그래밍에서는 동기화(Synchronization)를 최소화하면서도 정확성을 유지하는 설계를 수식과 코드로 증명하는 것이 박사 논문의 핵심입니다.
5.3단계의 기록 (증명)
> "이기종 컴퓨팅 연구는 연산의 병렬화와 메모리 계층 간의 데이터 이동 최적화가 결합된 예술이다."
> "MLIR GPU Dialect는 복잡한 하드웨어의 특성을 IR 수준에서 추상화하여, 연구자가 하위 기계어에 매몰되지 않고 알고리즘에 집중하게 한다."
> 
축하합니다. 이제 당신은 하드웨어와 소프트웨어의 경계에서 데이터를 자유자재로 다루는 시스템 아키텍트의 관점을 갖게 되었습니다.
이제 박사 과정의 마지막 단계인 **5.4단계: '최종 방어(Defense)를 위한 자동화 및 형식 검증(Formal Verification)'**으로 넘어가 보겠습니다. 내 컴파일러가 '수학적으로 완벽하게 오류가 없음'을 증명하는 방법입니다.
