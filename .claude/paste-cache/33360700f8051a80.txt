🧪 제7장: 신뢰의 구축 (v8.1) — [시스템의 결합]
1. v8.1 설계 목표: "경계 밖에서의 시선"
 * 통합 테스트(Integration Tests): src 디렉토리가 아닌 외부 tests 디렉토리에서 라이브러리를 실제 사용자처럼 호출하여 테스트합니다.
 * 공개 인터페이스(Public API) 검증: pub 키워드로 노출된 기능들이 외부에서 의도대로 연결되는지 확인합니다.
 * 라이브러리 구조화: 코드를 lib.rs로 분리하여 재사용 가능한 모듈로 재설계합니다.
2. 설계자의 관점: "사용자 경험의 코드화"
설계자는 자신이 만든 코드를 사용하는 '다른 개발자'의 입장이 되어봐야 합니다. 통합 테스트는 내 코드가 외부 시스템에 통합되었을 때 발생할 수 있는 잠재적 마찰을 미리 찾아내는 **'최종 방어선'**입니다.
🛠️ gogs 저장소 v8.1 설계: 외부 보안 감사 라이브러리
클로드코드(학생)에게 라이브러리 구조를 갖추고 외부 통합 테스트 환경을 구축하게 하세요.
// [제7장: 신뢰의 구축 v8.1 - src/lib.rs 내용]

/// gogs 시스템의 핵심 보안 로직을 담은 라이브러리입니다.
pub struct GogsVault {
    secret_key: String,
}

impl GogsVault {
    pub fn new(key: &str) -> Self {
        Self { secret_key: key.to_string() }
    }

    pub fn access(&self, input: &str) -> bool {
        self.secret_key == input
    }
}

🎓 선생님의 가이드: "v8.1 설계자의 통합 원칙"
통합 테스트를 설계할 때 명심해야 할 규칙입니다.
 * 철저한 외부인 마인드: 통합 테스트에서는 pub으로 공개된 기능만 사용할 수 있습니다. 내부 구현(private)에 의존하지 마십시오.
 * 시나리오 중심: 단순히 함수 하나를 테스트하는 것이 아니라, "금고를 생성하고 -> 키를 입력하고 -> 접근 권한을 확인한다"는 식의 **사용자 흐름(Workflow)**을 테스트하십시오.
 * 파일별 독립성: tests/ 폴더 안의 각 파일은 별개의 크레이트로 컴파일되므로, 파일 간의 독립적인 테스트 환경을 구축할 수 있습니다.
