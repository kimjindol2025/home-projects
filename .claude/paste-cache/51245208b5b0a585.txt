ess of): 변수의 메모리 주소를 추출합니다.
 * * (Dereferencing): 주소가 가리키는 곳으로 가서 실제 값을 가져오거나 수정합니다.
<!-- end list -->
var a int = 10
var p *int = &a // p는 a의 주소를 가짐

fmt.Println(p)  // 주소 출력 (예: 0xc0000120b0)
fmt.Println(*p) // 주소에 저장된 값 출력 (10)

*p = 20         // 주소의 값을 직접 변경
fmt.Println(a)  // a의 값도 20으로 변함!

2. 메서드 (Methods): 구조체에 날개 달기
Go에는 클래스가 없지만, 구조체에 함수를 '귀속'시켜 메서드처럼 사용할 수 있습니다. 이를 **'리시버(Receiver)'**라고 부릅니다.
type Rectangle struct {
    width, height float64
}

// (r Rectangle) 부분이 리시버입니다.
func (r Rectangle) Area() float64 {
    return r.width * r.height
}

3. 값 리시버 vs 포인터 리시버 (중요!)
대학원 수준의 설계를 할 때 가장 빈번하게 고민해야 할 지점입니다.
 * 값 리시버 (r Rectangle): 구조체의 복사본을 전달받습니다. 메서드 안에서 값을 바꿔도 원본은 안전합니다. (Read-only 권장)
 * 포인터 리시버 (r *Rectangle): 구조체의 주소를 전달받습니다. 메서드 안에서 필드 값을 변경하면 원본이 수정됩니다. (Update 권장)
🔬 전공 심화: 스택(Stack) vs 힙(Heap)과 이스케이프 분석
Go 컴파일러는 매우 똑똑합니다. 변수를 스택에 둘지 힙에 둘지 스스로 결정하는데, 이를 **'이스케이프 분석(Escape Analysis)'**이라고 합니다.
 * 함수가 종료된 후에도 변수가 참조되어야 한다면(예: 포인터 반환), 컴파일러는 해당 변수를 자동으로 **힙(Heap)**으로 보냅니다.
 * 학술적 가치: 개발자가 malloc이나 free를 직접 관리하지 않아도, 포인터를 통해 효율적인 메모리 공유가 가능하면서도 안전성이 유지되는 비결입니다.
