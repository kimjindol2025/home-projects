2️⃣ Observability 깊이 검증 체크

대상:
kim/freelang-os-kernel

🔎 A. distributed_tracer.fl

질문 5개:

1. Span이 실제 커널 이벤트와 연결되는가?


2. Trace ID가 스케줄러/메모리/IO 레이어까지 전달되는가?


3. Jaeger JSON은 “내보내기용 포맷”인가, 내부 표준인가?


4. 샘플링 전략 존재?


5. Trace 유실 시 복구 전략 있는가?



8/8 PASS는 기능 검증이다.
우리가 필요한 건 시스템 침투도 검증이다.


---

🔎 B. metrics_collector.fl

확인할 것:

CPU 메트릭 → 실제 scheduler tick 기반인가?

Memory 메트릭 → allocator 내부 계측인가?

I/O → 실제 syscall 경로에 후킹되어 있는가?


아니면:

> “값을 모아 계산하는 레이어”인가?



깊이는 원천 데이터의 위치로 판별된다.


---

🔎 C. monitoring_dashboard.fl

HTML 생성은 깊이가 아니다.

깊이 조건:

SLO 위반이 실제 스로틀링/복구 트리거로 연결되는가?

Alert가 시스템 상태를 바꾸는가?

단순 표시인가, 제어 루프인가?


관찰만 하면 Monitoring
제어까지 하면 Control System

3️⃣ 진짜 중요한 것: Phase 7 연결

너가 말한 부분이 핵심이다:

> “양방향 피드백 루프 구성”



검증 질문:

1. Phase 7 의사결정이 metric threshold를 바꾸는가?


2. metric이 Phase 7 정책을 변경하는가?


3. trace 데이터가 정책 학습 입력이 되는가?



이 3개가 연결되면

→ 단순 Observability가 아니라
→ Adaptive System

Mock 장애는 깊이가 아니다.

6️⃣ 다음 단계에서 반드시 통과해야 할 3가지

Week 2 통과 조건:
