🏛️ LLVM 전설: 2.3 의미 분석과 IR 생성 통합 (The Bridge)
단순히 문법이 맞다고 해서 실행 가능한 코드는 아닙니다. a + 1이라는 문장에서 a가 선언되었는지, 타입은 숫자가 맞는지 확인하는 과정이 **의미 분석(Semantic Analysis)**입니다.
1. 심볼 테이블(Symbol Table): 이름의 장부
컴파일러는 현재 어떤 변수와 함수가 사용 가능한지 기록하는 장부가 필요합니다.
 * Scope: 전역 변수와 지역 변수를 구분합니다. 새로운 블록({ })에 들어갈 때마다 새로운 장부를 겹쳐 쌓는 스택 구조를 가집니다.
 * 기록 내용: 이름, 타입, 할당된 LLVM ValueRef 주소.
2. 시맨틱 체크 (Semantic Check)
AST를 순회하며 다음 항목들을 검사합니다.
 * Variable Resolution: 선언되지 않은 변수를 사용하고 있는가?
 * Type Mismatch: 정수와 문자열을 더하려고 하는가?
 * Function Signature: 함수의 인자 개수와 타입이 호출부와 일치하는가?
3. 방문자 패턴(Visitor Pattern)과 코드 생성
AST의 각 노드를 하나씩 '방문'하며 이전에 배운 LLVM API 명령어를 호출합니다.
fn generateCode(node: *Node, builder: c.LLVMBuilderRef) c.LLVMValueRef {
    switch (node.*) {
        .number => |val| {
            return c.LLVMConstReal(c.LLVMDoubleType(), val);
        },
        .binary_op => |op| {
            const left = generateCode(op.left, builder);
            const right = generateCode(op.right, builder);
            return switch (op.op) {
                '+' => c.LLVMBuildFAdd(builder, left, right, "addtmp"),
                '*' => c.LLVMBuildFMul(builder, left, right, "multmp"),
                else => unreachable,
            };
        },
        .identifier => |name| {
            // 심볼 테이블에서 주소를 찾아 반환
            return symbol_table.get(name) orelse error.UndefinedVariable;
        }
    }
}

🔬 전공 심화: 맹목적 생성 vs 지능적 생성
전설적인 컴파일러는 단순히 명령어를 나열하지 않습니다.
 * Constant Folding: AST 단계에서 1 + 2를 발견하면 IR을 생성하지 않고 즉시 3으로 바꿉니다.
 * Short-circuiting: A && B 연산에서 A가 거짓이면 B를 생성하는 코드를 건너뛰도록 분기문(1.3 과정)을 자동 설계합니다.
📦 기록의 결실: 실행 파일 탄생
의미 분석을 통과하고 IR 생성이 완료되면, 이제 1.5에서 배운 JIT 엔진에 넣거나 .o 오브젝트 파일로 저장하여 실행 파일을 만들 수 있습니다. 드디어 당신이 정의한 문법이 CPU 위에서 춤을 추게 됩니다.
