🎓 Zig 전공 301: 3-3. 물리 메모리 관리자(PMM) 및 비트맵 설계
물리 메모리 관리의 핵심은 '페이지(Page)' 단위의 분할입니다. 현대 아키텍처는 보통 메모리를 4KB(4096 바이트) 단위로 나누어 관리합니다.
1. 비트맵(Bitmap) 알고리즘
메모리의 각 페이지가 사용 중인지(1) 비어 있는지(0)를 단 1비트로 표시하는 방식입니다.
 * 효율성: 32GB의 RAM을 관리하는 데 단 1MB의 비트맵 메모리만 있으면 됩니다.
 * Zig의 이점: u64 배열을 사용하여 64개의 페이지 상태를 한 번에 확인하고 조작할 수 있습니다.
2. 메모리 맵(Memory Map) 읽기
부팅 시 BIOS나 UEFI는 커널에게 "어디가 사용 가능한 RAM이고, 어디가 하드웨어 장치용인지" 알려주는 표를 넘겨줍니다. 커널은 이 기록을 바탕으로 비트맵을 초기화해야 합니다.
const PageSize = 4096;

const MemoryManager = struct {
    bitmap: []u64,
    total_pages: usize,

    // 특정 페이지가 사용 중인지 확인
    pub fn isUsed(self: *MemoryManager, page_index: usize) bool {
        const bit_idx = page_index % 64;
        const array_idx = page_index / 64;
        return (self.bitmap[array_idx] & (@as(u64, 1) << @intCast(bit_idx))) != 0;
    }

    // 빈 페이지 하나를 찾아 할당 (First-fit 알고리즘)
    pub fn allocPage(self: *MemoryManager) !usize {
        for (self.bitmap, 0..) |*entry, i| {
            if (entry.* != 0xFFFFFFFFFFFFFFFF) { // 빈 자리가 있다면
                const bit_idx = @ctz(~entry.*); // 0인 비트의 위치 찾기 (comptime 내장 함수 활용)
                entry.* |= (@as(u64, 1) << @intCast(bit_idx));
                return i * 64 + bit_idx;
            }
        }
        return error.OutOfMemory;
    }
};

3. 정적 할당과 동적 할당의 경계
커널 초기화 단계에서는 메모리 관리자 자체가 사용할 메모리조차 직접 할당해야 하는 '닭과 달걀' 문제에 직면합니다.
 * 설계: 커널 코드 바로 뒤의 여유 공간을 비트맵 저장소로 예약하도록 링커 스크립트에서 위치를 고정합니다.
🔬 전공 심화: @ctz와 comptime 최적화
Zig의 @ctz (Count Trailing Zeros) 함수는 CPU의 전용 명령어를 사용하여 비트맵에서 빈 공간을 찾는 속도를 극대화합니다.
 * comptime을 통해 비트맵의 크기나 메모리 정렬을 빌드 타임에 확정하면, 런타임에는 단 몇 줄의 기계어만 실행되어 최고 수준의 성능을 보장합니다.
📦 기록의 무결성: 페이지 정렬(Alignment)
하드웨어는 페이지가 정확히 4096 바이트 경계에 맞춰져 있을 때만 정상적으로 작동합니다.
 * Zig의 align(4096) 키워드를 사용하여 모든 메모리 할당이 하드웨어 규격을 준수하는지 강제합니다. **"규격의 기록이 곧 하드웨어와의 약속"**입니다.
📝 Assignment 3-3 (메모리 관리 설계 과제)
 * 비트맵 계산: 128MB의 메모리를 4KB 페이지 단위로 관리할 때, 비트맵 배열([]u64)의 크기가 얼마나 되어야 하는지 계산하고 기록하세요.
 * Free 함수 구현: 특정 페이지 번호를 받아 비트맵에서 해당 비트를 다시 0으로 만드는 freePage(index) 함수를 작성하세요.
 * 연속 할당 설계: 단일 페이지가 아닌, 여러 개의 연속된 페이지를 한꺼번에 할당받는 로직을 고민해 보세요. (예: allocPages(count))
 * 기록: 메모리 부족(OutOfMemory) 상황이 발생했을 때 커널이 어떻게 대처해야 할지(Panic? 혹은 스왑?) 시나리오를 설계하여 남기세요.
