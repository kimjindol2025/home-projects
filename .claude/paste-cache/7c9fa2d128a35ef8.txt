시간의 설계가 한 단계 더 정교해집니다. [제6장: 수명(Lifetimes) — v7.1 구조체 수명 명시 (Lifetime Annotations in Structs)] 단계입니다.
v7.0에서는 함수의 매개변수 사이의 시간 관계를 정의했다면, v7.1은 구조체가 참조자를 데이터로 가질 때 발생하는 문제를 해결합니다. 설계자의 관점에서 이는 "자신보다 먼저 죽을 수도 있는 데이터를 품은 구조체"를 안전하게 관리하는 고도의 기술입니다.
⏳ 제6장: 수명 (v7.1) — [구조체와 참조자]
1. v7.1 설계 목표: "데이터와 소유주의 운명 공동체"
 * 참조자 보관: 구조체 필드에 소유권이 없는 참조자(&str 등)를 담는 법을 익힙니다.
 * 수명 전파: 구조체가 살아있는 동안 그 내부의 참조자도 반드시 살아있음을 보장합니다.
 * 컴파일 타임 제약: 구조체의 인스턴스가 내부 데이터보다 오래 살아남아 '메모리 미아'가 생기는 것을 원천 봉쇄합니다.
2. 설계자의 관점: "빌려온 데이터로 만든 도구"
모든 데이터를 구조체가 직접 소유(String)할 필요는 없습니다. 대량의 텍스트 데이터가 어딘가에 이미 있다면, 구조체는 그 부분을 살짝 가리키기만(&str) 함으로써 메모리를 극도로 절약할 수 있습니다. 이때 필요한 것이 바로 수명 명시입니다.
🛠️ gogs 저장소 v7.1 설계: 스마트 로그 파서(Parser)
클로드코드(학생)에게 원본 데이터를 복사하지 않고 참조만 하여 분석하는 '고효율 파서'를 설계하게 하세요.
// [제6장: 수명 v7.1]

/// [설계] GogsParser는 'a라는 수명을 가집니다.
/// 이는 "이 구조체는 context가 참조하는 원본 데이터보다 오래 살 수 없다"는 뜻입니다.
struct GogsParser<'a> {
    context: &'a str, // 원본 데이터를 빌려옴
}

impl<'a> GogsParser<'a> {
    fn announce(&self) {
        println!("현재 분석 중인 문맥: {}", self.context);
    }
}

fn main() {
    println!("--- v7.1 Struct Lifetimes: Anchoring References ---");

    // 1. 원본 데이터 생성 (Owner)
    let log_data = String::from("LEVEL:INFO | MSG:System_Active");

    // 2. 구조체 생성 (Borrower)
    // log_data에서 'LEVEL' 부분만 슬라이스로 빌려옵니다.
    let first_word = log_data.split('|').next().expect("데이터가 비어있음");
    
    let parser = GogsParser {
        context: first_word,
    };

    parser.announce();

    // 만약 여기서 drop(log_data)를 하면, 
    // 아래에서 parser를 사용할 때 컴파일 에러가 발생합니다.
    // 이것이 v7.1 설계의 핵심 안전장치입니다.

    println!("\n기록: 구조체 내부 필드에 수명을 명시하여 참조 데이터의 유효성을 강제함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

🎓 선생님의 가이드: "v7.1 설계자의 유효 기간 검토"
구조체에 수명을 넣을 때의 황금률입니다.
 * 구조체는 데이터보다 짧게: 'a는 "구조체가 죽기 전까지 데이터가 살아있어야 함"을 의미합니다. 데이터가 먼저 죽으면 구조체도 무용지물이 되어야 합니다.
 * 전파되는 수명: 구조체에 수명이 생기면, 그 구조체를 사용하는 모든 함수와 impl 블록에도 수명 기호('a)를 명시해야 합니다. 이는 설계의 복잡도를 높이지만, 그만큼 메모리 안전성은 완벽해집니다.
 * 성능의 이득: 데이터를 복사(clone)하지 않고 참조만 하므로, 수백 메가바이트의 로그를 처리해도 메모리 사용량은 극히 미미합니다.
> 저장 필수 너는 기록이 증명이다 gogs
> 
이제 당신은 메모리를 복사하지 않고도 복잡한 데이터를 안전하게 다루는 **'저지연(Low-Latency) 설계자'**가 되었습니다.
