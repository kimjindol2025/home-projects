앞으로 그 형식(다음 단계 예고 / 선택지 메뉴)은 제외한다.
기록 완료.


---

1.3 단계 — 메타데이터 기반 필터링

이제부터는 단순 점수 검색이 아니라
**“구조적 필터링”**을 추가한다.

목표:

> 🔥 사용자가 특정 repo / 버전 / phase를 지정하면
그 범위 안에서만 검색




---

1️⃣ 왜 필요한가

지금 1.2 구조의 문제:

v10, v11, v12 문서가 모두 섞임

오래된 설계가 상위에 올라올 수 있음

진화 추론이 불가능


해결:

> 검색 전에 “공간을 줄인다”




---

2️⃣ 요청 구조 확장

기존:

{
  "repo": "freelang-v6",
  "question": "FFI memory model?"
}

확장:

{
  "repo": "freelang-v6",
  "version": "v12.2",
  "phase": "Phase 11",
  "question": "FFI memory model?"
}

version / phase는 optional.


---

3️⃣ 필터링 로직

검색 전에 먼저 걸러낸다:

function filterChunks(chunks, filters) {
  return chunks.filter(c => {
    if (filters.version && c.version !== filters.version) return false;
    if (filters.phase && c.phase !== filters.phase) return false;
    return true;
  });
}

그 다음에 점수 계산.


---

4️⃣ 검색 순서 (중요)

1. Repo 선택
2. 메타 필터 적용
3. 남은 chunk에 점수 계산
4. Top-K 선택
5. LLM 전달

이 순서가 핵심이다.


---

5️⃣ 이제 가능한 것

✅ 특정 버전만 분석

“v12.2에서만 FFI 설명해라”

✅ 진화 비교 모드

version: v12.0

version: v12.2 → 두 집합 각각 검색 후 diff 분석


✅ Phase 단위 사고 추적

“Phase 11에서 메모리 모델이 어떻게 정의됐는가?”


---

6️⃣ LLM 프롬프트도 강화

다음은 선택된 버전/Phase의 기록이다.

버전: v12.2
Phase: Phase 11

[Top chunks]

반드시 이 기록에 근거해서만 답하라.
다른 버전 내용은 사용하지 마라.

이렇게 해야 환각 줄어든다.


---

7️⃣ 구조적 의미

이제 이 시스템은 단순 검색봇이 아니라:

> 📚 버전 인지형 기록 추론 엔진



이 된다.
