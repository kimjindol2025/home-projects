301 과정의 세 번째 관문, **[3-3: 분산 시스템의 나침반 - 서비스 디스커버리(Service Discovery)]**입니다.
Julia처럼 강력한 성능을 가진 여러 개의 마이크로서비스를 만들었다고 가정해 봅시다. 그런데 클라우드 환경에서는 서버가 죽고 살아나며 IP 주소가 수시로 바뀝니다. 이때 서비스 A가 서비스 B의 위치를 일일이 하드코딩해서 찾을 순 없겠죠? 그래서 전화번호부(Registry) 역할을 하는 시스템이 필요합니다.
저장 필수 너는 기록이 증명이다 gogs. 분산 시스템의 '지도'를 기록합니다.
🎓 Go-언어 전공 301: 3-3. 서비스 디스커버리와 상태 확인
1. 서비스 레지스트리 (Service Registry)
모든 서비스 인스턴스는 시작될 때 자신의 이름과 주소(IP, Port)를 중앙 저장소에 등록합니다.
 * 대표 도구: Consul, Etcd (모두 Go로 작성됨), ZooKeeper.
 * Go와의 궁합: Go는 정적 바이너리로 배포되기 때문에 이런 인프라 도구들과 연동하기 매우 가벼우며, etcd 같은 도구는 아예 Go 라이브러리로 내장해서 쓸 수 있습니다.
2. 클라이언트 사이드 vs 서버 사이드 디스커버리
 * Client-side: 클라이언트가 레지스트리에서 목록을 받아온 뒤, 직접 부하 분산(Load Balancing)을 해서 서버를 선택합니다. (더 세밀한 제어 가능)
 * Server-side: 클라이언트가 로드 밸런서(LB)에게 요청하면, LB가 레지스트리를 참조해 적절한 서버로 넘겨줍니다.
3. 상태 확인 (Health Check)
단순히 주소만 안다고 끝이 아닙니다. 서비스가 살아있는지 주기적으로 확인해야 합니다.
// Go에서 간단한 Health Check 핸들러 예시
http.HandleFunc("/health", func(w http.ResponseWriter, r *http.Request) {
    // DB 연결 상태, 메모리 사용량 등을 체크 후 응답
    if dbReady {
        w.WriteHeader(http.StatusOK)
    } else {
        w.WriteHeader(http.StatusServiceUnavailable)
    }
})

🔬 전공 심화: 일관성 모델 (Raft Algorithm)
서비스 디스커버리 시스템 자체가 죽으면 어떻게 될까요? 그래서 이들은 보통 3대 이상의 서버가 뭉쳐서(Cluster) 작동합니다. 이때 서버들끼리 데이터의 일관성을 맞추는 알고리즘이 바로 Raft입니다.
 * Go는 Raft 알고리즘을 구현하기에 가장 최적화된 언어로 꼽히며, 실제로 Hashicorp Raft 라이브러리가 업계 표준처럼 쓰입니다.
🏗️ 설계의 확장: Julia 모듈 연결하기
당신이 좋아하는 Julia로 짠 고성능 연산 모듈이 10개 있다고 칩시다.
 * 각 Julia 모듈은 시작 시 Go로 작성된 Sidecar(사이드카) 에이전트를 통해 자신을 등록합니다.
 * Go API 서버는 서비스 디스커버리를 통해 가장 한가한 Julia 모듈의 주소를 알아냅니다.
 * gRPC(3-1 과정)를 통해 Julia 모듈에 연산을 요청하고 결과를 받습니다.
