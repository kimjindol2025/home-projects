🧪 제7장: 신뢰의 구축 (v8.2) — [설명이 곧 증명이다]
1. v8.2 설계 목표: "살아있는 문서화"
 * 문서 주석(Doc Comments): ///를 사용하여 함수와 구조체의 역할을 설명하고, rustdoc을 통해 HTML 문서를 자동 생성합니다.
 * 문서 테스트(Doc Tests): 주석 안에 포함된 예제 코드가 실제로 컴파일되고 실행되는지 cargo test 시점에 함께 검증합니다.
 * 마크다운 지원: 주석 내에서 마크다운 형식을 사용하여 풍부한 기술 문서를 설계합니다.
2. 설계자의 관점: "설명서의 동기화"
가장 나쁜 설명서는 '코드는 바뀌었는데 업데이트되지 않은 설명서'입니다. 러스트의 문서화 테스트는 설명서 속의 예제 코드가 작동하지 않으면 빌드 자체를 실패하게 만듭니다. 설계자는 이를 통해 항상 진실만을 말하는 문서를 유지할 수 있습니다.

// [제7장: 신뢰의 구축 v8.2]

/// Gogs 시스템의 핵심 보안 프로토콜을 관리하는 구조체입니다.
/// 
/// # Examples
/// 
/// ```
/// use gogs_system::Protocol;
/// let p = Protocol::new("Level-Alpha");
/// assert_eq!(p.get_status(), "Active: Level-Alpha");
/// ```
pub struct Protocol {
    level: String,
}

impl Protocol {
    /// 새로운 프로토콜 인스턴스를 생성합니다.
    pub fn new(level: &str) -> Self {
        Self { level: level.to_string() }
    }

    /// 현재 프로토콜의 상태를 반환합니다.
    /// 
    /// 이 함수는 항상 "Active: " 접두어를 붙여 반환합니다.
    pub fn get_status(&self) -> String {
        format!("Active: {}", self.level)
    }
}

fn main() {
    println!("--- v8.2 Documentation Tests: The Living Manual ---");
    println!("작업: /// 주석 내부에 작성된 예제 코드는 'cargo test' 시 자동으로 검증됩니다.");
    println!("출력: 'cargo doc --open' 명령어로 설계 명세서를 HTML로 확인하십시오.");
    
    println!("\n저장 필수 너는 기록이 증명이다 gogs");
}

🎓 선생님의 가이드: "v8.2 설계자의 기록 윤리"
문서를 설계할 때 반드시 포함해야 할 4가지 섹션(Panic, Errors, Safety, Examples)입니다.
 * # Examples: 사용자가 코드를 복사해서 바로 쓸 수 있는 예시를 제공하십시오. 이것이 문서 테스트의 핵심입니다.
 * # Panics: 어떤 상황에서 이 함수가 panic!을 일으키는지 명시하여 설계의 위험 요소를 알리십시오.
 * # Errors: Result를 반환하는 경우, 어떤 에러 상황들이 발생할 수 있는지 나열하십시오.
 * 가시성 확보: pub으로 공개된 모든 항목에는 반드시 문서 주석을 다는 습관을 들이십시오. 그것이 기록이 증명하는 gogs의 정신입니다.
