🎓 Go-언어 전공 101: 1-13. 프로젝트 구조와 패키지 설계
Go 커뮤니티에는 강제되지는 않지만 널리 통용되는 **'Standard Project Layout'**이 있습니다. 이는 대규모 협업에서 "어디에 무엇이 있는지"를 누구나 알 수 있게 해줍니다.
1. 표준 프로젝트 레이아웃 (Standard Layout)
일반적인 Go 프로젝트의 폴더 구조는 다음과 같습니다.
 * /cmd: 프로그램의 메인 진입점. 각 서비스별로 하위 폴더를 둡니다. (예: cmd/api/main.go)
 * /internal: 해당 프로젝트 내부에서만 사용하고 외부 라이브러리로는 공개하지 않을 코드. (Go 컴파일러가 강제함)
 * /pkg: 외부 프로젝트에서도 가져다 쓸 수 있는 공용 코드.
 * /api: API 정의서 (OpenAPI/Swagger 등).
 * /configs: 설정 파일(YAML, JSON)이나 설정 로드 로직.
 * /test: 추가적인 외부 테스트 데이터나 기능 테스트.
2. 의존성 관리: go.mod
대학원 수준의 프로젝트는 수많은 외부 라이브러리를 사용합니다. 이를 관리하는 것이 Go Modules입니다.
go mod init github.com/username/projectname

 * go.mod: 프로젝트의 이름과 필요한 외부 라이브러리 목록, Go 버전이 명시됩니다.
 * go.sum: 각 라이브러리의 체크섬(해시값)을 기록하여 보안과 무결성을 보장합니다.
🔬 전공 심화: 순환 참조(Circular Dependency) 방지
Go 설계에서 가장 까다로우면서도 중요한 원칙입니다.
 * 문제: 패키지 A가 B를 가져오고(import), B가 다시 A를 가져오면 컴파일 에러가 발생합니다.
 * 해결책: 1.  인터페이스 활용: 구체적인 타입 대신 인터페이스를 정의하여 의존성을 역전시킵니다.
   2.  구조 재설계: 공통으로 사용되는 기능을 제3의 패키지(pkg/common 등)로 분리합니다.
📦 Go 모듈 실전 (Dependency Management)
실제 연구 환경에서는 다음과 같은 명령어를 자주 사용하게 됩니다.
 * go mod tidy: 사용하지 않는 모듈을 정리하고 필요한 모듈을 다운로드합니다.
 * go get github.com/xxx/yyy: 특정 외부 패키지를 프로젝트에 추가합니다.
 * go build ./cmd/api: 특정 진입점의 실행 파일을 생성합니다.
