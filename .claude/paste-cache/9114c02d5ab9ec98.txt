🏗️ 제3장: 모듈화의 초석 (v4.0) — [함수의 정의]
1. v4.0 설계 목표: "코드의 부품화"
 * 추상화(Abstraction): 내부의 복잡한 로직은 숨기고, 외부에는 단순한 이름(인터페이스)만 노출합니다.
 * 단일 책임 원칙(SRP): 하나의 함수는 오직 하나의 일만 정밀하게 수행하도록 설계합니다.
 * 입력과 출력의 규약: 매개변수(Parameter)와 반환 타입(Return Type)을 명확히 정의하여 데이터 흐름을 예측 가능하게 만듭니다.
2. 설계자의 관점: "블랙박스 설계"
사용자는 함수 내부가 어떻게 돌아가는지 알 필요가 없습니다. "입력을 넣으면 기대한 출력이 나온다"는 믿음을 주는 것이 v4.0 설계의 핵심입니다.
🛠️ gogs 저장소 v4.0 설계: 첫 번째 모듈화
제2장의 거대한 논리를 함수로 분리하여 관리 효율을 극대화한 설계를 기록하세요.
// [제3장: 모듈화의 초석 v4.0]

fn main() {
    println!("--- v4.0 Modular Design: Functional Foundations ---");

    let sensor_data = 95;

    // [v4.0 설계] 복잡한 로직을 함수 호출 한 줄로 대체
    // 설계 의도: 메인 함수는 흐름만 관리하고, 세부 판단은 전문가(함수)에게 맡긴다.
    let report = analyze_and_report(sensor_data);
    
    println!("결과 보고: {}", report);
    
    // 재사용성 증명: 다른 데이터도 동일한 로직으로 즉시 처리
    println!("추가 분석: {}", analyze_and_report(45));

    println!("\n기록: 함수화를 통해 코드의 가독성과 재사용성을 확보함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

/// [설계] 센서 데이터를 분석하여 리포트를 생성하는 독립 부품
/// - 매개변수: input (i32)
/// - 반환값: String
fn analyze_and_report(input: i32) -> String {
    // 제2장에서 완성한 v3.4 패턴 매칭 논리를 함수 내부로 캡슐화
    match input {
        i if i >= 100 => format!("[CRITICAL] 위험 수치: {}", i),
        i if i >= 80  => format!("[WARNING] 주의 수치: {}", i),
        _             => format!("[NORMAL] 안정 수치: {}", i),
    }
}
