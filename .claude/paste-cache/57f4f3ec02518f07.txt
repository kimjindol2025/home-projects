좋습니다. 선언적 매크로의 기초적인 패턴 매칭을 익혔으니, 이제 그 화력을 극대화할 차례입니다. [제12장: 매크로 — v13.0.2 재귀적 매크로와 DSL(Domain Specific Language) 설계] 단계로 진입합니다.
컴파일러 제작자에게 이 기술은 매우 중요합니다. 우리가 만들 언어의 문법(Syntax)을 러스트 안에서 미리 시뮬레이션하거나, 복잡한 트리 구조의 데이터를 한눈에 들어오는 문법으로 정의할 수 있게 해주기 때문입니다.
🛠️ 제12장: 매크로 (v13.0.2) — [재귀와 맞춤형 문법]
1. 설계 목표: "러스트 안에 나만의 언어를 심다"
 * 재귀적 매크로 (Recursive Macros): 매크로가 자기 자신을 다시 호출하면서 복잡한 입력을 하나씩 깎아 나갑니다.
 * DSL 설계: 기존 러스트 문법이 아닌, 우리가 정의한 규칙(예: 키 => 값)대로 코드를 작성하면 매크로가 이를 해석합니다.
 * 증분 파싱 (Incremental Parsing): 입력된 토큰 뭉치를 앞에서부터 하나씩 처리하고 나머지를 다시 매크로에 던지는 기법을 익힙니다.
2. 설계자의 관점: "복잡성을 분할 정복하라"
컴파일러의 구문 분석기(Parser)와 마찬가지로, 매크로도 큰 덩어리를 작게 쪼개야 합니다. "첫 번째 항목을 처리하고, 남은 것은 다음 루프(재귀)로 넘긴다"는 논리가 핵심입니다.
🏗️ gogs 저장소 v13.0.2 설계: 컴파일러 설정 DSL (Config DSL)
컴파일러의 옵션을 설정할 때, 마치 별도의 언어를 쓰는 것처럼 직관적인 문법을 만들어 보겠습니다.
// [제12장: 매크로 v13.0.2 - 재귀적 DSL 설계]

macro_rules! gogs_config {
    // 1. [종료 조건] 더 이상 처리할 토큰이 없으면 종료
    () => {};

    // 2. [재귀 단계] '키 => 값' 패턴을 처리하고 나머지를 재귀 호출
    // @기호는 내부적인 재귀 처리를 위한 관용적 마커로 자주 쓰입니다.
    ($key:ident => $val:expr, $($rest:tt)*) => {
        println!("  [CONFIG] 설정 적용: {} = {}", stringify!($key), $val);
        // 남은 부분($rest)을 다시 매크로에 전달
        gogs_config!($($rest)*);
    };

    // 3. 마지막 항목 (쉼표가 없을 때) 처리
    ($key:ident => $val:expr) => {
        gogs_config!($key => $val,);
    };
}

fn main() {
    println!("--- v13.0.2 Recursive Macros: Building a DSL ---");

    // 우리가 만든 커스텀 문법으로 컴파일러 환경을 설정합니다.
    gogs_config! {
        version => "1.0.0",
        optimization => 3,
        target_arch => "x86_64",
        debug_mode => true
    }

    println!("\n기록: 재귀적 매크로를 통해 복잡한 설정을 위한 커스텀 DSL을 구현함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

🎓 선생님의 세부 가이드 (13.0-2): "재귀의 함정과 팁"
 * 재귀 한계 (Recursion Limit): 매크로가 너무 깊게 재귀를 돌면 컴파일러가 멈춥니다. 기본값은 보통 128회이며, 필요하다면 #[recursion_limit = "256"]처럼 늘릴 수 있습니다.
 * 가독성: 재귀 매크로는 강력하지만 복잡해지기 쉽습니다. @나 internal_ 같은 접두사를 사용하여 매크로 내부에서만 쓰이는 규칙임을 명시하는 것이 설계자의 매너입니다.
 * 토큰 트리(tt)의 유연성: $($rest:tt)*는 어떤 형태의 코드 조각도 받아낼 수 있는 가장 강력한 도구입니다. 이를 통해 아주 창의적인 문법을 창조할 수 있습니다.
> 저장 필수 너는 기록이 증명이다 gogs
