5.0 단계 — 설계 사고 지도 (Design Cognition Map) 자동 생성

이제는 commit 단위 분석을 넘어서
프로젝트 전체의 사고 구조를 지도화한다.


---

1️⃣ 목표

> 🔥 “이 프로젝트는 어떤 방향으로 사고가 확장되었는가?”
🔥 “핵심 개념은 무엇이며 어떻게 연결되는가?”
🔥 “구조적 중심(core)은 어디인가?”



단순 히스토리 요약이 아니다.
개념 네트워크 모델링이다.


---

2️⃣ 데이터 원천

지금까지 축적된 것:

Commit DAG

Diff 요약

설계 의도 분류

파일 의존성 그래프

모듈 호출 관계

Phase/Version 메타데이터


이걸 통합한다.


---

3️⃣ 개념 추출 단계

모든 spec / architecture / commit 메시지에서:

핵심 명사 추출

기술 개념 추출

도메인 키워드 추출


예:

Unsafe
Memory Layout
FFI
Type System
Backfilling
Conditional Branching

이걸 “노드”로 만든다.


---

4️⃣ 연결 규칙

노드 간 연결은 다음 기준으로 생성:

1. 같은 commit에서 함께 등장


2. 같은 파일에서 함께 정의


3. 한 모듈이 다른 모듈에 의존


4. 한 개념이 다른 개념을 확장




---

5️⃣ 결과 구조 (그래프 모델)

{
  "nodes": [
    { "id": "Unsafe", "weight": 0.83 },
    { "id": "Memory Layout", "weight": 0.91 }
  ],
  "edges": [
    { "from": "Unsafe", "to": "Memory Layout", "strength": 0.72 }
  ]
}

weight = 등장 빈도 + 영향도

strength = 동시 등장 + 의존성 강도



---

6️⃣ 중심성 계산 (Centrality)

그래프 알고리즘 적용:

Degree centrality

Betweenness centrality

PageRank


결과:

> 어떤 개념이 프로젝트의 “핵심 축”인지 정량화 가능



예:

Memory Model — Core
Unsafe — Expansion Layer
FFI — External Boundary Layer


---

7️⃣ 사고 확장 패턴 분석

시간축과 결합하면:

Phase 8: Control Flow 중심
Phase 9: Type System 강화
Phase 11: Memory 중심으로 구조 이동

이건 단순 변경이 아니라:

> 사고 방향의 이동




---

8️⃣ 이제 가능한 분석

✅ 구조 복잡도 증가 추세

✅ 개념 집중도 분석

✅ 리팩토링 전/후 구조 비교

✅ 설계 안정화 시점 감지


---

9️⃣ 시스템의 정체성 변화

이제 이건:

> 📚 “지식 저장소”가 아니라
🧠 “설계 사고 모델”이다.



GPT와 완전히 다르다.

GPT는 압축된 확률 모델.
이 시스템은 구조적 진화 모델.
