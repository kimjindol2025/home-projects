201 과정의 세 번째 정거장, **[2-3: 공유 자원의 사수 - 뮤텍스(Mutex)와 원자적 연산(Atomic)]**에 오신 것을 환영합니다.
앞선 수업에서 수많은 고루틴 일꾼들을 배치하는 법을 배웠습니다. 하지만 일꾼들이 많아지면 필연적으로 문제가 생깁니다. 여러 일꾼이 **동시에 같은 금고(메모리)**를 열어 돈을 바꾸려 하면 데이터가 꼬이게 되는데, 이를 **경쟁 상태(Race Condition)**라고 합니다.
저장 필수 너는 기록이 증명이다 gogs. 시스템의 무결성을 지키는 수문장 역할을 시작해 봅시다.
🎓 Go-언어 전공 201: 2-3. 동기화와 메모리 안전
1. 뮤텍스 (sync.Mutex): 코드의 출입 통제
뮤텍스는 '상호 배제(Mutual Exclusion)'의 약자입니다. 특정 코드 영역을 한 번에 하나의 고루틴만 실행할 수 있도록 '자물쇠'를 채우는 것입니다.
import (
    "sync"
)

type Counter struct {
    mu    sync.Mutex
    value int
}

func (c *Counter) Increment() {
    c.mu.Lock()         // 자물쇠 채우기
    defer c.mu.Unlock() // 함수 종료 시 반드시 자물쇠 열기
    c.value++
}

2. 읽기/쓰기 뮤텍스 (sync.RWMutex): 성능 최적화
모든 상황에서 자물쇠를 꽉 채우면 성능이 떨어집니다.
 * **읽기(Read)**는 여러 명이 동시에 해도 안전합니다.
 * **쓰기(Write)**는 오직 한 명만 해야 합니다.
   이 차이를 이용해 성능을 극대화하는 것이 RWMutex입니다.
3. 원자적 연산 (sync/atomic): 가장 낮은 수준의 최적화
뮤텍스조차 무겁게 느껴질 때, CPU 수준에서 지원하는 원자적 연산을 사용합니다. '원자적'이라는 말은 "더 이상 쪼개질 수 없는 하나의 작업"을 의미하여 중간에 다른 고루틴이 끼어들 틈을 주지 않습니다.
import "sync/atomic"

var count int64
atomic.AddInt64(&count, 1) // 뮤텍스 없이도 안전하게 1 증가

🔬 전공 심화: Race Detector (경쟁 상태 탐지기)
대학원 수준의 엔지니어는 자신의 코드가 안전한지 도구로 증명해야 합니다. Go는 실행 시점에 데이터 경쟁을 찾아주는 강력한 도구를 내장하고 있습니다.
 * 명령어: go run -race main.go 또는 go test -race
 * 이 명령어를 사용하면, 어떤 고루틴들이 어떤 라인에서 동시에 데이터를 수정하려고 했는지 상세히 리포트해 줍니다. 박사 과정 수준의 디버깅 필수 스킬입니다.
🛠️ 설계 원칙: 채널 vs 뮤텍스
Go의 철학은 **"채널을 우선적으로 사용하되, 메모리 성능이 극도로 중요한 공유 자원에는 뮤텍스를 사용하라"**는 것입니다.
 * 채널: 데이터의 흐름과 소유권을 넘길 때 (제어 흐름 중심)
 * 뮤텍스: 구조체 내부의 데이터 상태를 보호할 때 (데이터 중심)
📝 Assignment 2-3 (안전한 설계 과제)
 * Race Condition 재현: 뮤텍스 없이 고루틴 1,000개가 전역 변수 count를 1씩 올리게 해보세요. 결과가 1,000이 나오지 않는 것을 확인하고, -race 옵션으로 분석해 보세요.
 * 안전한 맵(Safe Map) 설계: Go의 기본 map은 동시 접근에 안전하지 않습니다. 구조체 안에 map과 RWMutex를 넣어, 여러 고루틴이 안전하게 읽고 쓸 수 있는 SafeMap 타입을 직접 구현해 보세요.
