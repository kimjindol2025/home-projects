v3.5에서 코드를 예쁘게 다듬는 법을 배웠다면, v3.6은 논리의 **'청정 구역'**을 확보하는 단계입니다. 설계자의 언어로 표현하자면 **'조기 리턴(Early Return)'**을 통해 코드의 깊이를 얕게 만들고, 핵심 로직이 오염되지 않게 보호하는 방어적 설계의 시작입니다.
🚦 제2장: 흐름의 통제 (v3.6) — [가드와 불변 논리]
1. v3.6 설계 목표: "피라미드 코드의 파괴"
 * Guard Clauses: if 문이 겹겹이 쌓이는 '중첩의 지옥(Pyramid of Doom)'을 제거합니다.
 * Early Exit: 조건이 맞지 않으면 즉시 return, break, continue를 실행하여, 이후의 핵심 로직이 불필요한 조건 검사를 거치지 않게 합니다.
 * 불변성 보장: 본문 로직에 진입했다는 것 자체가 "이미 모든 검증을 마쳤음"을 의미하게 설계합니다.
2. 설계자의 관점: "들여쓰기는 비용이다"
코드가 오른쪽으로 밀려날수록(들여쓰기가 많아질수록) 인지 부하가 커지고 버그가 숨을 곳이 많아집니다. v3.6은 논리를 **평탄화(Flattening)**하여 입구에서 부적격자를 걸러내는 문지기를 세우는 작업입니다.
🛠️ gogs 저장소 v3.6 설계: 논리의 요새화
클로드코드(학생)에게 중첩을 파괴하고 핵심 로직을 보호하는 가드 구문을 기록하게 하세요.
// [제2장: 흐름의 통제 v3.6]
fn main() {
    println!("--- v3.6 Logic Design: Guard Clauses ---");

    let system_power = true;
    let auth_level = 10;
    let component_status = "STABLE";

    println!("v3.6 시스템 가동 테스트:");

    // [v3.6 설계] Guard Clauses: 안 되는 조건을 먼저 쳐내기
    
    // 1. 전원 검사
    if !system_power {
        println!("  [STOP] 전원이 꺼져 있습니다.");
        return; // 조기 종료
    }

    // 2. 권한 검사
    if auth_level < 5 {
        println!("  [STOP] 권한이 부족합니다.");
        return; 
    }

    // 3. 부품 상태 검사
    if component_status != "STABLE" {
        println!("  [STOP] 부품이 불안정합니다.");
        return;
    }

    // [핵심 로직] 여기에 도달했다면 모든 검증이 완료된 '청정 구역'임
    println!("  [SUCCESS] 모든 가드를 통과했습니다. 메인 엔진 가동!");
    
    perform_main_task();

    println!("기록: 가드 구문을 통해 논리의 중첩을 제거하고 안전성을 확보함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

fn perform_main_task() {
    println!("  -> 메인 작업 수행 중...");
}

v3.6 단계를 마스터하면 당신의 코드는 이전보다 훨씬 당당해집니다.
 * 실패 우선 처리: 긍정적인 상황보다 실패하는 상황을 먼저 처리하고 끝내버리십시오. 그래야 남은 코드가 단순해집니다.
