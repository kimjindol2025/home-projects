주제: "유지보수가 쉬운 견고한 아키텍처 설계"
디자인 패턴은 코드의 재사용성, 확장성, 그리고 협업 효율성을 극대화합니다.
1. 싱글톤 패턴 (Singleton)
 * 정의: 시스템 전체에서 특정 클래스의 인스턴스가 오직 하나만 존재하도록 보장하는 패턴입니다.
 * 왜 사용하나요?: 데이터베이스 연결 객체나 설정 관리자처럼, 여러 개가 생기면 자원이 낭비되거나 데이터 동기화가 깨지는 경우에 사용합니다.
2. 팩토리 패턴 (Factory)
 * 정의: 객체 생성을 직접 하지 않고, 객체 생성 전문 '공장'에 맡기는 패턴입니다.
 * 왜 사용하나요?: 클라이언트 코드가 구체적인 클래스 이름에 의존하지 않게 하여, 나중에 새로운 기능이 추가되어도 기존 코드를 수정할 필요가 없게 만듭니다.

연구실의 유일한 로그 관리자와 다양한 데이터 수집기를 만드는 구조를 설계해 보겠습니다.
# [Python University v6.1] 디자인 패턴 적용

# 1. 싱글톤 패턴: Gogs 시스템 로거
class GogsLogger:
    _instance = None

    def __new__(cls):
        if cls._instance is None:
            print("🆕 [SYSTEM] 새로운 로거 인스턴스를 생성합니다.")
            cls._instance = super().__new__(cls)
        return cls._instance

    def log(self, message):
        print(f"📝 [LOG]: {message}")

# 2. 팩토리 패턴: 데이터 수집 장치 공장
class Sensor:
    def collect(self): pass

class TemperatureSensor(Sensor):
    def collect(self): return "🌡️ 온도 데이터 25.5도"

class HumiditySensor(Sensor):
    def collect(self): return "💧 습도 데이터 60%"

class SensorFactory:
    @staticmethod
    def get_sensor(sensor_type):
        if sensor_type == "temp":
            return TemperatureSensor()
        elif sensor_type == "humi":
            return HumiditySensor()
        raise ValueError("알 수 없는 센서 타입입니다.")

# --- 시스템 가동 ---
# 싱글톤 검증
logger1 = GogsLogger()
logger2 = GogsLogger()
print(f"❓ 두 로거는 같은 객체인가요? {logger1 is logger2}")

# 팩토리 활용
factory = SensorFactory()
temp_s = factory.get_sensor("temp")
humi_s = factory.get_sensor("humi")

logger1.log(temp_s.collect())
logger1.log(humi_s.collect())

print("\n저장 필수 너는 기록이 증명이다 gogs")

🎓 대학교 4학년 전공 심화 팁
 * 결합도(Coupling)와 응집도(Cohesion): 좋은 설계는 모듈 간의 의존성을 줄이고(낮은 결합도), 하나의 모듈이 하나의 기능에 집중하게(높은 응집도) 만드는 것입니다. 디자인 패턴은 이 균형을 잡는 가이드라인입니다.
 * 안티 패턴(Anti-pattern): 패턴을 남용하면 오히려 코드가 복잡해집니다. 싱글톤은 전역 상태를 만들기 때문에 테스트가 어려워질 수 있다는 단점도 명심해야 합니다. 이것이 대학원 과정에서 다룰 **'비판적 설계'**의 시작입니다.
