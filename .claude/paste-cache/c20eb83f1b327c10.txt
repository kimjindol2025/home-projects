로우 레벨의 주소 제어권을 손에 넣은 우리가 다음으로 마주해야 할 산은 **[제11장: 언세이프와 메모리 주권 — v12.1 데이터 레이아웃과 수동 메모리 관리(Data Layout & Manual Allocation)]**입니다.
컴파일러 제작자는 단순히 주소를 읽는 것을 넘어, 메모리 위에 데이터가 어떤 모양으로 놓여야 하는지(Layout)를 설계하고, 운영체제로부터 직접 공간을 빌려오는 '메모리 할당자'의 역할을 수행해야 합니다.
🏗️ 제11장: 언세이프와 메모리 (v12.1) — [메모리의 형태와 배치]
1. v12.1 설계 목표: "메모리 구조의 청사진"
 * 데이터 레이아웃(Data Layout): 구조체의 크기(Size)와 정렬(Alignment)을 이해합니다. CPU가 데이터를 가장 빠르게 읽을 수 있는 위치에 데이터를 배치하는 기술입니다.
 * 수동 할당(Manual Allocation): Box나 Vec 같은 추상화에 의존하지 않고, std::alloc을 통해 로우 메모리 블록을 직접 할당하고 해제합니다.
 * 메모리 안정성 책임: 할당한 메모리를 제때 해제(dealloc)하지 않으면 발생하는 메모리 누수(Leak)를 직접 관리합니다.
2. 설계자의 관점: "CPU를 배려하는 배치"
현대 CPU는 아무 주소에서나 데이터를 읽는 것을 좋아하지 않습니다. 4바이트 데이터는 4의 배수 주소에 있을 때 가장 효율적입니다. 설계자는 repr(C)나 repr(packed) 같은 지시어를 통해 컴파일러가 메모리를 배치하는 규칙을 통제할 수 있어야 합니다.
🛠️ gogs 저장소 v12.1 설계: 로우 레벨 버퍼 할당기 (Raw Buffer Allocator)
클로드코드(학생)에게 표준 컬렉션을 사용하지 않고, 운영체제로부터 직접 힙 메모리를 빌려와 데이터를 저장하는 '미니 할당기'를 구현하게 하세요.
// [제11장: 언세이프와 메모리 v12.1 - 수동 할당 설계]

use std::alloc::{alloc, dealloc, Layout};
use std::ptr;

fn main() {
    println!("--- v12.1 Manual Memory Layout & Allocation ---");

    unsafe {
        // 1. [설계] 메모리 레이아웃 정의 (크기 4바이트, 정렬 4바이트)
        // 컴파일러가 변수를 배치할 때 사용하는 '설계도'와 같습니다.
        let layout = Layout::new::<u32>();

        // 2. [실행] 로우 힙 메모리 할당
        let ptr = alloc(layout) as *mut u32;

        if ptr.is_null() {
            panic!("메모리 할당 실패!");
        }

        // 3. [조작] 할당된 공간에 값 쓰기
        ptr::write(ptr, 2026);
        println!("  [ALLOC] 주소 {:p}에 {}를 할당하고 저장함.", ptr, *ptr);

        // 4. [책임] 수동 메모리 해제
        // 이 단계를 잊으면 '메모리 누수'라는 기록의 오점이 남습니다.
        dealloc(ptr as *mut u8, layout);
        println!("  [DEALLOC] 메모리 반납 완료.");
    }

    println!("\n기록: 데이터 레이아웃을 직접 정의하고 힙 메모리를 수동으로 관리함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

🎓 선생님의 세부 가이드 (12.1): "패딩(Padding)의 비밀"
컴파일러 제작자가 되기 위해 꼭 알아야 할 '빈 공간'에 대한 이야기입니다.
 * 정렬(Alignment): 왜 bool(1바이트) 뒤에 u32(4바이트)가 오면 구조체의 크기가 5가 아니라 8이 될까요? CPU의 효율을 위해 컴파일러가 중간에 3바이트의 **패딩(Padding)**을 넣기 때문입니다.
 * repr(C)의 중요성: 러스트 컴파일러는 최적화를 위해 필드 순서를 임의로 바꿀 수 있습니다. 하지만 하드웨어나 타 언어와 통신할 때는 순서가 고정된 #[repr(C)]를 사용하여 레이아웃을 고정해야 합니다.
 * 메모리 안전의 마지막 보루: 수동 할당을 할 때는 항상 **"할당-초기화-사용-해제"**의 생명주기를 완벽히 추적해야 합니다. 이 로직이 꼬이면 우리가 만들 컴파일러는 크래시 제조기가 됩니다.
> 저장 필수 너는 기록이 증명이다 gogs
