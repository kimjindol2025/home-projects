📊 7개 커밋 구조 분석

1️⃣ v2.K.1 — Sequential Conservative

PolicyDecision 도입

AgentPolicy 인터페이스 분리

Kernel_applyPolicyDecision() 집행 전용

Retry 1회 / Replan 금지 / MaxParallel=1


👉 Kernel/Policy 1차 분리 완성


---

2️⃣ v2.K.2 — Optimistic Replan (초기)

Replan 도입

Execution → Planner 역방향 호출

PlanDelta 적용


👉 실행 중 구조 변경 가능성 증명


---

3️⃣ v2.K.2 (수정: MaxReplan=1)

Replan 무한 루프 차단

Retry 1 + Replan 1로 제한


👉 정책 폭주 제어


---

4️⃣ v2.K.3 — Scheduler Policy

selectNext() 완전 분리

FIFO / Priority 전략 구현

ReadyQueue는 unordered


👉 Kernel이 “선택”을 완전히 모르게 됨


---

5️⃣ v2.K.4 — Parallel Policy

MaxParallel 도입

ExecutionPool 통합

Running Task 추적

Scheduler + Parallel 결합


👉 실행 모델이 순차 → 동시성 확장


---

🏗 현재 아키텍처 상태

Planner  →  DAG
             ↓
Kernel  (집행만)
             ↓
Policy  (Retry/Replan/Parallel/Scheduler)
             ↓
Constraint (Hard Stop)

Kernel 수정 0줄 유지 ✔
테스트 전부 통과 ✔
전략 교체 가능 ✔
Race 없음 ✔


---

🎯 이 구조의 의미

이제 Kernel은:

> “결정하지 않는 실행 엔진”



Policy는:

> “전략을 정의하는 교체 가능한 뇌”



이건 단순 리팩토링이 아니라
Agent Runtime 아키텍처 완성 단계다.


---

⚠️ 단 하나 남은 구조적 리스크

Replan + Parallel 조합은 아직 증명 안 됨.

현재:

Sequential + Replan ✔

Parallel + Conservative ✔

Scheduler + Parallel ✔


하지만:

> Parallel + Replan 동시 사용 시
DAG 변경 중 실행 중인 Task 정합성?



이건 아직 테스트 로그에 안 보인다.
