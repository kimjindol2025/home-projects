드디어 우리는 러스트 아키텍처의 정점이자, 일반적인 객체 지향 언어에서는 흉내조차 내기 힘든 [제10장: 객체 지향과 패턴 — v11.2 타입 상태 패턴(Type State Pattern)] 단계에 도달했습니다.
v11.1의 상태 패턴이 런타임에 유연하게 변하는 방식이었다면, v11.2는 '상태'를 아예 '타입' 그 자체로 박제해버립니다. 이를 통해 설계자는 잘못된 상태에서의 함수 호출을 런타임 에러가 아닌 컴파일 에러로 차단하는 '무결한 설계'를 완성하게 됩니다.
🎭 제10장: 객체 지향과 패턴 (v11.2) — [타입 상태: 컴파일 타임 마법]
1. v11.2 설계 목표: "불가능한 상태를 표현 불가능하게 만들기"
 * Generic State: 구조체에 제네릭 타입을 부여하여 현재 어떤 상태인지 컴파일러에게 명시합니다.
 * Consuming Methods: 메서드가 self를 소유권째로 가져가 파괴하고, 새로운 상태의 타입을 반환하게 설계합니다.
 * Zero-Cost Abstraction: 런타임 체크(if문 등)가 사라지므로 성능은 극대화되면서 안전성은 완벽해집니다.
2. 설계자의 관점: "상태 전이는 곧 타입 변환이다"
설계자는 객체를 하나의 생명체로 봅니다. "번데기" 상태의 객체는 "날기" 메서드가 없습니다. 오직 "나비" 상태로 **변환(Transformation)**된 후에야 비로소 그 기능을 가질 수 있습니다. 이를 통해 개발자의 실수를 원천적으로 봉쇄합니다.
🛠️ gogs 저장소 v11.2 설계: 고성능 보안 미사일 발사 시스템
클로드코드(학생)에게 '장전-조준-발사'의 엄격한 순서를 어길 시 아예 컴파일조차 되지 않는 철통 보안 시스템을 설계하게 하세요.
// [제10장: 객체 지향 패턴 v11.2 - 타입 상태 패턴 설계]

// 1. 상태를 나타내는 '빈 구조체' (Zero-sized types)
struct Idle;
struct Targeted { coordinates: String }
struct Armed;

// 2. 미사일 시스템: 현재 상태(S)를 타입 매개변수로 가짐
struct MissileSystem<S> {
    state: S,
}

// 3. 초기 상태(Idle)에서의 설계
impl MissileSystem<Idle> {
    fn new() -> Self {
        MissileSystem { state: Idle }
    }

    // Idle 상태에서만 'target'을 지정하여 Targeted 상태로 변환 가능
    fn set_target(self, coord: &str) -> MissileSystem<Targeted> {
        println!("  [SYSTEM] 좌표 {} 조준 완료.", coord);
        MissileSystem { state: Targeted { coordinates: coord.to_string() } }
    }
}

// 4. Targeted 상태에서만 가능한 설계
impl MissileSystem<Targeted> {
    fn arm(self) -> MissileSystem<Armed> {
        println!("  [SYSTEM] 발사 대기 완료. 안전 장치 해제.");
        MissileSystem { state: Armed }
    }
}

// 5. Armed 상태에서만 'fire' 가능
impl MissileSystem<Armed> {
    fn fire(self) {
        println!("  [SYSTEM] 미사일 발사! 기록이 증명한다 gogs.");
    }
}

fn main() {
    println!("--- v11.2 The Magic of Type State Pattern ---");

    // 정상 흐름
    let system = MissileSystem::new();
    let targeted = system.set_target("37.5N, 127.0E");
    let armed = targeted.arm();
    armed.fire();

    /* [설계자의 승리: 컴파일 에러 케이스]
    let system2 = MissileSystem::new();
    system2.fire(); // 에러! Idle 상태에는 fire() 메서드가 존재하지 않음.
    
    let system3 = MissileSystem::new();
    let armed3 = system3.arm(); // 에러! Idle 상태에서는 바로 무장할 수 없음.
    */

    println!("\n기록: 타입 상태 패턴을 통해 논리적 순서를 컴파일 타임에 강제함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

🎓 선생님의 세부 가이드 (11.2): "설계의 정점"
타입 상태 패턴을 실전에서 활용할 때 기억할 점입니다.
 * 메모리 효율: 이 모든 상태 구조체들은 런타임에 메모리를 차지하지 않습니다(ZST). 오직 컴파일러의 '체크'용도로만 사용됩니다.
 * API의 명확성: 사용자는 .을 찍는 순간 현재 상태에서 호출 가능한 함수만 추천받게 됩니다. 문서가 필요 없는 직관적인 API가 탄생합니다.
 * 유연성 부족: 런타임에 상태가 계속 변해야 하는 경우(예: 비디오 플레이어의 재생/정지)는 v11.1의 런타임 상태 패턴이 더 적합합니다. 타입 상태 패턴은 **'한 방향으로 흐르는 엄격한 절차'**에 최적화되어 있습니다.
