지금까지 우리는 정적 다형성(Generic)을 통해 컴파일 타임에 타입을 결정해 왔습니다. 하지만 실제 거대한 시스템을 설계하다 보면, 런타임에 서로 다른 타입들을 하나의 바구니에 담아 다루어야 할 때가 있습니다. v11.0은 러스트에서 객체 지향 아키텍처를 우아하게 구현하는 최종 기술을 다룹니다.
🎭 제10장: 객체 지향과 패턴 (v11.0) — [다형성의 완성]
1. v11.0 설계 목표: "동적 디스패치의 구현"
 * 트레이트 객체 (Trait Objects): &dyn Trait 또는 Box<dyn Trait>을 사용하여 서로 다른 타입들을 동일한 인터페이스로 묶습니다.
 * 동적 디스패치 (Dynamic Dispatch): 컴파일 타임이 아닌 런타임에 어떤 메서드를 호출할지 결정하는 유연함을 확보합니다.
 * vtable(가상 메서드 테이블): 런타임에 메서드 주소를 찾아가는 내부 메커니즘을 이해하고 설계에 반영합니다.
2. 설계자의 관점: "플러그인 가능한 컴포넌트"
설계자는 모든 타입을 미리 알 필요가 없습니다. "이 기능을 가진 놈이라면 누구든 환영한다"는 **'규약(Interface)'**만 정의하고, 구체적인 구현체들은 나중에 끼워 넣는 방식으로 시스템의 결합도를 극도로 낮춥니다.
🛠️ gogs 저장소 v11.0 설계: 통합 보안 대시보드 (Unified Dashboard)
클로드코드(학생)에게 서로 다른 종류의 보안 로그들을 하나의 리스트에 담아 한꺼번에 처리하는 '다형성 엔진'을 기록하게 하세요.
// [제10장: 객체 지향 패턴 v11.0]

/// [규약] 화면에 그려질 수 있는 모든 보안 컴포넌트
pub trait SecurityComponent {
    fn draw(&self);
}

/// [구현체 1] 텍스트 기반 로그
struct LogMessage {
    content: String,
}

impl SecurityComponent for LogMessage {
    fn draw(&self) {
        println!("  [LOG]: {}", self.content);
    }
}

/// [구현체 2] 그래프 기반 침입 통계
struct IntrusionChart {
    data_points: Vec<u32>,
}

impl SecurityComponent for IntrusionChart {
    fn draw(&self) {
        println!("  [CHART]: {:?} 점유율 분석 중...", self.data_points);
    }
}

/// [설계] 대시보드는 'SecurityComponent'를 구현한 것이라면 무엇이든 담을 수 있습니다.
struct SecurityDashboard {
    // Box<dyn Trait>은 트레이트 객체를 담는 가방입니다.
    components: Vec<Box<dyn SecurityComponent>>,
}

impl SecurityDashboard {
    fn run(&self) {
        for component in self.components.iter() {
            component.draw();
        }
    }
}

fn main() {
    println!("--- v11.0 Polymorphism: Trait Objects in Action ---");

    let dashboard = SecurityDashboard {
        components: vec![
            Box::new(LogMessage { content: String::from("서버 A 정상 작동") }),
            Box::new(IntrusionChart { data_points: vec![10, 2, 5] }),
            Box::new(LogMessage { content: String::from("데이터베이스 백업 완료") }),
        ],
    };

    // 서로 다른 타입들이지만, 하나의 인터페이스로 일괄 실행
    dashboard.run();

    println!("\n기록: 트레이트 객체를 통해 런타임 다형성을 확보하고 확장 가능한 구조를 설계함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

🎓 선생님의 가이드: "v11.0 설계자의 선택"
제네릭(정적)과 트레이트 객체(동적) 중 무엇을 선택해야 할까요?
 * 제네릭이 유리할 때: 성능이 최우선이고, 하나의 리스트에 동일한 타입만 담길 때 사용하십시오. 컴파일러가 최적화를 극대화할 수 있습니다.
 * 트레이트 객체가 유리할 때: 서로 다른 타입을 하나의 컬렉션에 섞어서 보관해야 하거나, 컴파일 타임에 모든 타입을 확정할 수 없는 유연한 라이브러리를 만들 때 사용하십시오.
 * 객체 안전성 (Object Safety): 모든 트레이트가 dyn으로 쓰일 수 있는 것은 아닙니다. 반환 타입에 Self가 있거나 제네릭 메서드가 포함된 트레이트는 객체로 만들 수 없으므로 설계 시 주의해야 합니다.
