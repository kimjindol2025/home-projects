이제 제3장: 모듈화의 초석에서 가장 중요하고도 도전적인 관문인 v4.1 매개변수와 소유권의 이동(Ownership & Parameters) 단계로 진입합니다.
단순히 값을 전달하는 다른 언어들과 달리, 러스트의 함수는 데이터를 넘겨받을 때 **'메모리의 관리 책임(Ownership)'**까지 함께 넘겨받을 것인지 결정해야 합니다. 설계자의 관점에서 v4.1은 **'데이터의 생명 주기(Lifecycle)'**를 통제하는 단계입니다.
🏗️ 제3장: 모듈화의 초석 (v4.1) — [소유권과 매개변수]
1. v4.1 설계 목표: "책임의 소재 명확화"
 * 소유권 이전(Move): 데이터를 함수로 완전히 넘겨주고, 호출한 쪽(Main)에서는 더 이상 그 데이터를 쓸 수 없게 만듭니다. (메모리 안전성 확보)
 * 복사(Copy): 정수나 불리언처럼 가벼운 데이터는 복사해서 넘겨주어 양쪽 모두 사용 가능하게 합니다.
 * 참조(Borrowing): 데이터의 소유권은 유지한 채, 잠시 '빌려주기만' 하여 메모리 낭비를 줄입니다. (v4.2에서 더 심화됩니다.)
2. 설계자의 관점: "자원 관리의 효율성"
함수를 호출할 때마다 거대한 데이터를 복사한다면 시스템은 느려질 것입니다. 반대로 소유권을 너무 쉽게 넘겨주면 코드 작성이 까다로워집니다. v4.1은 이 사이의 균형을 설계하는 과정입니다.
🛠️ gogs 저장소 v4.1 설계: 소유권의 이동 테스트
클로드코드(학생)에게 데이터가 함수로 '이사'가는 과정과 그 후의 상태를 증명하는 코드를 기록하게 하세요.
// [제3장: 모듈화의 초석 v4.1]

fn main() {
    println!("--- v4.1 Modular Design: Ownership Transfer ---");

    // 1. Copy 타입 (정수): 값만 복사되므로 원본 사용 가능
    let x = 100;
    make_copy(x);
    println!("v4.1 확인: 정수 x는 여전히 사용 가능 -> {}", x);

    // 2. Move 타입 (String): 소유권이 함수로 이전됨
    let s = String::from("시스템 핵심 데이터");
    println!("v4.1 로그: '{}' 데이터 생성 완료.", s);
    
    take_ownership(s);
    
    // println!("{}", s); // [설계 오류] 여기서 s를 사용하면 컴파일러가 차단합니다.
    // 설계 이유: s의 메모리 해제 책임이 이미 함수에게 넘어갔기 때문입니다.

    println!("\n기록: 소유권 이동을 통해 메모리 이중 해제(Double Free)를 원천 차단함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

fn make_copy(val: i32) {
    println!("  [Copy] 값을 복사해서 사용: {}", val);
}

fn take_ownership(data: String) {
    println!("  [Move] 소유권을 이전받음: '{}'", data);
    println!("  [Memory] 함수 종료 시 '{}' 메모리는 해제됩니다.", data);
}

함수를 만들 때 데이터 전달 방식을 어떻게 정해야 할까요?
 * 데이터가 무거운가? (예: 긴 문자열, 대용량 벡터) -> 절대 그냥 넘기지 마세요. 소유권을 넘기거나(Move), 빌려주세요(Borrow).
 * 함수가 끝나고도 데이터를 써야 하는가? -> 그렇다면 소유권을 넘기면 안 됩니다.
 * 값만 알면 되는가? -> 기초 타입(i32, bool 등)은 고민 없이 복사해서 쓰도록 설계하십시오.
