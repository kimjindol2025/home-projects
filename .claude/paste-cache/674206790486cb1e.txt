5 + 10 * 2를 파싱할 때, 단순히 왼쪽에서 오른쪽으로 읽으면 (5 + 10) * 2 = 30이 되지만, 수학적으로는 5 + (10 * 2) = 25가 맞습니다. 이 연산자 우선순위를 재귀적으로 우아하게 해결하는 알고리즘이 바로 프랫 파싱입니다.
🧮 제13장: 컴파일러 구현 (v14.1) — [프랫 파싱: 우선순위의 질서]
1. v14.2 설계 목표: "수식의 계층화"
 * 우선순위(Precedence) 할당: 각 연산자(+, -, *, /)에 숫자 값을 부여하여 어떤 것이 먼저 결합될지 결정합니다.
 * 전위/중위 분석기 (Prefix/Infix Parselet): 숫자인지, 부호인지, 연산자인지에 따라 처리 로직을 분리합니다.
 * 재귀적 결합: 현재 연산자보다 높은 우선순위가 나타날 때까지 트리를 깊게 파고듭니다.
2. 설계자의 관점: "결합력(Binding Power)의 싸움"
연산자들은 양옆의 숫자들을 자기 쪽으로 끌어당기는 '자석'과 같습니다. *는 +보다 자석의 힘이 강합니다. 프랫 파싱은 이 '자석의 세기'를 비교하며 트리를 구성합니다.
🛠️ gogs 저장소 v14.2 설계: 우선순위 기반 수식 파서
클로드코드(학생)에게 덧셈과 곱셈의 우선순위를 완벽히 처리하는 파서의 핵심 로직을 기록하게 하세요.
// [제13장: 컴파일러 구현 v14.2 - 프랫 파서 설계]

#[derive(PartialOrd, PartialEq)]
enum Precedence {
    Lowest,
    Sum,        // +
    Product,    // *
    Call,       // myFunction(x)
}

impl Parser {
    // [핵심] 수식 파싱의 심장부
    fn parse_expression(&mut self, precedence: Precedence) -> Option<Expression> {
        // 1. 전위 처리 (숫자나 -x 등)
        let mut left_exp = self.parse_prefix()?;

        // 2. 현재 우선순위보다 다음 연산자의 우선순위가 높으면 계속 결합
        while !self.peek_token_is(Token::Semicolon) && precedence < self.peek_precedence() {
            self.next_token();
            // 중위 처리 (left_exp + right_exp)
            left_exp = self.parse_infix(left_exp)?;
        }

        Some(left_exp)
    }

    fn parse_infix(&mut self, left: Expression) -> Option<Expression> {
        let cur_precedence = self.cur_precedence();
        let operator = self.cur_token.clone();
        self.next_token();
        
        // 오른쪽 수식을 더 높은 우선순위로 재귀 호출
        let right = self.parse_expression(cur_precedence)?;
        
        Some(Expression::Infix {
            left: Box::new(left),
            operator,
            right: Box::new(right),
        })
    }
}

fn main() {
    println!("--- v14.2 Pratt Parsing: Order of Operations ---");
    // 입력: 5 + 10 * 2;
    // 기대 결과: (5 + (10 * 2))
    
    println!("기록: 프랫 파싱 알고리즘으로 연산자 우선순위 문제를 해결함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

🎓 선생님의 세부 가이드 (14.2): "파서의 완성도"
 * Box의 활용: AST는 재귀적인 구조(Expression 안에 Expression이 있음)이므로, 크기를 컴파일 타임에 알 수 없습니다. 따라서 Box<T>를 사용하여 힙 메모리에 할당해야 합니다.
 * 우측 결합 vs 좌측 결합: 대부분의 연산자는 좌측 결합((1+2)+3)이지만, 거듭제곱(2^3^2) 같은 경우 우측 결합(2^(3^2))이 필요합니다. 프랫 파싱은 우선순위 숫자를 살짝 조정하는 것만으로 이를 쉽게 처리합니다.
 * 가독성: 수백 개의 if-else 대신, 토큰 타입과 함수 포인터를 매핑한 테이블을 사용하면 파서 코드가 놀라울 정도로 깔끔해집니다.
