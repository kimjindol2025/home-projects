🟢 Phase 5.5 — Typed Execution Architecture

이 단계의 본질은:

> “VM을 Generic Interpreter → Typed Interpreter로 전환”




---

🔥 핵심 설계 원칙 (중요)

1️⃣ Generic Op는 완전히 제거하지 않는다

Add        // fallback
AddNum     // fast path
AddStr     // fast path

이유:

REPL / dynamic / any 타입 존재

Phase 6 Agent 런타임과 충돌 방지



---

2️⃣ Typed Op는 컴파일 타임 결정

절대 VM에서 분기하지 않는다.

// ❌ 금지
if (typeof a === "number")

// ✅ 허용
emit(Op.AddNum)


---

3️⃣ VM은 Branchless 방향으로 간다

Typed Op는:

타입 체크 없음

typeof 없음

instanceof 없음



---

📐 제안 아키텍처

1️⃣ Op 분류

Arithmetic

AddNum

SubNum

MulNum

DivNum


Compare

EqNum

LtNum

GtNum


String

AddStr

EqStr


Array

LoadArrayNum

StoreArrayNum

LoadArrayGeneric


Object

LoadPropStatic

LoadPropDynamic



---

🧠 Compiler 책임 확대

컴파일러가 결정:

TypeInfo → Op Selection → Optimized Bytecode

TypeChecker 실패 → 컴파일 중단
TypeInference unknown → Generic Op 사용


---

🧪 테스트 전략

Layer 1 — Bytecode 검증

num + num → AddNum 생성 확인

str + str → AddStr 생성 확인

unknown + num → AddGeneric 생성 확인


Layer 2 — VM 안전성

런타임 typeof 제거 확인

모든 타입 에러는 컴파일 단계에서 발생


Layer 3 — 성능 측정

Add 10M 루프 비교

기존 Op 대비 속도 차이 측정



---

⚠️ 위험 요소

1. Op 폭증 (enum 관리 문제)


2. 배열/객체 최적화가 GC/Unsafe와 충돌 가능


3. any 타입 전략을 명확히 정의해야 함




---

🏗 결정 필요 사항

Phase 5.5 설계 시작 전에 다음 중 하나를 선택해야 합니다:

1. Arithmetic만 Typed Op 적용 (점진적 도입)


2. Arithmetic + Compare까지 포함


3. VM 전체 Typed 전환 (대수술)
