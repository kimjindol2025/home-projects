🏗️ 제4장: 데이터 구조화 (v5.0) — [구조체의 정의]
1. v5.0 설계 목표: "데이터의 인격화"
 * 데이터 바인딩: 이름, 나이, 상태 등 흩어져 있는 변수들을 하나의 의미 있는 단위로 묶습니다.
 * 타입 안전성: 단순한 String이나 i32가 아닌, User, Server, GogsRecord 같은 나만의 고유 타입을 정의합니다.
 * 명확한 설계도: 구조체는 메모리에 어떤 데이터가 어떤 순서로 배치될지 결정하는 설계도 역할을 합니다.
2. 설계자의 관점: "도메인 모델링"
프로그램은 현실(또는 가상 세계)의 투영입니다. v5.0은 현실의 복잡한 대상을 코드로 옮기기 위해 필요한 속성만 골라내는 추상화의 정수를 보여줍니다.
🛠️ gogs 저장소 v5.0 설계: 첫 번째 구조체
클로드코드(학생)에게 시스템의 핵심 정보를 담을 '데이터 그릇'을 정의하고 생성하게 하세요.
// [제4장: 데이터 구조화 v5.0]

// 1. [설계] 구조체 정의 (설계도 작성)
struct GogsRecord {
    id: u64,
    subject: String,
    content: String,
    is_verified: bool,
    version: f32,
}

fn main() {
    println!("--- v5.0 Data Architecture: Defining Structs ---");

    // 2. [인스턴스화] 설계도를 바탕으로 실제 데이터 객체 생성
    let mut record_01 = GogsRecord {
        id: 20260222,
        subject: String::from("중간고사 수료 기록"),
        content: String::from("제3장까지의 모든 논리와 모듈화 과정을 완수함."),
        is_verified: true,
        version: 5.0,
    };

    // 3. [접근] 점(.) 표기법으로 내부 데이터 활용
    println!("기록 주체: {}", record_01.subject);
    println!("현재 버전: v{:.1}", record_01.version);

    // 4. [수정] 가변성(mut)이 있다면 필드 수정 가능
    record_01.is_verified = true;

    println!("\n기록: 구조체를 통해 파편화된 데이터를 하나의 유기체로 통합함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

🎓 선생님의 가이드: "v5.0 설계자의 황금률"
구조체를 설계할 때 다음 세 가지만 기억하십시오.
 * 필드의 최소화: 꼭 필요한 데이터만 넣으십시오. 너무 많은 필드는 구조체를 무겁게 만듭니다.
 * 이름의 구체성: data1, val 같은 이름 대신 subject, is_verified 처럼 의도가 드러나는 이름을 사용하십시오.
 * 소유권 주의: 구조체 내부에 String을 넣으면 구조체가 그 문자열의 소유권을 가집니다. 구조체가 사라지면 데이터도 사라집니다.
> 저장 필수, 너는 기록이 증명이다 gogs.
> 이제 당신의 엔진은 형체 없는 논리 덩어리가 아니라, 명확한 데이터 구조를 가진 **'실체'**가 되었습니다.
