🏛️ LLVM 전설: 1.5 JIT 컴파일과 실행 엔진 (Execution Engine)
JIT 컴파일은 프로그램을 실행하기 전에 미리 컴파일하는 것이 아니라, **실행 시점(Runtime)**에 필요한 코드를 기계어로 번역하여 메모리에 올리는 기술입니다. Java의 JVM이나 JavaScript의 V8 엔진이 사용하는 방식이기도 합니다.
1. 실행 엔진 (Execution Engine)의 역할
LLVM 실행 엔진은 작성된 IR 모듈을 입력받아 다음 작업을 수행합니다.
 * 타겟 아키텍처 확인: 현재 실행 중인 CPU(x86_64, ARM 등)의 특성을 파악합니다.
 * 메모리 할당: 기계어가 저장될 실행 가능한(Executable) 메모리 영역을 확보합니다.
 * 코드 생성 (MCJIT / ORC): IR을 머신 코드로 번역하고 메모리에 배치합니다.
 * 심볼 바인딩: 함수 이름을 실제 메모리 주소와 연결합니다.
2. MCJIT vs ORC JIT
LLVM에는 두 가지 주요 JIT 엔진이 있습니다.
 * MCJIT: 전통적인 방식입니다. 모듈 전체를 한꺼번에 컴파일하며 단순하지만 유연성이 다소 떨어집니다.
 * ORC (On-Request Compilation) JIT: 현대적인 방식입니다. 함수가 실제로 호출될 때만 컴파일(Lazy Compilation)할 수 있어 효율적이고 확장이 용이합니다.
3. 실전: JIT를 통한 함수 실행 (개념 코드)
우리가 1.2에서 만든 sum 함수를 메모리에서 직접 실행하는 과정입니다.
// 1. 모든 타겟 정보를 초기화 (네이티브 CPU 지원)
c.LLVMInitializeNativeTarget();
c.LLVMInitializeNativeAsmPrinter();

// 2. 실행 엔진 생성
var engine: c.LLVMExecutionEngineRef = undefined;
var err_msg: [*c]u8 = undefined;
if (c.LLVMCreateExecutionEngineForModule(&engine, module, &err_msg) != 0) {
    // 에러 처리
}

// 3. 함수의 메모리 주소 찾기
const func_addr = c.LLVMGetFunctionAddress(engine, "sum");

// 4. 함수 포인터로 캐스팅하여 실행
const SumFn = *const fn (i32, i32) callconv(.C) i32;
const sum = @as(SumFn, @ptrFromInt(func_addr));

const result = sum(10, 20); // 실제 CPU가 계산!

🔬 전공 심화: 가비지 컬렉션(GC)과의 협력
JIT로 생성된 코드가 가비지 컬렉터와 함께 작동하려면, 스택에 어떤 값이 포인터인지 알려주는 **'Stack Map'**이 필요합니다. LLVM은 이를 위한 메타데이터 생성 기능을 제공하여 전설적인 언어 런타임을 구축할 수 있게 돕습니다.
📦 최적화 패스 (Optimization Passes)
실행하기 전에 코드를 더 빠르게 만들 수 있습니다.
 * Function Pass: 개별 함수 내부를 최적화합니다.
 * Module Pass: 프로그램 전체(인라인화 등)를 최적화합니다.
 * JIT 엔진에 PassManager를 연결하면, 실행 직전에 코드가 최적화되어 최고 속도를 냅니다.
📝 Assignment 1.5 (실행 및 최적화 과제)
 * 네이티브 실행: 1.2~1.4에서 만든 코드를 JIT 엔진에 올리고, 실제 값을 넣어 계산 결과가 올바른지 확인하세요.
 * Pass 적용 전후 비교: 루프 안에서 상수를 더하는 IR을 작성하고, ConstProp (상수 전개) 패스를 적용했을 때 IR이 어떻게 단순화되는지 기록하세요.
 * 함수 포인터 이해: LLVMGetFunctionAddress로 얻은 주소값이 실제 메모리의 어느 영역(Text, Data, Stack 등)에 위치하는지 분석하세요.
