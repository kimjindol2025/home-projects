---

✅ 재설계 5단계

🔵 1단계 — HeapObject 전환 (메모리 모델 교체)

이 단계는 모든 것의 기초다.
GC, Unsafe, 동시성 전부 여기 위에 올라간다.


---

🎯 1단계 목표

현재:

Value = { tag, val }

변경 후:

Value = pointer → HeapObject

모든 값은 HeapObject 기반으로 통일.


---

🧠 핵심 설계 원칙

1. Stack에는 절대 raw value 없음


2. Primitive도 모두 박싱(Boxed)


3. HeapObject가 단일 진실 소스


4. GC 메타는 Value가 아니라 HeapObject에 존재




---

🏗 1단계 최종 구조

1️⃣ HeapObject 정의

interface HeapObject {
  id: number
  typeId: number
  refCount: number
  gcMark: boolean
  flags: number
  payload: any
}


---

2️⃣ TypeInfo 테이블 도입

interface TypeInfo {
  id: number
  name: string
  size: number
  isPrimitive: boolean
}

VM 내부:

TypeTable: Map<number, TypeInfo>


---

3️⃣ Value 재정의

type Value = {
  ptr: number   // HeapObject id
}

Value는 더 이상 실제 데이터를 가지지 않는다.


---

🔄 마이그레이션 순서 (절대 순서 변경 금지)

STEP 1 — Heap 레이어 생성

HeapManager 생성

allocate(typeId, payload)

get(ptr)

free(ptr)


⚠️ 기존 Value 구조 유지한 채 Heap만 먼저 도입


---

STEP 2 — Primitive 박싱 시작

기존:

{ tag: "num", val: 42 }

변경:

heap.allocate(TYPE_NUM, 42)

VM 연산은 전부 heap.get(ptr) 통해 접근.

이 단계에서 테스트 30~40% 깨질 가능성 있음.


---

STEP 3 — 모든 Value를 pointer 기반으로 전환

배열 내부 값 → pointer

object map 값 → pointer

function closure → pointer

iterator → pointer


이 단계가 가장 위험.


---

STEP 4 — refCount 기본 구현

allocate → refCount = 1
assign → increment
overwrite → decrement

zero → free

이 시점에 GC는 아직 없음.


---

STEP 5 — Root Set 추적 구조 추가

Root 후보:

Global variables

Current CallStack

Closure refs

Builtin env


Root 목록을 VM이 관리하도록 변경.


---

📋 완료 기준

✔ 모든 값이 HeapObject 기반
✔ Stack에 primitive 없음
✔ refCount 정상 동작
✔ 기존 테스트 100% 통과
✔ 메모리 누수 없음 (단순 케이스)


---

🚨 실패 신호

double free 발생

closure 해제 후 접근 가능

Stack에 raw 값 남아 있음

pointer aliasing 추적 불가



---

⏳ 예상 소요

2~4주 (디버깅 포함)


---

📊 1단계 완료 후 얻는 것

항목    개선 전    개선 후

GC 가능성    불가    가능
Unsafe 격리    어려움    가능
타입 강화    제한적    중앙화 가능
동시성 준비    불가    기반 확보
