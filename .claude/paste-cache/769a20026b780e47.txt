1. [40] Embedding: 상속을 넘어선 합성 (Composition)
Go는 클래스 상속이 없습니다. 대신 구조체 안에 다른 구조체를 넣는 '임베딩'을 사용합니다. 이는 "객체는 무엇인가(Is-a)"보다 **"객체가 무엇을 가지고 있는가(Has-a)"**에 집중하는 설계 방식입니다.
type Engine struct {
    Horsepower int
}

func (e Engine) Start() { fmt.Println("엔진 가동!") }

type Car struct {
    Engine // 임베딩 (이름 없이 타입만 선언)
    Model  string
}

func main() {
    myCar := Car{Engine: Engine{Horsepower: 300}, Model: "Gopher-GT"}
    // Car는 Engine의 필드와 메서드에 직접 접근 가능
    fmt.Println(myCar.Horsepower) 
    myCar.Start() 
}

 * 핵심: Car는 Engine을 포함함으로써 그 기능을 자신의 것처럼 사용합니다. 다중 임베딩을 통해 여러 기능을 유연하게 결합할 수 있습니다.
2. [45] Goroutines: 초경량 쓰레드
OS 쓰레드가 수 MB의 메모리를 사용하는 반면, 고루틴은 단 몇 KB로 시작합니다. 수만 개를 동시에 띄워도 시스템에 무리가 가지 않는 비결입니다.
func task(name string) {
    fmt.Println(name, "작업 중...")
}

func main() {
    go task("비동기-1") // 'go' 키워드 하나로 고루틴 생성
    task("동기-2")
    // 메인 함수가 종료되면 고루틴도 종료되므로 주의!
}

3. [46, 48] Channels & Select: 고루틴 간의 대화
"메모리를 공유하여 통신하지 말고, 통신하여 메모리를 공유하라"는 Go의 대원칙을 실현하는 도구입니다.
 * Channel: 고루틴 사이의 데이터를 전달하는 파이프라인.
 * Select: 여러 채널 중 데이터가 준비된 쪽을 먼저 처리하는 분기문.
<!-- end list -->
func main() {
    ch1 := make(chan string)
    ch2 := make(chan string)

    go func() { ch1 <- "데이터 A" }()
    go func() { ch2 <- "데이터 B" }()

    // 먼저 도착하는 데이터를 처리
    select {
    case msg1 := <-ch1:
        fmt.Println("수신:", msg1)
    case msg2 := <-ch2:
        fmt.Println("수신:", msg2)
    case <-time.After(time.Second): // 타임아웃 처리도 가능
        fmt.Println("시간 초과")
    }
}

🔬 전공 심화 통찰: 왜 이 조합인가?
대학원 수준의 시스템을 설계할 때, 이 세 가지는 다음과 같이 연결됩니다.
 * Embedding으로 부품화된 객체를 만듭니다.
 * Goroutine으로 각 부품을 동시에 독립적으로 실행합니다.
 * Channel & Select로 부품들 사이의 데이터를 안전하게 조율합니다.
이것이 바로 구글이 수조 건의 요청을 처리하는 서버를 만드는 방식입니다.
