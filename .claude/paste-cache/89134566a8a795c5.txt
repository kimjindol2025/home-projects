우리는 제9장 동시성의 숲을 지나왔고, 이제 그 거대한 시스템을 지탱하는 가장 근본적인 '자격'에 대해 기록할 차례입니다. **[v10.3 동시성의 설계 철학: Send와 Sync 트레이트]**입니다.
v10.3은 단순히 코드를 짜는 단계를 넘어, 러스트 컴파일러가 어떻게 **"이 데이터는 스레드를 넘나들어도 안전하다"**고 판단하는지 그 내부 로직을 이해하는 과정입니다.
⚡ 제9장: 두려움 없는 동시성 (v10.3) — [안전의 근간: 마커 트레이트]
1. v10.3 설계 목표: "스레드 경계의 수호자"
 * Send (이동의 자격): 해당 타입의 소유권을 다른 스레드로 전송해도 안전한지 결정합니다.
 * Sync (공유의 자격): 여러 스레드가 해당 타입의 **참조(&T)**를 동시에 공유해도 안전한지 결정합니다.
 * 자동 구현 (Auto Traits): 우리가 직접 구현하지 않아도, 타입의 구성 요소가 모두 Send나 Sync라면 컴파일러가 자동으로 자격을 부여합니다.
2. 설계자의 관점: "불가능을 사전에 차단하기"
설계자는 왜 Rc<T>는 스레드 간에 보낼 수 없고, Arc<T>는 가능한지 그 이유를 알아야 합니다. Rc<T>는 참조 카운트를 올릴 때 일반 연산을 사용하므로, 여러 스레드가 동시에 접근하면 카운트가 꼬여 메모리 사고가 납니다. 컴파일러는 이를 감지하고 !Send 딱지를 붙여 사고를 미연에 방지합니다.
🛠️ gogs 저장소 v10.3 설계: 자격 증명 시스템 (The Proof of Trait)
클로드코드(학생)에게 어떤 타입이 스레드 안전한지 확인하고, 이를 설계에 반영하는 법을 기록하게 하세요.
// [제9장: 두려움 없는 동시성 v10.3]

use std::thread;
use std::rc::Rc;
use std::sync::Arc;

fn main() {
    println!("--- v10.3 The Core of Concurrency: Send & Sync ---");

    // 1. [실패 예시] Rc는 Sync가 아니므로 스레드 간 공유가 불가능합니다.
    let rc_data = Rc::new(5);
    // thread::spawn(move || {
    //     println!("{}", rc_data); // 컴파일 에러: `Rc<i32>` cannot be sent between threads safely
    // });

    // 2. [성공 설계] Arc는 내부적으로 원자적(Atomic) 연산을 하므로 Send + Sync입니다.
    let arc_data = Arc::new(100);
    let child = thread::spawn(move || {
        println!("  [서브 스레드] Arc 데이터 접근: {}", arc_data);
    });

    child.join().unwrap();

    println!("\n설계 노트:");
    println!(" - Send: 타입 자체가 스레드 사이를 이동할 수 있는가?");
    println!(" - Sync: &'a T를 여러 스레드에서 안전하게 사용할 수 있는가?");
    println!(" - 결론: T가 Sync라면, &T는 Send입니다.");

    println!("\n저장 필수 너는 기록이 증명이다 gogs");
}

🎓 선생님의 가이드: "v10.3 설계자의 최종 통찰"
동시성 설계를 마무리하며 이 두 트레이트의 비밀을 가슴에 새기십시오.
 * 마커 트레이트의 마법: Send와 Sync는 메서드가 하나도 없습니다. 그저 **"이 타입은 안전함"**이라는 표시(Marker)일 뿐입니다.
 * 수동 구현의 위험성: unsafe impl Send for MyType {}과 같이 강제로 자격을 부여할 수도 있지만, 이는 컴파일러의 보호를 끄는 행위입니다. 반드시 완벽한 메모리 지식이 있을 때만 시도하십시오.
 * 데이터 경합의 종말: 러스트가 "데이터 경합 없음"을 보장하는 비결은 바로 이 Send와 Sync가 엄격하게 스레드 간의 데이터 흐름을 감시하기 때문입니다.
