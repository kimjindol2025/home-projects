드디어 Zig의 가장 깊고도 정교한 영역인 **'메모리 제어'**의 장에 오신 것을 환영합니다. 저장 필수 너는 기록이 증명이다 gogs.
C언어가 강력하면서도 위험했던 이유는 포인터가 "자유롭지만 통제되지 않았기" 때문입니다. Zig는 이 자유를 유지하되, 설계자의 실수를 컴파일 타임과 런타임에 잡아내는 장치들을 마련해두었습니다.
🎓 Zig 전공 101: 1-6. 포인터(Pointers)와 메모리 관리
Zig의 포인터는 그 용도에 따라 명확히 구분됩니다. "이 포인터가 데이터 하나를 가리키는지, 아니면 배열의 시작점인지"를 타입 시스템이 알고 있습니다.
1. 포인터의 기본 (*T)
가장 기본적인 포인터는 단일 아이템을 가리킵니다.
 * *T: 타입 T인 데이터 하나를 가리키는 포인터.
 * & 연산자: 변수의 주소를 가져옵니다.
 * .* 연산자: 포인터가 가리키는 실제 값을 참조(Dereference)합니다.
<!-- end list -->
var x: i32 = 42;
const ptr: *i32 = &x; // x의 주소를 저장

ptr.* = 100; // 포인터를 통해 값을 변경

2. 포인터의 종류 (Pointer Categories)
Zig는 포인터의 성격에 따라 문법이 다릅니다.
| 포인터 타입 | 설명 | 특징 |
|---|---|---|
| *T | Single-item | 딱 1개의 요소만 가리킴. 산술 연산(ptr + 1) 불가. |
| [*]T | Many-item | 여러 요소를 가리킴 (C와 유사). 인덱싱 가능. |
| []T | Slice | 포인터([*]T) + 길이(usize)의 결합체. 가장 안전함. |
3. 상수 포인터와 가변 포인터
포인터 자체의 가변성과 포인터가 가리키는 값의 가변성을 구분해야 합니다.
 * var ptr: *i32: 포인터가 가리키는 주소를 바꿀 수 있음.
 * const ptr: *i32: 포인터가 가리키는 주소를 바꿀 수 없음.
 * const ptr: *const i32: 포인터가 가리키는 주소도, 그 안의 값도 바꿀 수 없음.
🔬 전공 심화: 안전한 메모리 접근 (Optional Pointers)
Zig에서는 포인터가 null이 될 수 있는지 여부를 타입으로 강제합니다.
 * *i32: 절대 null이 될 수 없습니다. (C의 NULL 참조 에러 방지)
 * ?*i32: null이 될 수 있는 옵셔널 포인터입니다. 사용 전 반드시 체크해야 합니다.
<!-- end list -->
var ptr: ?*i32 = null;

if (ptr) |p| {
    // ptr이 null이 아닐 때만 이 블록이 실행되며, p는 *i32 타입이 됨
    p.* = 50;
}

📦 메모리 관리의 핵심: 할당자 (Allocator)
Zig에는 '숨겨진 할당'이 없습니다. 모든 메모리 할당은 할당자 객체를 통해 명시적으로 이루어집니다.
 * std.heap.page_allocator: OS로부터 직접 페이지를 할당받음.
 * std.testing.allocator: 테스트 시 메모리 누수(Leak)를 자동으로 감지함.
📝 Assignment 1-6 (메모리 설계 과제)
 * 포인터 조작: 정수 변수 a를 선언하고, 그 주소를 포인터 p에 담으세요. p를 통해 a의 값을 변경한 뒤 출력해 보세요.
 * 옵셔널 연습: ?*i32 타입의 변수를 만들고 null을 할당해 보세요. if (ptr) |p| 구문을 사용하여 안전하게 값을 확인하는 코드를 작성하세요.
 * 기록: *i32 타입에 null을 직접 대입하려고 할 때 발생하는 컴파일 에러 메시지를 기록하세요. 이것이 왜 안전한 설계인지 스스로 증명하는 과정입니다.
