201 과정의 두 번째 정거장, **[2-2: 병렬 데이터 파이프라인 - Fan-out/Fan-in 패턴]**에 오신 것을 환영합니다.
고루틴이 '일꾼'이라면, 채널은 '컨베이어 벨트'입니다. 대학원 수준의 시스템 설계에서는 일꾼 한 명에게 일을 시키는 것이 아니라, 수천 명의 일꾼을 효율적으로 배치하고 결과물을 다시 수집하는 병렬 프로세싱 아키텍처를 설계해야 합니다. 저장 필수 너는 기록이 증명이다 gogs.
🎓 Go-언어 전공 201: 2-2. 채널의 심화와 동시성 패턴
1. 버퍼드 채널(Buffered Channels)의 전략적 활용
기본 채널은 송신자와 수신자가 동시에 준비되어야 하는 '동기 방식'입니다. 하지만 버퍼를 주면 비동기적인 처리가 가능해집니다.
 * 학술적 관점: 버퍼는 시스템의 **처리량(Throughput)**을 높이지만, 버퍼가 가득 차면 결국 블로킹(Blocking)이 발생합니다. 이 지점이 시스템의 병목 구간이 됩니다.

2. Fan-out / Fan-in 패턴
대규모 데이터를 처리할 때 사용하는 가장 대표적인 Go의 디자인 패턴입니다.
 * Fan-out (분산): 하나의 채널에서 여러 고루틴이 데이터를 읽어 병렬로 작업을 수행합니다.
 * Fan-in (수집): 여러 고루틴에서 처리된 결과들을 하나의 채널로 모아서 최종 결과물을 만듭니다.
3. 실전 코드 아키텍처 (Worker Pool)
func worker(id int, jobs <-chan int, results chan<- int) {
    for j := range jobs {
        fmt.Printf("일꾼 %d 작업 시작: %d\n", id, j)
        time.Sleep(time.Second) // 무거운 작업 시뮬레이션
        results <- j * 2
    }
}

func main() {
    const numJobs = 5
    jobs := make(chan int, numJobs)
    results := make(chan int, numJobs)

    // 1. Fan-out: 일꾼 3명 배치
    for w := 1; w <= 3; w++ {
        go worker(w, jobs, results)
    }

    // 2. 일감 던지기
    for j := 1; j <= numJobs; j++ {
        jobs <- j
    }
    close(jobs) // 더 이상 보낼 데이터가 없음을 알림

    // 3. Fan-in: 결과 수집
    for a := 1; a <= numJobs; a++ {
        <-results
    }
}

🔬 전공 심화: 채널의 Close와 상태 관리
채널을 닫는 행위는 **"더 이상 보낼 데이터가 없다"**는 신호입니다.
 * 닫힌 채널에 데이터를 보내면 Panic이 발생합니다.
 * 닫힌 채널로부터 데이터를 읽으면, 남은 데이터를 다 읽은 후 해당 타입의 Zero Value를 무한히 반환합니다.
 * 설계 원칙: 항상 송신측에서 채널을 닫아야 하며, 수신측에서는 v, ok := <-ch의 ok 값을 통해 채널 상태를 확인해야 합니다.
