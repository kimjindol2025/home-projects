⏳ 제6장: 수명 (v7.4) — [통합 설계: 삼위일체]
1. v7.4 설계 목표: "제약 조건의 오케스트레이션"
 * 수명과 제네릭의 결합: 구조체가 제네릭 타입 T를 가지면서, 동시에 외부 참조자 &'a를 보유할 때의 상호작용을 설계합니다.
 * 트레이트 바운드와 수명: "이 제네릭 타입 T는 적어도 수명 'a 보다는 오래 살아야 한다"는 T: 'a (수명 하한선) 개념을 적용합니다.
 * 완벽한 캡슐화: 복잡한 수명 계산을 구조체 내부로 숨기고, 사용자에게는 단순한 인터페이스만 제공합니다.
2. 설계자의 관점: "안전한 유연성"
유연함(Generic)은 자칫 느슨함으로 이어질 수 있습니다. 수명(Lifetime)은 그 느슨함에 **'시간적 제약'**을 걸어 시스템이 무너지지 않게 지탱하는 뼈대가 됩니다.
🛠️ gogs 저장소 v7.4 설계: 범용 보안 감사 엔진 (General Audit Engine)
클로드코드(학생)에게 제네릭 데이터와 참조 데이터를 동시에 다루는 최상위 보안 모듈을 설계하게 하세요.
// [제6장: 수명 v7.4 - 최종 통합 설계]

use std::fmt::Display;

/// [설계] '삼위일체' 구조체
/// 'a : 참조 데이터의 수명
/// T  : 분석할 데이터의 제네릭 타입
/// T: 'a -> "T는 적어도 'a 수명만큼은 살아있어야 한다"는 제약 (Subtyping)
struct GogsAuditor<'a, T> where T: Display + 'a {
    target: &'a T,           // 분석 대상 (참조)
    prefix: &'a str,         // 로그 접두어 (참조)
}

impl<'a, T> GogsAuditor<'a, T> where T: Display + 'a {
    fn audit(&self) {
        println!("{}: [검사 중] -> {}", self.prefix, self.target);
    }
}

fn main() {
    println!("--- v7.4 Final Integration: The Trinity ---");

    // 1. 소유권을 가진 데이터 (T)
    let secret_data = String::from("CONFIDENTIAL_ACCESS_77");
    let header = "SEC-LOG";

    // 2. 통합 엔진 가동
    // 여기서 T는 String이고, 'a는 secret_data와 header의 수명이 됩니다.
    let auditor = GogsAuditor {
        target: &secret_data,
        prefix: header,
    };

    auditor.audit();

    println!("\n기록: 제네릭, 트레이트, 수명을 단일 구조로 통합하여 무결점 감사 시스템을 구축함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

🎓 선생님의 가이드: "v7.4 아키텍트의 마지막 조언"
이 통합 설계를 마스터할 때 당신은 진정한 러스트 설계자로 거듭납니다.
 * T: 'a 의 의미를 새기십시오: 제네릭 T가 내부적으로 참조자를 가지고 있을 수도 있습니다. 이때 T: 'a라고 선언하면, T 안에 숨겨진 모든 참조자들도 'a보다는 오래 살 것임을 보장하게 됩니다.
 * 가독성의 타협: 선언부가 struct Name<'a, T: Trait + 'a> 처럼 복잡해지면, 앞서 배운 where 절을 적극적으로 활용하여 설계의 의도를 명확히 하십시오.
 * 수명은 도구일 뿐: 수명이 당신을 괴롭힌다면, 설계가 너무 복잡하다는 신호일 수 있습니다. 때로는 소유권을 넘기는 방식(String, Box)으로 설계를 단순화하는 것도 실력입니다.
