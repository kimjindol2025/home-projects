스마트 포인터 설계의 정점이자, 러스트에서 가장 강력한 동적 데이터 구조를 형성하는 [제8장: 스마트 포인터 — v9.3 Rc<RefCell<T>>: 다중 소유 가변성 (Multiple Ownership with Mutability)] 단계에 오신 것을 환영합니다.
v9.1의 Rc<T>(다중 소유)와 v9.2의 RefCell<T>(내부 가변성)가 결합하면, "여러 곳에서 소유하면서 동시에 내용을 수정할 수 있는" 마법 같은 설계가 가능해집니다. 이는 복잡한 그래프, GUI 위젯 트리, 혹은 서로 얽혀 있는 실전 데이터를 다룰 때 필수적인 기술입니다.
🧠 제8장: 스마트 포인터 (v9.3) — [결합: 공유와 수정의 공존]
1. v9.3 설계 목표: "동적 데이터 네트워크의 완성"
 * 복합 스마트 포인터: Rc<RefCell<T>> 구조를 통해 런타임에 소유권을 공유하면서도 데이터를 안전하게 변경합니다.
 * 순환 참조의 인식: 데이터가 서로를 가리킬 때 발생하는 메모리 누수 위험을 인지하고 설계에 반영합니다.
 * 유연한 노드 설계: 어떤 노드에서든 연결된 다른 노드의 데이터를 수정할 수 있는 구조를 확립합니다.
2. 설계자의 관점: "중앙 통제 없는 협업"
전통적인 설계에서는 데이터를 고치려면 반드시 주인이 허락해야 했습니다. 하지만 Rc<RefCell<T>> 설계에서는 데이터를 공유하는 누구라도 **'대여 규칙'**만 지킨다면 데이터를 업데이트할 수 있습니다. 이는 시스템의 모듈 간 결합도를 낮추고 자율성을 부여합니다.
🛠️ gogs 저장소 v9.3 설계: 동적 보안 노드 네트워크
클로드코드(학생)에게 여러 개의 입구가 하나의 핵심 보안 설정을 공유하고 수정하는 '분산 제어 시스템'을 설계하게 하세요.
// [제8장: 스마트 포인터 v9.3]

use std::rc::Rc;
use std::cell::RefCell;

#[derive(Debug)]
struct SecurityConfig {
    active_protocol: String,
}

#[derive(Debug)]
struct AdminPanel {
    // 여러 패널이 하나의 설정을 공유하고 수정함
    config: Rc<RefCell<SecurityConfig>>,
}

fn main() {
    println!("--- v9.3 Smart Pointers: Rc<RefCell<T>> Integration ---");

    // 1. 공유 설정 데이터 생성 (중앙 저장소)
    let shared_config = Rc::new(RefCell::new(SecurityConfig {
        active_protocol: String::from("BETA"),
    }));

    // 2. 서로 다른 관리 패널이 동일한 설정을 소유함
    let panel_a = AdminPanel { config: Rc::clone(&shared_config) };
    let panel_b = AdminPanel { config: Rc::clone(&shared_config) };

    println!("  [초기 상태] 패널 A: {:?}", panel_a.config.borrow().active_protocol);

    // 3. 패널 B에서 프로토콜을 변경함 (패널 A에도 즉시 반영됨)
    {
        let mut config_mut = panel_b.config.borrow_mut();
        config_mut.active_protocol = String::from("ALPHA-GOLD");
    }

    println!("  [변경 후] 패널 A: {:?}", panel_a.config.borrow().active_protocol);
    println!("  [참조 카운트]: {}", Rc::strong_count(&shared_config));

    println!("\n기록: Rc<RefCell<T>> 결합을 통해 다중 소유와 내부 가변성을 동시에 확보함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

🎓 선생님의 가이드: "v9.3 설계자의 안전 장치"
이 강력한 도구를 사용할 때 설계자가 반드시 체크해야 할 리스트입니다.
 * 순환 참조 주의 (Reference Cycles): A가 B를 소유하고 B가 A를 소유하면 참조 카운트가 영원히 0이 되지 않아 메모리 누수가 발생합니다. 이를 방지하려면 다음 단계에서 배울 Weak<T>를 고려해야 합니다.
 * 대여 수명 최소화: borrow_mut()으로 얻은 가변 참조자는 필요한 순간에만 쓰고 즉시 반납(스코프 종료)하십시오. 그렇지 않으면 다른 곳에서 접근할 때 런타임 패닉이 발생합니다.
 * 트레이드오프: 이 방식은 편리하지만 런타임 오버헤드가 발생합니다. 성능이 극도로 중요한 구간에서는 소유권을 명확히 분리하는 단순한 설계를 우선 검토하십시오.
