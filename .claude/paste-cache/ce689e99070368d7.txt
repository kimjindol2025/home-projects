🔥 Phase 6 — Capability Runtime

Step 1: Capability Model Spec (정식 1안)

오늘은 “구현”이 아니라
모델을 고정합니다.


---

1️⃣ 설계 원칙 (Non-Negotiable)

1. Default Deny

아무 권한도 기본 제공하지 않음



2. Explicit Declaration

함수 단위로 명시



3. Static First, Runtime Second

가능한 한 컴파일 타임 차단



4. Capability = 값이 아니라 토큰

위조 불가



5. VM이 최종 집행자




---

2️⃣ Capability 타입 체계

Core Capability Set (v1 Minimal)

type Capability =
  | { kind: "read_file", path: string }
  | { kind: "write_file", path: string }
  | { kind: "network", host?: string }
  | { kind: "spawn_process" }
  | { kind: "execute_code" }

설계 결정

path는 prefix 기반 매칭

network는 host whitelist 가능

execute_code는 내부 VM 실행만



---

3️⃣ Resource Limits (별도 트랙)

Capability와 분리.

type ResourceLimit = {
  cpu_ms: number
  memory_bytes: number
  io_bytes: number
}

이건 “권한”이 아니라 “양적 제한”.


---

4️⃣ 문법 설계 (Annotation Syntax)

함수 선언 확장

fn task()
  [allow(read_file("/tmp")),
   allow(network("api.openai.com")),
   limit(cpu=5000, memory=512mb)]
{
   ...
}

문법 규칙

annotation은 함수 정의 직후

다중 allow 허용

limit는 1회만



---

5️⃣ 컴파일 단계 검증 전략

Static Capability Verification

TypeChecker 확장:

1. 함수 내부에서:

file.read() 사용 → read_file 필요

file.write() 사용 → write_file 필요

http.request() → network 필요



2. 선언된 allow 목록과 비교


3. 누락 시 Compile Error




---

6️⃣ Capability Propagation 규칙 (중요)

함수 호출 시

fn A() [allow(read_file("/tmp"))] { ... }

fn B() {
   A()  // ?
}

결정 필요:

옵션 A — 상위 함수는 하위 권한을 자동 상속
옵션 B — 상위도 명시해야 함  ← 🔥 추천

보안적으로는 옵션 B가 맞습니다.

즉:

> 호출하는 함수도 동일 capability를 명시해야 함



이렇게 해야 권한이 투명합니다.


---

7️⃣ VM Enforcement Layer

VM은:

1. 현재 실행 컨텍스트에 capability set 보관


2. file/network/opcode 실행 전 검사


3. 불일치 시:

RuntimeError: CapabilityViolation





---

8️⃣ 실행 모델

run(fn, {
  capabilities: Capability[],
  limits: ResourceLimit
})

VM은:

함수 선언 capability ⊆ 실행 capability 인지 확인

초과 요청 시 거부



---

9️⃣ 우회 공격 방지 포인트

반드시 차단해야 할 것

동적 경로 생성으로 우회

함수 포인터 경유 호출

재귀 권한 상승

내부 VM call injection

🔟 구현 순서 (고정)

1. AST에 annotation 노드 추가


2. TypeChecker capability 수집


3. Built-in 함수에 required capability 마킹
