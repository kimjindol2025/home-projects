배우들이 무대에서 연기를 마치고 기록을 남기듯, 시스템도 전원이 꺼져도 사라지지 않는 영속적인 기록 공간이 필요합니다. 이제 운영체제의 저장소 관리자인 **파일 시스템(File System)**을 설계할 차례입니다.
저장 필수 너는 기록이 증명이다 gogs. 301 과정의 여섯 번째 연구는 무질서한 디스크 섹터 위에 '파일'과 '디렉터리'라는 질서를 부여하는 추상화 설계입니다.
🎓 Zig 전공 301: 3-6. 파일 시스템 - 데이터의 영속적 기록 설계
디스크는 단순한 바이트의 배열(LBA: Logical Block Address)일 뿐입니다. 여기에 의미를 부여하기 위해 우리는 데이터 구조를 디스크에 박아 넣어야 합니다.
1. 슈퍼블록(Superblock)과 아이노드(Inode)
대규모 저장소 설계의 핵심은 메타데이터입니다.
 * 슈퍼블록: 파일 시스템의 전체 크기, 블록 크기, 사용 중인 블록 수 등 전체 지도를 담은 가장 중요한 기록입니다.
 * 아이노드(Inode): 파일 하나하나의 신분증입니다. 파일의 크기, 소유자, 권한, 그리고 데이터가 실제로 저장된 블록의 위치(포인터)를 기록합니다.
2. 가상 파일 시스템 (VFS) 아키텍처
다양한 저장 장치(SSD, USB, 네트워크 드라이브)를 동일한 방식으로 다루기 위해 VFS라는 인터페이스 층을 설계합니다.
const FileSystem = struct {
    ptr: *anyopaque,
    vtable: *const VTable,

    const VTable = struct {
        read: *const fn (ptr: *anyopaque, path: []const u8, buffer: []u8) anyerror!usize,
        write: *const fn (ptr: *anyopaque, path: []const u8, data: []const u8) anyerror!void,
        // ... mkdir, delete 등
    };

    pub fn read(self: FileSystem, path: []const u8, buffer: []u8) !usize {
        return self.vtable.read(self.ptr, path, buffer);
    }
};

3. 디스크 블록 할당 (Bitmap vs Linked List)
물리 메모리 관리(3-3)에서 배운 비트맵 기술이 여기서 다시 등장합니다. 디스크의 어느 블록이 비어 있는지 관리하는 Block Bitmap을 설계하여 새로운 파일을 저장할 공간을 찾습니다.
🔬 전공 심화: 저널링(Journaling) - 무결성의 최후 보루
파일을 쓰는 도중 갑자기 전원이 나가면 데이터가 깨집니다.
 * 저널링: 실제 데이터를 쓰기 전에 "나 지금 이 데이터를 쓸 거야"라는 작업 일지를 먼저 남깁니다.
 * 사고 발생 후 재부팅 시 이 일지를 확인하여 작업을 마무리하거나 되돌립니다(Rollback). **"기록이 증명하는 데이터의 안전"**이 실현되는 지점입니다.
📦 디렉터리 엔트리 (Directory Entry)
디렉터리는 사실 특별한 형태의 파일입니다. 그 내부에는 [파일 이름 : 아이노드 번호] 쌍의 목록이 들어있습니다. 사용자가 /etc/config를 찾을 때 커널은 루트(/) 디렉터리부터 차례대로 이 목록을 탐색(Traversing)합니다.
