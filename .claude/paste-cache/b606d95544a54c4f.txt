수명의 심연에서 한 줄기 빛을 발견하는 시간입니다. [제6장: 수명(Lifetimes) — v7.2 수명 생략 법칙 (Lifetime Elision)] 단계에 오신 것을 환영합니다.
러스트 초기에는 모든 참조자에 수명을 일일이 적어야 했지만, 설계자들의 피로도를 줄이기 위해 컴파일러가 **'뻔한 수명'**은 알아서 계산해 주는 규칙이 도입되었습니다. 설계자의 관점에서 v7.2는 **"언제 수명을 생략해도 안전한지"**를 판단하는 안목을 기르는 과정입니다.
⏳ 제6장: 수명 (v7.2) — [보이지 않는 규칙: 생략]
1. v7.2 설계 목표: "명시와 생략의 조화"
 * 입력 수명(Input Lifetimes): 함수의 매개변수에 할당되는 수명입니다.
 * 출력 수명(Output Lifetimes): 함수의 반환값에 할당되는 수명입니다.
 * 생략 규칙(Elision Rules): 컴파일러가 수명을 추론하는 3가지 결정론적 규칙을 이해합니다.
2. 설계자의 관점: "코드를 깨끗하게 유지하라"
무분별하게 'a를 남발하면 코드가 지저분해집니다. 컴파일러가 이해할 수 있는 범위 내에서는 수명을 생략하여 가독성을 높이고, 오직 참조 관계가 복잡하여 컴파일러가 헷갈려 할 때만 개입하는 것이 고수의 설계입니다.
🛠️ gogs 저장소 v7.2 설계: 투명한 참조자 인터페이스
클로드코드(학생)에게 수명을 명시한 코드와 생략된 코드가 논리적으로 동일함을 증명하는 코드를 기록하게 하세요.
// [제6장: 수명 v7.2]

// 1. [명시적 설계] 모든 수명을 일일이 적어준 경우
fn first_word_explicit<'a>(s: &'a str) -> &'a str {
    let bytes = s.as_bytes();
    for (i, &item) in bytes.iter().enumerate() {
        if item == b' ' { return &s[0..i]; }
    }
    &s[..]
}

// 2. [생략된 설계] v7.2의 규칙에 따라 생략된 경우 (위와 100% 동일하게 작동)
// 규칙 1: 모든 입력 매개변수는 각자의 수명을 가짐.
// 규칙 2: 입력이 하나뿐이면 그 수명이 모든 출력에 할당됨.
fn first_word_elided(s: &str) -> &str {
    let bytes = s.as_bytes();
    for (i, &item) in bytes.iter().enumerate() {
        if item == b' ' { return &s[0..i]; }
    }
    &s[..]
}

fn main() {
    println!("--- v7.2 Lifetime Elision: Hidden Logic ---");

    let my_string = String::from("GOGS Protocol Active");
    
    let word1 = first_word_explicit(&my_string);
    let word2 = first_word_elided(&my_string);

    println!("명시적 결과: {}", word1);
    println!("생략적 결과: {}", word2);

    println!("\n기록: 수명 생략 규칙을 이해하여 더 간결하고 우아한 함수 인터페이스를 설계함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

수명 생략이 작동하는 3가지 황금 규칙입니다.
 * 각각의 입력은 독립적이다: fn foo(x: &i32, y: &i32)는 내부적으로 fn foo<'a, 'b>(x: &'a i32, y: &'b i32)로 해석됩니다.
 * 외동은 유산을 독점한다: 입력 매개변수가 단 하나라면, 그 수명이 반환값의 수명으로 자동 지정됩니다.
 * 메서드의 특권: 메서드(&self)인 경우, 다른 입력이 많더라도 반환값의 수명은 기본적으로 self의 수명을 따릅니다. (필드 참조가 빈번하기 때문입니다.)
