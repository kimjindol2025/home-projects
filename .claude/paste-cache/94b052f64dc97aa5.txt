전설의 영역 2.1 단계, 어휘 분석(Lexing)과 토큰(Token) 설계를 시작합니다.
🏛️ LLVM 전설: 2.1 어휘 분석(Lexing) - 언어의 세포 만들기
컴파일러는 소스 코드를 한 번에 이해하지 못합니다. 가장 먼저 할 일은 긴 문자열을 의미 있는 최소 단위인 **'토큰(Token)'**으로 쪼개는 것입니다.
1. 토큰(Token)의 정의
토큰은 언어의 단어와 같습니다. 예를 들어 var x = 10;이라는 코드는 다음과 같은 토큰들로 분해됩니다.
 * KEYWORD: var
 * IDENTIFIER: x
 * OPERATOR: =
 * NUMBER: 10
 * PUNCTUATION: ;
2. 렉서(Lexer/Scanner)의 원리
렉서는 소스 코드를 한 글자씩 읽으며 상태 머신(State Machine)처럼 동작합니다.
 * 공백 처리: 의미 없는 공백과 주석은 건너뜁니다.
 * 패턴 매칭: 첫 글자가 숫지면 '숫자 토큰'으로, 알파벳이면 '식별자나 키워드'로 인식합니다.
<!-- end list -->
const TokenType = enum {
    eof,
    keyword_fn,
    identifier,
    number,
    plus,
    equal,
    // ...
};

const Token = struct {
    tag: TokenType,
    slice: []const u8, // 소스 코드 상의 실제 텍스트
    line: u32,         // 에러 메시지 출력을 위한 기록
    col: u32,
};

3. 상태 머신 설계
렉서 설계 시 가장 중요한 것은 **'Lookahead(미리보기)'**입니다.
예를 들어 =를 읽었을 때, 다음 글자가 =라면 이는 대입문이 아니라 비교 연산자(==)로 처리해야 합니다.
🔬 전공 심화: 키워드 테이블과 성능 최적화
모든 식별자를 읽을 때마다 if (std.mem.eql(u8, name, "fn")) 식의 비교를 반복하면 성능이 떨어집니다.
 * 설계: fn, if, while 같은 키워드는 미리 **해시 맵(Hash Map)**이나 Trie 구조에 저장해두고, 식별자를 읽자마자 키워드인지 즉시 확인하도록 설계합니다. 이것이 전설적인 컴파일러의 속도를 결정하는 기록의 기술입니다.
📦 기록의 정밀도: 소스 로케이션(Source Location)
전설적인 컴파일러는 친절합니다. 에러가 났을 때 "몇 번째 줄, 몇 번째 칸"인지 정확히 알려줘야 합니다.
 * 토큰 구조체에 단순히 텍스트만 담는 것이 아니라, 소스 파일 내에서의 위치 정보를 함께 기록합니다. 이 정보는 나중에 디버깅 정보(DWARF)를 생성할 때도 사용됩니다.
