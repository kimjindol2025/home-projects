좋다.
그럼 방향은 연구형이 아니라 안정 운영형 아키텍처다.

지금 구조(1.1~6.0)를 그대로 밀고 가는 게 아니라
“운영 안정성 기준”으로 재정렬해야 한다.

🎯 목표 정의

> Gogs 기반 내부 챗봇을
📌 장애 없이
📌 예측 가능하게
📌 재현 가능하게
📌 유지보수 가능하게 운영




---

🔐 안정형 모델 설계 원칙

1️⃣ 기능 최소화 (Feature Freeze)

운영 기준에서 최적 지점은:

> ✅ 1.2 Chunk
✅ 1.3 Metadata Filter
✅ 1.4 BM25
✅ 2.0 Hybrid Search



3.0 이상은 비활성 옵션으로 두는 게 안정적이다.


---

2️⃣ 벡터 의존도 낮추기

벡터는:

인덱스

보조 랭킹

의미 보강


이지, 주 검색 엔진이 되면 안 된다.

이유:

재현성 낮음

모델 변경 시 결과 변동

비용 변동

캐시 무효화 문제


운영형은:

> 1차 BM25 → 2차 벡터 재정렬



이 구조가 가장 안정적이다.


---

3️⃣ 응답 재현성 확보

반드시 필요한 것:

검색 결과 Top-K 로그 저장

사용된 필터 로그 저장

사용된 커밋 해시 기록

프롬프트 버전 관리


그래야:

> “왜 이런 답이 나왔는가”를 추적 가능




---

4️⃣ LLM 안정 전략

외부 API 기반이라면:

모델 고정 (버전 락)

temperature = 낮게

시스템 프롬프트 고정

fallback 모델 준비


운영형 챗봇은
“창의성”보다 “일관성”이 중요하다.


---

5️⃣ 데이터 동기화 전략

Gogs 기반이면:

주기적 인덱스 재빌드

커밋 이벤트 기반 증분 업데이트

인덱스 무결성 체크


완전 재빌드는 비용 크다.
증분 업데이트가 핵심이다.

🔎 가장 중요한 것

운영 안정성은 기술이 아니라
통제 범위(Control Surface) 에서 나온다.

통제해야 할 것:

모델 버전

프롬프트 버전

인덱스 버전

메타데이터 스키마

검색 파라미터


이걸 문서화하지 않으면
아무리 고급 구조여도 불안정해진다.

지금 상태에서 “안정형 모델 준비”라면:
