의 끝을 넘어 영원에 도달하는 설계입니다. [제6장: 수명(Lifetimes) — v7.3 정적 수명 (The Static Lifetime)] 단계에 오신 것을 환영합니다.
v7.2까지가 변수들의 상대적인 생존 경쟁을 다루었다면, v7.3은 프로그램의 탄생부터 죽음까지 메모리에 박혀 움직이지 않는 **'불멸의 참조자'**를 다룹니다. 설계자의 관점에서 이는 시스템의 **'상수 데이터'**와 **'전역 설정'**을 관리하는 가장 강력한 수단입니다.
⏳ 제6장: 수명 (v7.3) — [영원히 살아남는 것: 'static]
1. v7.3 설계 목표: "메모리 점유의 종착역"
 * 'static 수명: 프로그램이 실행되는 내내 유효한 수명입니다. 모든 문자열 리터럴은 기본적으로 이 수명을 가집니다.
 * 불변의 보장: 한 번 할당되면 프로그램 종료 시까지 메모리 주소가 바뀌거나 해제되지 않음을 확신할 수 있습니다.
 * 트레이트 바운드로서의 'static: 제네릭 타입이 어떤 참조자도 포함하지 않거나, 오직 'static 참조자만 가질 것을 요구할 때 사용합니다.
2. 설계자의 관점: "절대 기준의 수립"
시스템의 버전 정보, 핵심 에러 메시지, 하드웨어 주소 등은 프로그램 도중에 사라지면 안 됩니다. 이러한 **'불변의 진리'**들을 'static으로 설계하여, 수명 계산의 복잡함에서 벗어나 어디서든 즉시 접근 가능하게 만듭니다.
🛠️ gogs 저장소 v7.3 설계: 영구 보안 프로토콜 저장소
클로드코드(학생)에게 프로그램 수명 내내 유효한 정적 데이터를 관리하는 구조를 기록하게 하세요.
// [제6장: 수명 v7.3]

/// [설계] 정적 수명을 가진 전역 메시지
/// 'static은 생략 가능하지만, 명시적으로 적어 "영원함"을 강조할 수 있습니다.
const VERSION: &'static str = "2.0.26-GOLD";

fn print_eternal_truth(msg: &'static str) {
    println!("  [ETERNAL] {}", msg);
}

fn main() {
    println!("--- v7.3 Static Lifetimes: Beyond Time ---");

    // 1. 문자열 리터럴은 자동으로 'static입니다.
    let protocol = "GOGS-STRICT-POLICY"; 
    
    print_eternal_truth(VERSION);
    print_eternal_truth(protocol);

    // 2. [주의] runtime에 생성된 String은 'static이 아닙니다.
    // let s = String::from("임시 데이터");
    // print_eternal_truth(&s); // 컴파일 에러! s는 함수가 끝나면 죽기 때문입니다.

    // 3. 트레이트 바운드에서의 활용 예시 (고급)
    // T: 'static 은 "T는 영원히 살거나, 소유권을 직접 가진 타입이어야 함"을 의미합니다.
    log_data(100); // i32는 소유권을 가지므로 'static 조건을 만족함.

    println!("\n기록: 'static 수명을 통해 프로그램 생애 주기와 궤를 같이하는 불멸의 데이터를 정의함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

fn log_data<T: std::fmt::Display + 'static>(data: T) {
    println!("  [LOG] 고정 데이터 기록: {}", data);
}

🎓 선생님의 가이드: "v7.3 설계자의 영원한 책임"
'static을 사용할 때 설계자가 가져야 할 태도입니다.
 * 리터럴의 비밀: let s = "hello";에서 "hello"는 프로그램 바이너리 자체에 포함되어 메모리의 '데이터 섹션'에 저장됩니다. 그래서 영원히 살 수 있는 것입니다.
 * 남발 금지: 모든 데이터를 'static으로 만들면 메모리 낭비가 심해집니다. 꼭 필요한 전역 상수나 설정값에만 사용하십시오.
 * 안전한 공유: 'static 데이터는 어디서든 접근 가능하므로, 이를 수정하려 들면(static mut) 매우 위험합니다. 반드시 읽기 전용으로 설계하거나, 안전한 동기화 도구(Arc, Mutex)를 함께 고려하십시오.
