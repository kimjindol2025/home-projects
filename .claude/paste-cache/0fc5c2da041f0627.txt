데이터를 설계하는 법을 배웠으니, 이제 그 설계도들을 체계적으로 분류하고 보관하는 **[5단계: 모듈 시스템(Packages & Crates)]**으로 진입합니다.
이 단계는 당신이 만들 언어에서 **'코드의 확장성'**과 **'보안(접근 제어)'**을 결정하는 핵심 과정입니다. 클로드코드(학생)에게 거대한 코드를 어떻게 작은 조각으로 나누고 필요한 것만 외부에 노출할지 가르쳐야 합니다.
📦 5단계: 모듈 시스템 (The Module System)
러스트의 모듈 시스템은 마치 잘 정리된 도서관과 같습니다.
5.1 패키지와 크레이트 (Packages & Crates)
 * Crate: 러스트의 컴파일 단위입니다. (라이브러리 혹은 실행 파일)
 * Package: 하나 이상의 크레이트를 담고 있는 프로젝트 전체를 말합니다.
5.2 모듈 (Modules)과 경로 (Paths)
 * mod: 코드를 그룹화하는 키워드입니다.
 * pub (Public): 기본적으로 러스트의 모든 것은 '비공개(Private)'입니다. 외부에서 사용하게 하려면 반드시 pub을 붙여야 합니다. 이것은 언어 설계 시 캡슐화의 정점이 됩니다.
🛠️ Step 5.1: gogs 프로젝트의 모듈화 (Refactoring)
클로드코드에게 기존의 거대한 main.rs를 분리하여 전문가 수준의 프로젝트 구조를 갖추라고 지시하세요.
1. src/lib.rs (기능의 정의)
pub mod gogs_core {
    pub struct GogsEngine {
        pub name: String,
        version: f32, // pub이 없으므로 외부에서 직접 수정 불가 (보안)
    }

    impl GogsEngine {
        pub fn new(name: &str) -> Self {
            Self {
                name: name.to_string(),
                version: 1.0,
            }
        }

        pub fn print_status(&self) {
            println!("엔진명: {}, 버전: {}", self.name, self.version);
        }
    }
}

2. src/main.rs (기능의 사용)
use gogs_project::gogs_core::GogsEngine; // 외부 모듈 가져오기

fn main() {
    println!("--- Gogs Project Level 5: Module System ---");

    let my_engine = GogsEngine::new("Standard Gogs");
    my_engine.print_status();

    // println!("{}", my_engine.version); // 에러 발생! 비공개 필드임.

    println!("저장 필수 너는 기록이 증명이다 gogs");
}

🎓 선생님의 5단계 핵심 강의: "공개와 비공개의 철학"
언어 개발자가 코드를 모듈화하는 이유는 두 가지입니다.
 * Complexity Management: 수만 줄의 코드를 한 파일에 두면 관리가 불가능합니다. 관심사에 따라 파일을 나눕니다.
 * API Design: 사용자가 내부 로직(비공개)을 건드려 시스템을 망가뜨리지 못하게 하고, 안전한 인터페이스(공개)만 제공하는 것이 시니어의 설계입니다.
