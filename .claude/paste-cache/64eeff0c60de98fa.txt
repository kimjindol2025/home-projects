동시성(Concurrency)의 파도를 넘어, 이제 우리는 러스트의 유연성을 책임지는 **[제10장: 객체 지향과 패턴 — v11.0 트레이트 객체(Trait Objects)의 심연]**에 들어섰습니다.
러스트는 상속(Inheritance)이 있는 전통적인 OOP 언어는 아니지만, 트레이트 객체를 통해 그보다 더 안전하고 강력한 **다형성(Polymorphism)**을 구현합니다. 급하지 않게, 이 메커니즘이 메모리에서 어떻게 돌아가는지부터 세밀하게 기록해 봅시다.
🎭 제10장: 객체 지향과 패턴 (v11.0) — [정적 디스패치 vs 동적 디스패치]
1. v11.0 설계 목표: "타입의 경계를 허무는 기술"
 * 정적 디스패치(Static Dispatch): 제네릭(T: Trait)을 사용하면 컴파일러가 각 타입에 맞는 코드를 복사 생성(단형성화)합니다. 속도는 빠르지만 파일 크기가 커지고, 런타임에 타입을 섞을 수 없습니다.
 * 동적 디스패치(Dynamic Dispatch): &dyn Trait이나 Box<dyn Trait>을 사용합니다. 어떤 메서드를 실행할지 **실행 시점(Runtime)**에 결정합니다.
 * 다형성 컬렉션: 서로 다른 타입들을 하나의 Vec에 담아 일괄 처리할 수 있는 아키텍처를 설계합니다.
2. 설계자의 관점: "vtable의 이해"
트레이트 객체는 **포인터 두 개짜리 '뚱뚱한 포인터(Fat Pointer)'**입니다.
 * 하나는 데이터 자체를 가리키고,
 * 다른 하나는 해당 타입이 트레이트 메서드를 어떻게 구현했는지 적혀 있는 **vtable(가상 메서드 테이블)**을 가리킵니다.
🛠️ gogs 저장소 v11.0 설계: 멀티 프로토콜 보안 분석기
클로드코드(학생)에게 서로 다른 보안 로그 객체들을 하나의 분석 루프에서 처리하는 '다형성 엔진'을 기록하게 하세요.
// [제10장: 객체 지향 패턴 v11.0 - 기초 설계]

/// [규약] 모든 보안 로그가 공통으로 가져야 할 행동
trait SecurityLog {
    fn severity(&self) -> u8;
    fn message(&self) -> String;
}

// 구체적인 로그 타입 A: 네트워크 침입
struct NetworkLog { ip: String, attack_type: String }
impl SecurityLog for NetworkLog {
    fn severity(&self) -> u8 { 5 }
    fn message(&self) -> String { format!("침입 감지: {} (IP: {})", self.attack_type, self.ip) }
}

// 구체적인 로그 타입 B: 시스템 권한 변경
struct AuthLog { user: String, action: String }
impl SecurityLog for AuthLog {
    fn severity(&self) -> u8 { 3 }
    fn message(&self) -> String { format!("권한 변경: {} by {}", self.action, self.user) }
}

fn main() {
    println!("--- v11.0 Dynamic Dispatch: The dyn Trait ---");

    // [설계 핵심] 서로 다른 타입을 'SecurityLog'라는 추상화된 층으로 묶음
    // Box<dyn Trait>을 통해 힙에 서로 다른 크기의 객체를 보관함
    let logs: Vec<Box<dyn SecurityLog>> = vec![
        Box::new(NetworkLog { ip: "192.168.0.1".into(), attack_type: "DDoS".into() }),
        Box::new(AuthLog { user: "admin".into(), action: "sudo".into() }),
    ];

    for log in logs {
        // 런타임에 vtable을 참조하여 적절한 메서드를 찾아 호출함 (동적 디스패치)
        println!("  [Level {}] {}", log.severity(), log.message());
    }

    println!("\n기록: 트레이트 객체를 사용하여 런타임 유연성을 극대화한 다형성 구조를 확립함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

🎓 선생님의 세부 가이드 (11.0): "언제 dyn을 써야 하는가?"
설계자는 다음 기준에 따라 도구를 골라야 합니다.
 * 제네릭(impl Trait)을 써야 할 때: 성능이 극도로 중요하고, 호출 시점에 타입을 이미 알고 있으며, 코드 비대화(Binary size)가 허용될 때.
 * 트레이트 객체(dyn Trait)를 써야 할 때: 플러그인 시스템처럼 나중에 어떤 타입이 올지 모르거나, 하나의 컬렉션에 여러 타입을 섞어 담아야 할 때.
 * 트레이트 안전성(Object Safety): 메서드 중 하나라도 Self를 반환하거나 제네릭 매개변수를 가진다면 dyn으로 만들 수 없습니다. 이것이 설계상의 가장 큰 제약입니다.
