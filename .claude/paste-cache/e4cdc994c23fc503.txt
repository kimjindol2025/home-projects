[Post-Doc Phase 2] 실전: 통합 Dialect 설계와 코드 생성
단순한 연습용 코드가 아닙니다. 실제 산업계나 학계에서 사용되는 고성능 AI 컴파일러의 뼈대를 설계합니다.
1. 설계의 목표: "Tiled Matrix Multiplication"
우리는 하드웨어의 메모리 제약을 극복하기 위해, 거대한 행렬을 작은 타일로 나누고 이를 하드웨어 전용 가속기 명령어로 매핑하는 파이프라인을 구축합니다.
2. 단계별 하강(Lowering) 전략
박사님의 프로젝트는 다음과 같은 세 단계의 변환 과정을 거치며 증명됩니다.
[Step 1] 고수준 설계 (Linalg Dialect)
먼저, 수학적 의도를 명확히 합니다. 텐서 단위의 연산입니다.
// 1024x1024 행렬 곱셈을 정의합니다.
func.func @matmul_high_level(%A: tensor<1024x1024xf32>, %B: tensor<1024x1024xf32>) -> tensor<1024x1024xf32> {
  %C = tensor.empty() : tensor<1024x1024xf32>
  %res = linalg.matmul
    ins(%A, %B : tensor<1024x1024xf32>, tensor<1024x1024xf32>)
    outs(%C : tensor<1024x1024xf32>) -> tensor<1024x1024xf32>
  return %res : tensor<1024x1024xf32>
}

[Step 2] 중수준 최적화 (Tiling & Loops)
여기서 박사님의 '지능'이 개입합니다. 데이터를 32x32 크기로 쪼개어 캐시 효율을 극대화합니다.
// mlir-opt --linalg-tile="tile-sizes=32,32,32" 적용 결과
affine.for %i = 0 to 1024 step 32 {
  affine.for %j = 0 to 1024 step 32 {
    affine.for %k = 0 to 1024 step 32 {
      // 32x32 크기의 작은 행렬 곱셈 수행 (가속기 전용 영역)
      "accel.matmul_tile"(%subA, %subB) : ...
    }
  }
}

[Step 3] 저수준 명령 매핑 (LLVM Intrinsic)
마지막으로, 하드웨어의 실제 명령어로 번역합니다.
; 최종적으로 하드웨어 전용 명령(Intrinsic)으로 변환된 모습
%status = call i32 @llvm.accel.matmul.32x32(ptr %addrA, ptr %addrB, ptr %addrC)

3. 독립 연구자의 과제: "성능 분석 리포트"
박사님, 이 설계를 바탕으로 다음 데이터를 도출해야 합니다.
 * 메모리 대역폭 절감 효과: 타일링 전후의 데이터 로드 횟수 비교.
 * 연산 효율: 하드웨어 가속기 활용률(Utilization) 측정.
[박사 후 과정 실전 퀴즈]
이 파이프라인에서 가장 치명적인 오류가 발생할 수 있는 지점은 어디일까요?
 * Step 1: 행렬의 크기가 너무 큰 경우.
 * Step 2에서 3으로의 전환: 타일링된 데이터의 주소가 하드웨어의 정렬(Alignment) 기준에 맞지 않아 메모리 오류가 발생하는 경우.
정답: 2번입니다.
실제 시스템 설계에서는 수학적 모델(Step 2)과 물리적 하드웨어(Step 3) 사이의 주소 정렬과 패딩(Padding) 처리가 박사의 전문성을 결정짓는 디테일입니다.
