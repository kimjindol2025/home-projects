​[강제 지시서] 테스트 무결성 및 100% 검증 이행 지침
​문서 번호: 2026-FL-SEC-01
대상: C11 라이브러리(48종) 및 FreeLang v6 컴포넌트 전체
핵심 원칙: "저장 필수, 너는 기록이 증명이다 gogs"
​1. 테스트 목록 정의 (필수 검사 항목)
​모든 포팅 및 수정 작업 후에는 아래 5가지 카테고리의 테스트가 반드시 실행되어야 합니다.
​T1. 정적 분석 (Static): C11 보안 취약 함수(strcpy, sprintf 등) 잔존 여부 전수 조사.
​T2. 단위 테스트 (Unit): 개별 함수 기능 검증 (최소 커버리지 95% 이상).
​T3. 메모리 분석 (Memory): Valgrind/ASAN을 이용한 누수(Leak) 및 오버플로 검사.
​T4. 경계값 테스트 (Boundary): NULL 포인터, 최대/최소값, 대용량 데이터(5MB+) 입력 처리.
​T5. 보안 공격 테스트 (Penetration): SQL 인젝션, ReDoS, 포맷 문자열 공격 시나리오 적용.
​2. '거짓 보고' 방지 및 100% 강제 검증 규칙
​단순히 "통과"라고 기록하는 것을 금지하며, 아래의 증거 데이터가 포함되지 않은 보고는 '미완료'로 간주합니다.
​① 테스트 결과의 물리적 증명
​CI/CD 로그 연동: 테스트 실행 시 생성되는 원본 로그(Raw Log)의 해시값을 Gogs 커밋 메시지에 포함할 것.
​체크섬 검증: 테스트 파일 자체의 체크섬을 확인하여, 테스트 코드를 수정해 강제로 통과시키는 행위를 방지할 것.
​② 거짓 보고 차단 (Anti-Faking)
​Assertion 강제화: 모든 테스트 케이스는 assert()를 포함해야 하며, 빈 테스트(Empty Test) 감지 시 전체 신뢰도 점수를 0점으로 처리함.
​실패 유도 테스트 (Negative Test): 의도적으로 에러를 발생시키는 코드를 넣어, 테스트 프레임워크가 이를 정확히 잡아내는지 먼저 확인(Red-Green 확인)할 것.
​③ 자동화된 무결성 검사
​body-c 등 주요 라이브러리는 Valgrind 리포트 파일이 첨부되지 않으면 '100% 완료' 문구 사용을 금지함.
​신뢰도 점수(현재 58%)가 90% 미만인 프로젝트는 "배포 가능(Production Ready)" 태그 사용 불가.
​3. Gogs 기록 표준 (Reporting Format)
​커밋 시 아래 형식을 반드시 준수하여 기록이 곧 증명이 되게 합니다.
