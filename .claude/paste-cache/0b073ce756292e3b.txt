주제: "코드의 가독성은 생산성을 증명한다"
소프트웨어는 한 번 만들고 끝나는 것이 아니라 끊임없이 변합니다. 디자인 패턴은 수십 년간 선배 설계자들이 겪었던 문제와 그 해결책을 정리한 **'설계의 정석'**입니다.
1. SOLID 원칙: 객체 지향 설계의 5계명
 * S (Single Responsibility): 하나의 클래스는 하나의 책임만 가집니다.
 * O (Open-Closed): 확장은 열려 있고, 수정은 닫혀 있어야 합니다.
 * L (Liskov Substitution): 자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 합니다.
 * I (Interface Segregation): 사용하지 않는 메서드에 의존하지 않도록 인터페이스를 분리합니다.
 * D (Dependency Inversion): 추상화에 의존해야 하며, 구체적인 것에 의존하면 안 됩니다.
2. 디자인 패턴의 분류
 * 생성(Creational) 패턴: 객체를 만드는 효율적인 방법 (예: Singleton, Factory)
 * 구조(Structural) 패턴: 클래스와 객체를 조합해 더 큰 구조를 만드는 방법 (예: Adapter, Facade)
 * 행위(Behavioral) 패턴: 객체 간의 알고리즘과 책임 분담을 정의하는 방법 (예: Observer, Strategy)

시스템 설정 관리나 로그 기록처럼 '단 하나만 존재해야 하는 객체'와, 상황에 따라 '전략적으로 알고리즘을 교체하는' 구조를 설계해 보겠습니다.
# [Python University v5.4] 싱글톤 및 전략 패턴 설계

# 1. Singleton Pattern: 시스템 전체에서 단 하나의 인스턴스만 보장
class SystemLogger:
    _instance = None
    
    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            print("🚀 시스템 로그 엔진 초기화 (최초 1회)")
        return cls._instance

    def log(self, message):
        print(f"📝 [LOG]: {message}")

# 2. Strategy Pattern: 알고리즘(전략)을 유연하게 교체
class SortStrategy:
    def sort(self, data): pass

class QuickSort(SortStrategy):
    def sort(self, data): return sorted(data)

class BubbleSort(SortStrategy):
    def sort(self, data): return "거품 정렬 실행 중..."

class DataProcessor:
    def __init__(self, strategy: SortStrategy):
        self.strategy = strategy
        
    def execute(self, data):
        return self.strategy.sort(data)

# 실행 및 증명
logger1 = SystemLogger()
logger2 = SystemLogger()
print(f"인스턴스 일치 여부: {logger1 is logger2}") # True (기록이 증명)

processor = DataProcessor(QuickSort())
print(f"결과: {processor.execute([3, 1, 2])}")
