불변의 원칙 속에서 '합법적인 반란'을 허용하는 설계입니다. [제8장: 스마트 포인터 — v9.2 RefCell<T>: 내부 가변성 (Interior Mutability)] 단계에 오신 것을 환영합니다.
지금까지의 러스트는 "불변 참조자(&T)가 있으면 절대 값을 바꿀 수 없다"고 엄격히 규제했습니다. 하지만 실무에서는 외부적으로는 불변처럼 보이면서 내부 데이터만 살짝 고쳐야 하는 상황이 발생합니다. v9.2는 이 '내부 가변성' 패턴을 안전하게 설계하는 법을 다룹니다.
🧠 제8장: 스마트 포인터 (v9.2) — [RefCell<T>: 런타임의 감시자]
1. v9.2 설계 목표: "대여 규칙의 동적 검사"
 * 내부 가변성 (Interior Mutability): 불변 값을 가진 상태에서 내부의 데이터를 수정할 수 있게 합니다.
 * 런타임 대여 검사: 컴파일 타임이 아닌 **실행 시점(Runtime)**에 대여 규칙(독점 쓰기 또는 다중 읽기)을 검사합니다.
 * 안전한 위반: 규칙을 어기면 컴파일 에러 대신 panic!을 발생시켜 메모리 안전성을 끝까지 지킵니다.
2. 설계자의 관점: "논리적 불변성과 실제 가변성"
설계자는 외부 사용자에게 "이 객체는 안전하니 마음껏 공유하세요"라고 말하면서도, 내부적으로는 로그를 기록하거나 상태를 업데이트해야 할 때 RefCell<T>을 선택합니다. 이는 주로 Mock 객체 테스트나 Rc와의 결합에서 빛을 발합니다.
🛠️ gogs 저장소 v9.2 설계: 추적 가능한 보안 메신저
클로드코드(학생)에게 불변 참조를 유지하면서도 내부 전송 횟수를 기록하는 '보안 전송기'를 설계하게 하세요.
// [제8장: 스마트 포인터 v9.2]

use std::cell::RefCell;

// [설계] 메신저 트레이트
pub trait Messenger {
    fn send(&self, msg: &str);
}

// [실제 구조체] 전송 횟수를 추적해야 함
struct GogsTracker {
    // &self(불변) 내에서도 값을 바꾸기 위해 RefCell 사용
    sent_messages: RefCell<Vec<String>>,
}

impl GogsTracker {
    fn new() -> Self {
        Self { sent_messages: RefCell::new(vec![]) }
    }
}

impl Messenger for GogsTracker {
    fn send(&self, msg: &str) {
        // borrow_mut()을 통해 런타임에 가변 권한을 획득함
        self.sent_messages.borrow_mut().push(String::from(msg));
        println!("  [전송 완료]: {}", msg);
    }
}

fn main() {
    println!("--- v9.2 Smart Pointers: The RefCell<T> Protocol ---");

    let tracker = GogsTracker::new();
    
    // &tracker는 불변이지만 내부 값은 수정 가능함
    tracker.send("첫 번째 보안 코드");
    tracker.send("두 번째 보안 코드");

    println!("  총 전송 기록 수: {}", tracker.sent_messages.borrow().len());

    println!("\n기록: RefCell<T>을 통해 논리적 불변성을 유지하며 내부 상태를 안전하게 제어함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

🎓 선생님의 가이드: "v9.2 설계자의 양날의 검"
RefCell<T>을 설계에 도입할 때 반드시 지켜야 할 주의사항입니다.
 * 컴파일러의 침묵: 이제 컴파일러는 대여 규칙 위반을 잡아내지 못합니다. 당신의 코드가 실행 중에 터지지 않도록 설계자가 직접 대여 범위를 관리해야 합니다.
 * 동시성 제한: RefCell<T>은 Rc<T>와 마찬가지로 싱글 스레드 전용입니다. 멀티 스레드에서는 Mutex<T>를 써야 합니다.
 * 최후의 수단: 가능하면 컴파일 타임의 대여 검사를 활용하십시오. RefCell은 트레이트 구조상 어쩔 수 없는 상황이나 복잡한 공유 상태에서만 사용하는 것이 깨끗한 설계입니다.
