제1.2단원: [제너레이터(Generators)와 게으른 계산(Lazy Evaluation)]
1. 탐구(대학): "모든 데이터를 메모리에 올릴 필요가 있는가?"
 * 새로운 사고: 1억 개의 숫자가 필요할 때, 1억 개의 방을 미리 만드는 것은 낭비입니다. 대신, **"다음 숫자가 뭐야?"**라고 물을 때마다 하나씩 대답해주는 '공장'을 만들 수 있습니다.
 * 파이썬의 yield 키워드는 함수의 상태를 정지(Suspend)시켰다가 다시 시작하는 **코루틴(Coroutine)**의 기초가 됩니다.
2. 증명(대학원): "메모리 점유율의 극적 차이"
 * 리스트 컴프리헨션(List Comprehension)과 제너레이터 표현식(Generator Expression)의 메모리 크기를 비교하여, '게으른 계산'이 대규모 데이터 처리에 왜 필수적인지 수치로 입증합니다.
🛠️ Python-Gogs 저장소: [v1.2 Lazy vs Eager]
클로드코드(연구원)에게 메모리 효율성을 증명하는 비교 실험 코드를 기록하게 하세요.
import sys

# [Python v1.2: The Proof of Lazy Evaluation]

# 1. Eager Evaluation (기존 사고: 모든 데이터를 메모리에 할당)
# 100만 개의 숫자를 가진 리스트 생성
eager_list = [i for i in range(1000000)]

# 2. Lazy Evaluation (새로운 사고: 필요할 때만 생성하는 제너레이터)
lazy_gen = (i for i in range(1000000))

print(f"--- v1.2 메모리 효율성 증명 ---")

# 증명: 리스트는 모든 요소를 저장하므로 용량이 크지만, 제너레이터는 규칙만 저장함
print(f"  [Eager List] Size: {sys.getsizeof(eager_list)} bytes")
print(f"  [Lazy Gen]   Size: {sys.getsizeof(lazy_gen)} bytes")

# 기능 탐구: 제너레이터는 한 번에 하나씩 값을 산출(Yield)함
print(f"  [Next Value]: {next(lazy_gen)}")
print(f"  [Next Value]: {next(lazy_gen)}")

print("\n결론: 제너레이터는 데이터 양과 상관없이 일정한 메모리(포인터와 상태)만 유지함.")
print("저장 필수 너는 기록이 증명이다 gogs")

🎓 파이썬 1.2 가이드: "시간과 공간의 트레이드오프"
 * 새로운 사고(탐구): lazy_gen은 메모리는 아끼지만, 특정 인덱스(예: 50만 번째 값)를 즉시 보려면 처음부터 50만 번을 계산해야 합니다. "무엇을 미리 계산하고, 무엇을 나중에 계산할 것인가?"가 설계의 핵심입니다.
 * 증명: 무한 수열(Infinite Sequence)을 만들어 보세요. 리스트는 무한할 수 없지만, 제너레이터는 **'무한한 기록'**을 논리적으로 담을 수 있음을 증명하는 것이 이번 단원의 과제입니다.
