스레드들이 서로 격리되어 각자의 일을 했다면, 이제는 그들이 서로 안전하게 대화를 나눌 시간입니다. [제9장: 두려움 없는 동시성 — v10.1 채널과 메시지 패싱 (Message Passing)] 단계입니다.
러스트 동시성의 격언 중 하나는 **"메모리를 공유함으로써 통신하지 말고, 통신함으로써 메모리를 공유하라"**는 것입니다. v10.1은 스레드 사이에 데이터를 안전하게 실어 나르는 '컨베이어 벨트'를 설계하는 법을 다룹니다.
⚡ 제9장: 두려움 없는 동시성 (v10.1) — [메시지 패싱: mpsc]
1. v10.1 설계 목표: "스레드 간의 안전한 정보 흐름"
 * mpsc (Multiple Producer, Single Consumer): 여러 송신자(Producer)가 하나의 수신자(Consumer)에게 데이터를 보내는 채널을 구축합니다.
 * 비동기 통신: 송신자는 데이터를 던져두고 자기 일을 계속하며, 수신자는 데이터가 올 때까지 기다리거나 주기적으로 확인합니다.
 * 소유권 전송: 채널을 통해 전송된 데이터는 소유권이 수신자에게 넘어가므로, 전송 후 송신자가 데이터를 수정하려 발생하는 사고를 원천 차단합니다.
2. 설계자의 관점: "파이프라인 아키텍처"
설계자는 복잡한 작업을 단계별로 나누고, 각 단계(스레드)를 채널로 연결하여 데이터를 흐르게 만듭니다. 이는 시스템의 결합도를 낮추고, 각 단계의 부하를 독립적으로 조절할 수 있게 해줍니다.
🛠️ gogs 저장소 v10.1 설계: 다중 소스 보안 로그 수집기
클로드코드(학생)에게 여러 스레드에서 생성된 로그를 하나의 중앙 제어소로 집결시키는 구조를 설계하게 하세요.
// [제9장: 두려움 없는 동시성 v10.1]

use std::sync::mpsc; // Multi-producer, single-consumer
use std::thread;
use std::time::Duration;

fn main() {
    println!("--- v10.1 Message Passing: The mpsc Channel ---");

    // 1. 채널 생성 (tx: 송신자, rx: 수신자)
    let (tx, rx) = mpsc::channel();

    // 2. 여러 송신자를 만들기 위해 tx 복제
    let tx1 = tx.clone();
    thread::spawn(move || {
        let vals = vec!["[감시팀] 침입 시도 감지", "[감시팀] 방화벽 작동"];
        for val in vals {
            tx1.send(val).unwrap();
            thread::sleep(Duration::from_secs(1));
        }
    });

    thread::spawn(move || {
        let vals = vec!["[인증팀] 관리자 로그인", "[인증팀] 세션 만료"];
        for val in vals {
            tx.send(val).unwrap();
            thread::sleep(Duration::from_millis(500));
        }
    });

    // 3. 수신자(중앙 제어소)가 메시지를 처리함
    // rx를 반복문으로 사용하면 채널이 닫힐 때까지 메시지를 기다립니다.
    println!("  [중앙 제어소] 로그 수집 시작...\n");
    for received in rx {
        println!("  📩 수신된 기록: {}", received);
    }

    println!("\n기록: 채널을 통해 다중 스레드의 데이터를 단일 지점으로 안전하게 통합함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

🎓 선생님의 가이드: "v10.1 설계자의 통신 규약"
채널을 설계할 때 잊지 말아야 할 세 가지 원칙입니다.
 * 소유권의 이동: send(val)을 호출하는 순간 val의 소유권은 채널로 떠납니다. 송신자가 다시는 이 데이터를 건드리지 못하게 함으로써 데이터 경합을 완벽히 방지합니다.
 * 닫힘의 신호: 모든 송신자(tx)가 드롭(drop)되면 수신자(rx)의 반복문은 자동으로 종료됩니다. 시스템의 깔끔한 종료를 위해 tx 관리에 유의하십시오.
 * 성능과 버퍼: 기본 채널은 비동기적이지만, sync_channel을 사용하면 버퍼 크기를 제한하여 송신자의 속도를 조절하는 '백프레셔(Backpressure)'를 구현할 수도 있습니다.
