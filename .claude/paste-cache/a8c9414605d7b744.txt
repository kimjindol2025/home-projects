원 과정의 세 번째 관문이자 실전 시스템 구축의 핵심인 **4.3단계: 프레임워크의 완성, C++ 통합과 빌드 시스템(CMake)**을 시작합니다.
"저장 필수 너는 기록이 증명이다 gogs" — 이제 TableGen으로 설계한 명세서(.td)와 최적화 규칙들을 실제 동작하는 바이너리 파일로 뽑아낼 차례입니다. 대학원 연구실에서 본인만의 컴파일러 프로젝트를 관리하기 위한 **'뼈대'**를 세우는 작업입니다.
[MLIR 대학원 코스 4.3] 설계도에 생명 불어넣기
TableGen이 코드를 자동으로 생성해 주더라도, 결국 그 코드들이 기존 MLIR 인프라와 어떻게 연결되는지 결정하는 것은 **C++**과 CMake입니다.
1. C++ Integration (연결 고리)
자동 생성된 헤더 파일(.h.inc)과 소스 파일(.cpp.inc)을 내 프로젝트에 포함(Include)시켜야 합니다.
 * Header 가드: 자동 생성된 코드 중 원하는 부분(예: Op 정의만, 혹은 가이드라인만)을 골라서 불러옵니다.
 * Registry 등록: 내가 만든 Dialect를 MLIR 시스템이 인식할 수 있도록 DialectRegistry에 등록하는 코드를 작성합니다.
2. CMake: 빌드 시스템 설계
대학원 수준의 대규모 프로젝트는 수십 개의 파일로 이루어집니다. 이를 효율적으로 컴파일하기 위해 CMake를 사용합니다.
 * add_mlir_dialect_library: 내가 만든 Dialect를 라이브러리로 만듭니다.
 * mlir_tablegen: 빌드 시점에 .td 파일을 읽어 C++ 코드를 생성하도록 명령합니다.
[실전] 프로젝트 구조 (대학원 표준 템플릿)
보통 성공적인 MLIR 기반 논문 프로젝트는 다음과 같은 폴더 구조를 가집니다.
 * include/: .td 파일과 커스텀 Dialect의 헤더 파일들.
 * lib/: 실제로 로직을 구현한 C++ 소스 파일들.
 * tools/: 나만의 mlir-opt 같은 실행 파일을 만드는 곳.
 * test/: 내가 만든 규칙이 잘 작동하는지 검증하는 테스트 코드(.mlir).
[4.3 실습] 빌드 에러 해결사의 눈
만약 여러분이 새로운 연산을 추가했는데, 컴파일 시점에 "해당 연산의 정의를 찾을 수 없다"는 에러가 발생한다면 무엇을 확인해야 할까요?
 * .td 파일에 오타가 있는지 확인한다.
 * CMake에서 TableGen이 코드를 생성하도록 설정했는지, 그리고 생성된 .inc 파일을 C++에서 정확히 #include 했는지 확인한다.
정답: 2번입니다.
대학원 과정에서 가장 많이 겪는 '설정 오류'입니다. 명세서(TableGen)만 잘 쓴다고 끝나는 것이 아니라, 빌드 파이프라인이 제대로 연결되어 있어야 합니다.
4.3단계의 기록 (증명)
> "TableGen으로 생성된 코드는 C++ 프로젝트에 #include 되어 실제 기능을 수행한다."
> "CMake는 이 모든 변환 과정과 컴파일 과정을 조율하는 설계도 역할을 한다."
> 
이제 당신은 나만의 언어를 정의하고, 최적화하고, 실제 프로그램으로 빌드하는 법까지 마쳤습니다.
