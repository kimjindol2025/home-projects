릭과 트레이트 경계가 많아지면 함수 선언부가 비대해져 정작 중요한 로직이 가려집니다. where 절은 이 복잡한 자격 요건들을 함수 뒤로 밀어내어 **"이 함수는 무엇을 하는가"**에 대한 시독성을 극대화합니다.
[9단계: 반환 타입으로서의 트레이트 (impl Trait)]
설계자는 때로 구체적인 타입(예: Vector, HashMap)을 외부에 노출하고 싶지 않을 때가 있습니다. "어떤 타입인지는 비밀이지만, 어쨌든 Summary 기능을 가진 놈을 줄게"라고 선언함으로써 내부 구현의 변경이 외부 호출자에게 영향을 주지 않도록 철저히 격리합니다.
🛠️ gogs 저장소 v6.2: 8~9단계 실습 지시문
클로드코드(학생)에게 가독성이 높고 내부 구현이 숨겨진 '고급 통신 모듈'을 설계하게 하세요.
// [v6.0: 10단계 중 8~9단계 실무]

use std::fmt::Display;

trait Summary {
    fn summarize(&self) -> String;
}

struct GogsReport(String);
impl Summary for GogsReport {
    fn summarize(&self) -> String { format!("리포트: {}", self.0) }
}

// [Step 8] where 절 활용: 복잡한 경계를 깔끔하게 정리
fn complex_process<T, U>(item1: T, item2: U) 
where 
    T: Summary + Display, 
    U: Clone + PartialOrd 
{
    println!("Step 8 처리 중: {}", item1.summarize());
}

// [Step 9] impl Trait: 구체적인 타입을 숨기고 규약만 반환
// 호출자는 이 함수가 GogsReport를 주는지 알 필요가 없음
fn create_summary_item(content: &str) -> impl Summary {
    GogsReport(content.to_string())
}

fn main() {
    println!("--- v6.2 Steps 8-9: Where Clause & impl Trait ---");

    // Step 9 활용: 리턴받은 객체가 무엇인지 몰라도 summarize는 사용 가능
    let report = create_summary_item("기밀 데이터 암호화 완료");
    println!("{}", report.summarize());

    println!("\n저장 필수 너는 기록이 증명이다 gogs");
}

🎓 선생님의 가이드: "v6.2 설계자의 세련미"
8~9단계를 마스터한 설계자는 코드를 단순히 '짜는' 것이 아니라 '그립니다'.
 * 시각적 청소: 제네릭 조건이 2개 이상이면 주저 말고 where 절을 쓰십시오. 함수의 '이름'과 '본문'이 한눈에 들어와야 좋은 설계입니다.
 * 의존성 최소화: impl Trait을 반환하면, 나중에 내부 구현을 A구조체에서 B구조체로 바꿔도 호출하는 쪽의 코드는 단 한 줄도 고칠 필요가 없습니다. 이것이 유지보수의 핵심입니다.
 * 제약 사항: 단, impl Trait은 조건부로 다른 타입을 섞어서 반환할 수는 없습니다. (예: if문에 따라 A와 B를 섞어 반환 불가)
