🎓 Zig 전공 101: 1-13. 대규모 시스템을 위한 프로젝트 아키텍처 설계
Zig는 강제되는 폴더 구조가 없지만, 커뮤니티에서 권장되는 **'지속 가능한 구조'**가 존재합니다. 이는 유지보수와 협업의 효율성을 결정짓는 핵심 설계입니다.
1. 표준 프로젝트 레이아웃 (Standard Layout)
대규모 프로젝트는 역할을 명확히 분리해야 합니다.
 * /src: 실제 소스 코드가 위치합니다.
   * main.zig: 실행 파일(Executable)의 진입점. 보통 초기화 로직만 둡니다.
   * root.zig: 라이브러리(Library)의 진입점. 외부로 노출할 기능을 정의합니다.
 * /src/modules: 비즈니스 로직별로 폴더를 나눕니다 (예: auth/, database/, api/).
 * /build.zig: Zig 프로젝트의 심장입니다. 컴파일 옵션, 의존성 관리, 테스트 실행 등을 정의합니다.
 * /zig-out: 빌드 결과물이 저장되는 폴더 (Git에 포함하지 않음).
2. 의존성 관리 (Package Manager)
Zig 0.11부터 도입된 패키지 매니저는 외부 라이브러리를 안전하게 가져옵니다.
 * build.zig.zon: 프로젝트 이름, 버전, 그리고 외부 의존성(URL, Hash)을 기록하는 파일입니다. (ZON: Zig Object Notation)
 * 보안: 해시값을 통해 외부 코드가 변조되지 않았음을 증명합니다. **"기록이 증명하는 무결성"**의 실체입니다.
3. 인터페이스 패턴 (Interface Pattern)
Zig에는 interface 키워드가 없지만, 구조체와 함수 포인터를 사용하여 다형성을 구현합니다. 이는 런타임 오버헤드를 최소화하면서도 유연한 설계를 가능하게 합니다.
const Writer = struct {
    ptr: *anyopaque,
    writeFn: *const fn (ptr: *anyopaque, data: []const u8) anyerror!void,

    pub fn write(self: Writer, data: []const u8) !void {
        return self.writeFn(self.ptr, data);
    }
};

🔬 전공 심화: 빌드 시스템 활용 (build.zig)
Zig의 빌드 스크립트는 단순한 설정 파일이 아니라 하나의 Zig 프로그램입니다.
 * 컴파일 시점에 특정 플래그에 따라 코드를 다르게 빌드하거나(Conditional Compilation),
 * C 라이브러리를 링크하고,
 * 테스트 스텝을 추가하여 zig build test 한 줄로 전체 검증을 수행합니다.
📦 아키텍처 설계 원칙: 유닛 테스트 통합
대규모 시스템에서는 **"기록된 테스트가 곧 신뢰"**입니다. Zig는 소스 코드 바로 옆에 test 블록을 작성하는 것을 권장합니다.
test "수학 모듈 검증" {
    try std.testing.expect(add(1, 2) == 3);
}

분리된 파일이 아닌 실제 로직 옆에 테스트를 둠으로써, 코드의 의도를 명확히 기록하고 증명합니다.
