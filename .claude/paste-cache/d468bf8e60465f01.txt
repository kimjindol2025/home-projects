기초를 차곡차곡 쌓아 올리는 모습이 아주 좋습니다. 대학원 수준의 거대한 시스템을 설계하려면, 부품(연산)과 기계(함수)를 담는 **'공장 전체의 도면'**이 필요합니다.
"저장 필수 너는 기록이 증명이다 gogs" — 이제 우리는 **중등 2.2단계: 가장 큰 울타리, 모듈(Module)**을 배웁니다.
[MLIR 중등 2.2] 가장 큰 주머니: "모듈(Module)"
지금까지 우리는 %이름(변수), arith.addf(도구), func.func(기계)를 배웠습니다. 이 모든 것을 하나로 묶어서 하나의 파일로 만드는 가장 큰 단위가 바로 **모듈(Module)**입니다.
1. 모듈의 역할
모듈은 마치 **'공장 건물의 외벽'**과 같습니다.
 * 모든 MLIR 파일은 기본적으로 하나의 module 안에 담깁니다.
 * 이 안에는 여러 개의 함수(func)가 들어갈 수 있고, 함수들끼리 서로 데이터를 주고받을 수 있습니다.
2. MLIR의 전체 구조 (완성판)
우리가 지금까지 배운 모든 기초를 합치면 이런 모양이 됩니다. 이것이 바로 대학원 연구의 기본 템플릿입니다.
// 1. 공장(Module)을 세웁니다.
module {
  
  // 2. 기계(Function)를 설치합니다. @기계이름
  func.func @smart_calculator(%arg0: f32) -> f32 {
    
    // 3. 도구(Operation)를 써서 계산합니다. 브랜드.도구
    %cst = arith.constant 1.0 : f32          // 숫자 1.0을 준비함
    %result = arith.addf %arg0, %cst : f32   // 입력값에 1.0을 더함
    
    // 4. 결과물을 내보냅니다.
    return %result : f32
  }
  
}

3. 대학원 졸업을 위한 '밑바닥' 지식: 기호(Symbol)
 * % (Value): 공장 안에서 돌아다니는 재료입니다. (함수 밖으로 못 나감)
 * @ (Symbol): 공장에 고정된 기계나 장치의 이름입니다. (공장 전체에서 어디서든 부를 수 있음)
[2.2 실습] 공장 도면 검토하기
대학원생인 당신은 아래의 설계도를 보고 오류를 찾아내야 합니다.
module {
  func.func @step1(%arg0: i32) -> i32 {
    %0 = arith.addi %arg0, %arg0 : i32
    return %0 : i32
  }

  func.func @step2(%arg1: i32) -> i32 {
    // 위에서 만든 @step1 기계를 쓰고 싶다면?
    %1 = func.call @step1(%arg1) : (i32) -> i32
    return %1 : i32
  }
}

[질문] 위 코드에서 @step1과 %0의 차이는 무엇일까요?
 * @step1: ( ) - 공장 어디서든 부를 수 있는 기계 이름
 * %0: ( ) - 특정 함수 안에서만 잠깐 쓰이는 결과물 이름
정답: 1. Symbol(기호), 2. Value(값)
이 차이를 아는 것이 매우 중요합니다. 기계(@)는 고정되어 있고, 재료(%)는 계속 흘러갑니다.
2.2단계의 기록 (증명)
> "MLIR 파일의 시작과 끝은 module { ... }로 감싼다."
> "고정된 이름(함수 등)은 @를 쓰고, 변하는 값은 %를 쓴다."
> 
이제 여러분은 MLIR의 **외형(Syntax)**을 완벽히 마스터했습니다!
