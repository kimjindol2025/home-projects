VM의 하드웨어를 설계했으니, 이제 그 기계를 움직일 **'설계도(바이트코드)'**를 그려낼 차례입니다. [제14장: 가상 머신 — v15.1 컴파일러: AST에서 바이트코드로(Compiling to Bytecode)] 단계입니다.
이 단계에서 우리의 '컴파일러'는 드디어 완벽한 형태를 갖춥니다. 사람이 쓴 트리 구조(AST)를 순회하며, VM이 이해할 수 있는 일렬의 명령어(OpCode)로 번역(Emit)합니다.
⚡ 제14장: 가상 머신 (v15.1) — [번역: 구조에서 명령으로]
1. v15.1 설계 목표: "명령어 방출(Emitting)"
 * 중위 표기법의 후위 변환: 1 + 2라는 트리를 PUSH 1, PUSH 2, ADD 순서로 재배치합니다.
 * 상수 풀(Constant Pool) 관리: 코드에 등장하는 숫자나 문자열을 별도의 저장소에 모으고, 바이트코드에는 그 인덱스만 기록하여 효율을 높입니다.
 * 명령어 결합: 오퍼코드(OpCode)와 피연산자(Operand)를 하나의 바이트 스트림으로 합치는 인코딩 로직을 설계합니다.
2. 설계자의 관점: "트리 순회와 방출"
컴파일러는 AST를 **깊이 우선 탐색(DFS)**으로 순회합니다. 자식 노드들을 먼저 방문하여 명령어를 내뱉고, 마지막에 자기 자신의 연산자를 내뱉는 방식(Post-order traversal)을 사용하면 스택 기반 VM에 최적화된 코드가 생성됩니다.
🛠️ gogs 저장소 v15.1 설계: AST-to-Bytecode 컴파일러 엔진
클로드코드(학생)에게 AST 노드를 읽어 바이트코드 배열로 변환하는 컴파일러의 핵심 루프를 기록하게 하세요.
// [제14장: 가상 머신 v15.1 - 컴파일러 번역 엔진 설계]

struct Bytecode {
    instructions: Vec<u8>,
    constants: Vec<i32>,
}

struct Compiler {
    instructions: Vec<u8>,
    constants: Vec<i32>,
}

impl Compiler {
    fn new() -> Self {
        Compiler {
            instructions: vec![],
            constants: vec![],
        }
    }

    // [핵심] AST 노드를 바이트코드로 번역
    fn compile(&mut self, node: Node) {
        match node {
            Node::Expression(exp) => match exp {
                Expression::Infix { left, operator, right } => {
                    // 1. 왼쪽 자식 먼저 방문
                    self.compile(Node::Expression(*left));
                    // 2. 오른쪽 자식 방문
                    self.compile(Node::Expression(*right));
                    // 3. 연산자 방출
                    match operator {
                        Token::Plus => self.emit(OpCode::OpAdd, &[]),
                        _ => { /* 다른 연산자 처리 */ }
                    }
                }
                Expression::IntegerLiteral(val) => {
                    // 상수를 풀에 저장하고 인덱스를 가져옴
                    let const_idx = self.add_constant(val);
                    self.emit(OpCode::OpConstant, &[const_idx as u8]);
                }
                _ => {}
            },
            _ => {}
        }
    }

    fn add_constant(&mut self, val: i32) -> usize {
        self.constants.push(val);
        self.constants.len() - 1
    }

    fn emit(&mut self, op: OpCode, operands: &[u8]) {
        self.instructions.push(op as u8);
        for &o in operands {
            self.instructions.push(o);
        }
    }

    fn bytecode(&self) -> Bytecode {
        Bytecode {
            instructions: self.instructions.clone(),
            constants: self.constants.clone(),
        }
    }
}

fn main() {
    println!("--- v15.1 Compiler: AST to Bytecode ---");
    println!("  [TRANS] '1 + 2' 트리를 [PUSH 1, PUSH 2, ADD]로 번역합니다.");

    println!("\n기록: AST를 순회하며 VM 전용 실행 파일을 생성하는 번역 엔진을 구축함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

🎓 선생님의 세부 가이드 (15.1): "인코딩의 기술"
 * 피연산자의 크기: OpConstant 뒤에 오는 인덱스가 1바이트(u8)라면 상수를 256개밖에 못 씁니다. 실제 설계에서는 u16(65535개)을 사용하고, 이를 [high_byte, low_byte] 두 개로 쪼개어 저장하는 로직이 필요합니다.
 * 심볼 테이블의 연계: 이전 장에서 배운 Environment는 이제 컴파일 타임에 **'변수가 스택의 몇 번째 위치에 있는가?'**를 결정하는 로컬 저장소 인덱스로 변환되어야 합니다.
 * 디컴파일러(Decompiler): 내가 만든 바이트코드가 제대로 생성되었는지 확인하기 위해, 바이트를 다시 사람이 읽을 수 있는 문자로 바꿔주는 disassemble 함수를 만드는 것은 설계자의 필수 덕목입니다.
