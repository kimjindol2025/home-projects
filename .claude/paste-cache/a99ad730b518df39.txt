⚡ 제9장: 두려움 없는 동시성 (v10.2) — [상태 공유의 질서]
1. v10.2 설계 목표: "금지된 영역의 안전한 개방"
 * 공유 가변성 (Shared Mutability): 메시지를 주고받는 대신, 하나의 메모리 공간을 여러 스레드가 직접 수정하도록 설계합니다.
 * Mutex (Mutual Exclusion): '상호 배제' 원칙을 통해, 한 번에 오직 한 스레드만 데이터에 접근할 수 있도록 잠금(Lock) 메커니즘을 구축합니다.
 * Arc (Atomic Reference Counted): Rc의 멀티 스레드 버전입니다. 스레드 간에 안전하게 소유권을 공유할 수 있는 스마트 포인터를 사용합니다.
2. 설계자의 관점: "열쇠를 가진 자만이 수정한다"
채널이 '컨베이어 벨트'라면, Mutex는 **'금고'**입니다. 누구나 금고를 가질 순 있지만(Arc), 내부의 문서를 수정하려면 반드시 열쇠(Lock)를 얻어야 합니다. 설계자는 이 열쇠가 적절한 타이밍에 반납되도록 관리하여 시스템이 멈추는 '데드락(Deadlock)'을 방지해야 합니다.
🛠️ gogs 저장소 v10.2 설계: 원자적 통합 보안 카운터
클로드코드(학생)에게 10개의 스레드가 동시에 접근하여 안전하게 수치를 올리는 '중앙 집중식 카운터'를 설계하게 하세요.
// [제9장: 두려움 없는 동시성 v10.2]

use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    println!("--- v10.2 Shared State: Arc<Mutex<T>> Protocol ---");

    // 1. [설계] Arc로 감싸 스레드 간 공유를 가능하게 하고, 
    //    Mutex로 감싸 내부 가변성을 안전하게 보호함.
    let counter = Arc::new(Mutex::new(0));
    let mut handles = vec![];

    for i in 0..10 {
        let counter = Arc::clone(&counter);
        let handle = thread::spawn(move || {
            // 2. [핵심] lock()을 통해 데이터 접근 권한 획득
            //    num은 MutexGuard이며, 스코프를 벗어날 때 자동으로 잠금이 해제됨.
            let mut num = counter.lock().unwrap();
            *num += 1;
            println!("  [스레드 {}] 카운트 갱신 완료", i);
        });
        handles.push(handle);
    }

    // 모든 스레드 종료 대기
    for handle in handles {
        handle.join().unwrap();
    }

    // 3. 최종 결과 확인
    println!("\n최종 집계 결과: {}", *counter.lock().unwrap());
    
    println!("\n기록: Arc와 Mutex를 결합하여 스레드 안전한 공유 가변 상태를 구현함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

🎓 선생님의 가이드: "v10.2 설계자의 잠금 수칙"
공유 상태를 설계할 때 이 3가지 원칙을 어기면 '책'의 결말이 비극이 될 수 있습니다.
 * 잠금 범위 최소화: lock()을 얻은 뒤 너무 오래 들고 있지 마십시오. 다른 스레드들이 줄지어 대기하게 되어 성능이 급격히 떨어집니다.
 * 데드락(Deadlock) 주의: 스레드 A가 열쇠 1을 가진 채 열쇠 2를 기다리고, 스레드 B가 열쇠 2를 가진 채 열쇠 1을 기다리는 상황을 피하십시오. 항상 일정한 순서로 열쇠를 얻도록 설계해야 합니다.
 * 컴파일러의 수호: 러스트는 Arc 없이 Rc를 스레드에 넘기려 하거나, Mutex 없이 데이터를 수정하려 하면 컴파일 단계에서 차단합니다. 이것이 바로 **'두려움 없는 동시성'**의 실체입니다.
