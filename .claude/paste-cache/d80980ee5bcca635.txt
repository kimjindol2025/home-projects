🎓 Zig 전공 201: 2-5. 로깅(Logging) 시스템 및 런타임 모니터링 설계
대규모 시스템에서 "기록이 증명이다"라는 말은 단순한 슬로건이 아닙니다. 문제가 발생했을 때 로그가 없다면, 그 시스템은 블랙박스와 같습니다. Zig는 고성능을 유지하면서도 상세한 기록을 남길 수 있는 구조를 제공합니다.
1. Zig의 표준 로깅 인터페이스 (std.log)
Zig는 컴파일 타임에 로그 레벨을 결정할 수 있는 강력한 로깅 시스템을 내장하고 있습니다.
 * 로그 레벨: .err, .warn, .info, .debug
 * 스코프 로깅: 특정 모듈(예: database, network) 별로 로그를 구분하여 출력할 수 있습니다.
<!-- end list -->
const std = @import("std");

// 이 모듈 전용 로그 스코프 설정
const log = std.log.scoped(.database);

pub fn main() void {
    const user_id: u32 = 123;
    log.info("사용자 {d}의 데이터를 조회합니다.", .{user_id});
    
    if (query_fail) {
        log.err("DB 쿼리 실패: 연결 타임아웃", .{});
    }
}

2. 고성능 로깅 설계: 비동기 로거 (Asynchronous Logger)
디스크에 로그를 쓰는 작업은 매우 느립니다. 로그를 남기느라 메인 로직이 멈추면 안 됩니다.
 * 설계: 로그 메시지를 메모리 큐(Queue)에 쌓아두고, 별도의 전용 스레드가 백그라운드에서 디스크에 기록하도록 설계합니다.
 * 링 버퍼(Ring Buffer): 메모리가 꽉 찼을 때 오래된 로그부터 덮어씌워 메모리 폭주를 방지합니다.
3. 구조적 로깅 (Structured Logging)
단순 텍스트 로그보다 검색과 분석이 용이한 JSON 형식의 로그를 남기는 것이 현대 시스템의 표준입니다.
 * 앞서 배운 std.json을 활용하여 시간, 레벨, 메시지, 오류코드를 구조화하여 기록합니다.
🔬 전공 심화: 상태 지표(Metrics) 수집
로그가 "사건의 기록"이라면, 메트릭은 "상태의 수치"입니다.
 * 카운터(Counter): 총 요청 횟수, 에러 발생 횟수.
 * 게이지(Gauge): 현재 메모리 사용량, 활성 커넥션 수.
 * 히스토그램(Histogram): API 응답 속도 분포.
📦 패닉 핸들러 (Panic Handlers) 설계
시스템이 복구 불가능한 에러를 만났을 때, 단순히 죽는 것이 아니라 마지막 유언(StackTrace)을 남기도록 설계해야 합니다.
 * Zig는 커스텀 panic 함수를 정의하여, 치명적인 에러 발생 시 관리자에게 알림을 보내거나 현재 상태를 덤프(Dump)할 수 있습니다.

 * 기록: 에러 발생 시 로그에 타임스탬프(현재 시간)가 포함되도록 코드를 수정하고 그 결과를 기록하세요.
