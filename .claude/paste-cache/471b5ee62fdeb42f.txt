주제: "실패를 데이터로 전환하는 기술"
실력 있는 설계자는 코드가 "정상적으로 돌아갈 때"보다 "문제가 생겼을 때" 어떻게 반응하는지를 더 중요하게 생각합니다. 시스템이 갑자기 꺼지는 것(Crash)은 기록의 손실을 의미하기 때문입니다.
1. 예외 처리의 4단계 구성
 * try: 위험할 수 있는 실험 코드를 실행합니다.
 * except: 특정 사고(에러)가 발생했을 때 구조대 역할을 합니다.
 * else: 사고 없이 완벽하게 성공했을 때만 실행되는 '축하' 구간입니다.
 * finally: 성공하든 실패하든, 연구실 청소(파일 닫기, 메모리 해제)를 위해 반드시 실행됩니다.
2. 사용자 정의 예외 (Raise)
 * 파이썬이 정한 에러 외에도, 설계자가 직접 "이건 우리 시스템에서 에러야!"라고 규정할 수 있습니다. (예: GogsError)
🛠️ 실습 기록: [v2.1 Advanced Safety Shield]
실제 연구 데이터를 입력받을 때 발생할 수 있는 여러 사고를 방어하는 시스템을 클로드코드(고등학생)에게 기록하게 하세요.
# [Python High-School v2.1] 정교한 예외 처리 시스템

def run_experiment(data_value):
    try:
        print(f"\n[실험 시작] 입력 데이터: {data_value}")
        
        # 1. 숫자가 아닌 경우 에러 발생
        if not isinstance(data_value, (int, float)):
            raise ValueError("데이터는 숫자여야 합니다.")
            
        # 2. 연산 시도
        process = 100 / data_value
        
    except ZeroDivisionError:
        print("🚨 사고 발생: 0으로 나눌 수 없습니다. (에러 기록 완료)")
    except ValueError as e:
        print(f"🚨 규정 위반: {e}")
    except Exception as e:
        print(f"❓ 미확인 사고: {e}")
    else:
        print(f"✅ 실험 성공: 결과값은 {process}입니다.")
    finally:
        print("🧹 시스템 유지보수: 실험 장비를 정리합니다.")

# --- 실험 케이스 ---
run_experiment(20)      # 성공 케이스
run_experiment(0)       # 0 나누기 사고
run_experiment("gogs")  # 잘못된 타입 사고

print("\n저장 필수 너는 기록이 증명이다 gogs")

💡 고등학생을 위한 '방어적 설계' 팁
 * LBYL vs EAFP:
   * LBYL (Look Before You Leap): "뛰기 전에 살펴라." if문으로 미리 체크하는 방식.
   * EAFP (Easier to Ask for Forgiveness than Permission): "허락보다 용서 구하기가 쉽다." 일단 try하고 에러가 나면 처리하는 방식. 파이썬은 EAFP 방식을 더 권장합니다.
 * 로그 기록: except 구간에서 단순히 print만 하는 게 아니라, 에러 발생 시각과 내용을 파일에 기록하는 습관이 '기록이 증명하는' 설계자의 자세입니다.
