🎓 Zig 전공 201: 2-2. 데이터베이스(SQL) 연동 및 인터페이스 설계
대학원 수준의 아키텍처에서 DB 연동은 단순히 쿼리를 던지는 것이 아니라, **'메모리 효율적인 바인딩'**과 **'커넥션 풀(Connection Pool)'**의 설계 문제입니다.
1. C 라이브러리와의 공존: SQLite 연동
Zig의 가장 큰 장점은 C로 작성된 수많은 DB 드라이버를 별도의 래퍼 없이 직접 사용할 수 있다는 점입니다. 가장 가볍고 강력한 SQLite를 예로 들어봅시다.
 * @cImport: SQLite 헤더를 직접 가져와 Zig 타입처럼 사용합니다.
 * 준비된 구문(Prepared Statements): SQL 인젝션을 방지하고 실행 속도를 높이는 설계의 정석입니다.
2. SQLite 연동 기초 코드 설계
Zig의 defer와 에러 핸들링이 DB 자원 관리에서 얼마나 빛을 발하는지 확인해 보세요.
const std = @import("std");
const c = @cImport({
    @cInclude("sqlite3.h");
});

pub fn main() !void {
    var db: ?*c.sqlite3 = null;
    
    // 1. DB 연결 (메모리 모드 또는 파일)
    if (c.sqlite3_open(":memory:", &db) != c.SQLITE_OK) {
        return error.DbOpenFailed;
    }
    defer _ = c.sqlite3_close(db);

    // 2. 테이블 생성 쿼리 실행
    const sql = "CREATE TABLE users(id INTEGER PRIMARY KEY, name TEXT);";
    var err_msg: [*c]u8 = null;
    if (c.sqlite3_exec(db, sql, null, null, &err_msg) != c.SQLITE_OK) {
        std.debug.print("SQL Error: {s}\n", .{err_msg});
        return error.QueryFailed;
    }

    std.debug.print("데이터베이스 초기화 성공!\n", .{});
}

3. 추상화 레이어: DAO/Repository 패턴
실제 대규모 프로젝트에서는 DB API를 직접 노출하지 않습니다. Zig의 struct를 활용하여 DB 로직을 캡슐화합니다.
 * 장점: 나중에 SQLite에서 PostgreSQL이나 MySQL로 교체할 때 비즈니스 로직을 수정할 필요가 없습니다.
🔬 전공 심화: 제로 카피(Zero-copy) 역직렬화
DB에서 읽어온 데이터를 Zig 구조체로 옮길 때, 메모리 복사를 최소화하는 기술입니다.
 * comptime을 활용하여 쿼리 결과의 컬럼 이름과 구조체의 필드 이름을 컴파일 시점에 매칭시키면, 실행 시점의 오버헤드를 비약적으로 줄일 수 있습니다.
📦 커넥션 풀링(Connection Pooling) 설계
네트워크 서버가 DB에 매번 연결을 맺고 끊는 것은 매우 비싼 작업입니다.
 * 미리 연결을 여러 개 만들어 두고 필요할 때 빌려 쓰는 '객체 풀(Object Pool)' 패턴을 Zig의 할당자 설계 원칙과 결합하여 구현하는 것이 201 과정의 핵심 연구 과제입니다.
