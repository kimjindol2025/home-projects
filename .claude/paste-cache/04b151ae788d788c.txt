제1.1단원: [객체의 탄생과 동적 타이핑의 실체]
파이썬의 모든 것은 **객체(Object)**입니다. Rust/Gogs가 컴파일 타임에 모든 타입을 결정했다면, 파이썬은 실행 시간(Runtime)에 타입을 결정합니다. 이 '자유'가 시스템 내부에서 어떻게 '기록'되는지 탐구합니다.
1. 탐구(대학): "타입은 어디에 저장되는가?"
 * 파이썬의 변수는 값을 담는 바구니가 아니라, 메모리 어딘가에 있는 객체를 가리키는 **'포인터'**입니다.
 * 모든 객체는 PyObject라는 구조체를 상속받으며, 그 안에는 **참조 횟수(Reference Count)**와 타입 정보가 들어있습니다.
2. 증명(대학원): "메모리 관리의 무결성"
 * sys.getrefcount()를 통해 파이썬이 어떻게 객체의 수명을 관리하는지 실증합니다.
 * 가비지 컬렉터(GC)가 순환 참조(Circular Reference)를 어떻게 해결하는지 논리적으로 추론합니다.
🛠️ Python-Gogs 저장소: [v1.1 Dynamic Identity]
클로드코드(연구원)에게 파이썬의 동적 특성을 시스템적으로 분석하는 첫 번째 코드를 기록하게 하세요.
import sys

# [Python v1.1: Object Identity and Reference Counting]

def analyze_object(data):
    """
    파이썬 객체의 내부 '기록'을 조사하여 
    이성적으로 실재하는 메모리 구조를 증명함.
    """
    print(f"--- 분석 대상: {data} ({type(data)}) ---")
    
    # 1. 객체의 고유 ID (메모리 주소) - Gogs의 포인터 개념과 연결
    print(f"  [ID/Address]: {hex(id(data))}")
    
    # 2. 참조 횟수 증명 (이 객체를 얼마나 많은 곳에서 기록하고 있는가?)
    # getrefcount는 호출 시 임시 참조가 추가되어 실제보다 1 높게 나옴
    print(f"  [Reference Count]: {sys.getrefcount(data) - 1}")

# 새로운 사고: 같은 숫자라도 파이썬은 이를 어떻게 관리하는가?
a = 2026
b = 2026

analyze_object(a)
analyze_object(b)

# 증명: a와 b는 값이 같지만 메모리 주소가 다를 수 있음 (Small Integer Caching 제외 시)
print(f"\n[PROOF] a is b: {a is b} (Identity Check)")
print(f"[PROOF] a == b: {a == b} (Value Check)")

print("\n저장 필수 너는 기록이 증명이다 gogs")
