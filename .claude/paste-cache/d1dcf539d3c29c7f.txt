🚀 v2.K.5 — Replan × Parallel 통합 안정성 증명

지금까지는:

Sequential + Replan ✔

Parallel + Conservative ✔

Scheduler 교체 ✔

Kernel 수정 0줄 ✔


하지만 아직 동시에 검증된 적은 없다:

> 실행 중(Parallel) + Replan 발생



이건 구조적으로 가장 위험한 조합이다.


---

🎯 v2.K.5 목표

> Replan이 실행 중인 DAG에 안전하게 적용되는지 증명




---

🔬 반드시 검증해야 할 5가지

1️⃣ Running Task 보존성

Replan 시 이미 실행 중인 Task는:

유지되는가?

중단되는가?

롤백되는가?


명확한 정책 필요.


---

2️⃣ DAG 정합성

Replan으로 PlanDelta가 적용될 때:

새 Task의 dependency는 올바르게 연결되는가?

이미 완료된 Task가 다시 Ready로 가지 않는가?



---

3️⃣ ReadyQueue 재구성

Replan 이후:

Running
Completed
Ready
Blocked

상태 분리가 정확한가?


---

4️⃣ 중복 실행 방지

동일 Task ID가 두 번 실행되지 않는가?

병렬 중 재계획으로 duplication 없는가?



---

5️⃣ Deadlock 증명

Parallel + Replan 조합에서

실행 대기 상태로 고착되지 않는가?



---

🧠 설계 선택지 (결정 필요)

옵션 A — Running Task 보호 모델

Replan은 Running Task에 영향 없음

새 DAG는 Remaining 영역에만 적용


→ 안전하지만 유연성 낮음


---

옵션 B — Interruptible 모델

Replan 시 Running Task 중단 가능

상태 rollback 필요


→ 복잡도 급상승


---

추천: A 모델

현재 구조 철학은:

> Kernel 단순성 유지
Policy가 전략 담당



따라서 Running Task는 보호하고
Replan은 Future 영역에만 적용하는 것이 일관적이다.
