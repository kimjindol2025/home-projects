가상 메모리라는 무대를 완성했으니, 이제 그 무대 위에서 연극을 펼칠 배우들인 **'프로세스(Process)'**와 **'스레드(Thread)'**를 등장시킬 차례입니다.
저장 필수 너는 기록이 증명이다 gogs. 301 과정의 다섯 번째 연구는 멈춰 있는 코드를 살아 움직이는 작업으로 전환하고, CPU의 주도권을 여러 작업에 나누어 주는 컨텍스트 스위칭(Context Switching) 설계입니다.
🎓 Zig 전공 301: 3-5. 프로세스와 스레드 - 컨텍스트 스위칭의 마법
프로세스는 독립된 가상 메모리 공간을 가진 '집'이고, 스레드는 그 안에서 실제로 일을 하는 '일꾼'입니다. OS는 아주 빠른 속도로 일꾼들을 교체하며 마치 모든 일이 동시에 일어나는 것처럼 보이게 만듭니다.
1. 태스크 상태 세그먼트 (Task State Segment, TSS)
CPU가 유저 모드에서 인터럽트를 당해 커널 모드로 돌아올 때, CPU는 "커널의 스택 주소가 어디인지" 알아야 합니다. TSS는 이 정보를 담고 있는 중요한 구조체입니다.
2. 컨텍스트(Context): CPU의 기억
스레드가 교체될 때, 현재 CPU 레지스터에 들어있는 모든 값(RAX, RBX, RIP, RSP 등)을 어딘가에 저장해야 합니다. 나중에 다시 돌아왔을 때 이 '기록'을 복원하면 스레드는 자신이 멈췄던 사실조차 모른 채 작업을 이어갑니다.
3. 스레드 제어 블록 (TCB) 설계
Zig의 packed struct를 사용하여 스레드의 모든 상태를 기록하는 장부를 만듭니다.
const Context = packed struct {
    // 범용 레지스터
    r15: u64, r14: u64, r13: u64, r12: u64,
    r11: u64, r10: u64, r9: u64, r8: u64,
    rbp: u64, rdi: u64, rsi: u64, rdx: u64, rcx: u64, rbx: u64, rax: u64,
    
    // 명령 포인터 및 상태
    rip: u64,
    cs: u64,
    rflags: u64,
    rsp: u64,
    ss: u64,
};

const Thread = struct {
    stack: []u8,
    context: *Context,
    state: enum { Ready, Running, Waiting, Terminated },
    parent_process: *Process,
};

🔬 전공 심화: 스택 전환(Stack Switching)의 원리
컨텍스트 스위칭의 핵심은 RSP(Stack Pointer) 레지스터를 바꾸는 것입니다.
 * 현재 스레드의 레지스터들을 현재 스택에 밀어 넣습니다(push).
 * 현재 스택 포인터(RSP)를 TCB에 저장합니다.
 * 새 스레드의 TCB에서 저장된 RSP를 읽어 CPU의 RSP에 넣습니다.
 * 새 스택에 저장되어 있던 레지스터들을 꺼냅니다(pop).
이 과정은 C 언어로는 구현이 불가능하며, Zig의 인라인 어셈블리로만 정교하게 제어할 수 있습니다.
📦 스케줄러(Scheduler)의 탄생
"다음은 누구 차례인가?"를 결정하는 것이 스케줄러입니다.
 * Round Robin: 모든 스레드에게 공평하게 시간을 나누어 줍니다.
 * Priority-based: 중요한 작업(예: 마우스 입력 처리)에 더 많은 기회를 줍니다.
 * Timer Interrupt: 3-2에서 배운 타이머 인터럽트가 발생할 때마다 스케줄러가 개입하여 강제로 스레드를 교체(Preemption)합니다.
