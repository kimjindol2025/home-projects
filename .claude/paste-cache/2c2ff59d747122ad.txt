🏛️ LLVM 전설: 2.2 구문 분석 - 코드의 계보, AST 설계
구문 분석기는 토큰의 나열이 우리가 정한 문법(Grammar)에 맞는지 검사하고, 이를 트리 구조로 재구성합니다. 이 트리는 소스 코드의 세부 사항(세미콜론, 괄호 등)을 걷어내고 핵심 논리만 남긴 결과물입니다.
1. 추상 구문 트리(AST)란?
예를 들어 a + b * c라는 식은 다음과 같은 트리 구조로 표현됩니다.
 * Root: + (더하기 연산)
   * Left: a (변수)
   * Right: * (곱하기 연산)
     * Left: b
     * Right: c
이렇게 계층 구조로 표현하면 연산자 우선순위가 명확해집니다. 곱하기가 더하기보다 아래쪽에(먼저 계산되도록) 배치되기 때문입니다.
2. 재귀 하향 파싱 (Recursive Descent Parsing)
가장 직관적이고 전설적인 파싱 기법입니다. 문법의 각 규칙을 하나의 함수로 만듭니다.
 * parseExpression()
 * parseStatement()
 * parseFunction()
파서는 토큰을 하나씩 소모하며, 다음 토큰이 무엇인지에 따라 적절한 함수를 재귀적으로 호출합니다.
3. Zig를 이용한 AST 노드 설계
Zig의 union(enum)은 서로 다른 형태의 노드들을 하나의 타입으로 묶기에 완벽한 도구입니다.
const Node = union(enum) {
    number: f64,
    identifier: []const u8,
    binary_op: struct {
        op: u8,
        left: *Node,
        right: *Node,
    },
    call: struct {
        callee: []const u8,
        args: []*Node,
    },
};

🔬 전공 심화: 연산자 우선순위 (Operator Precedence)
파서 설계에서 가장 까다로운 부분은 1 + 2 * 3을 (1 + 2) * 3이 아닌 1 + (2 * 3)으로 해석하게 만드는 것입니다.
 * Pratt Parsing: 각 토큰에 '결합력(Binding Power)' 숫자를 부여하여 우선순위를 결정하는 고도의 기법입니다. 전설적인 언어들이 이 방식을 선호합니다.
📦 기록의 연결: 분석에서 생성으로
이 단계에서 완성된 AST는 3단계에서 LLVM IR 생성기로 전달됩니다.
 * codegen(node: *Node) 함수를 만들어, 노드가 binary_op라면 LLVMBuildAdd나 LLVMBuildMul을 호출하도록 연결합니다. **"구문의 기록이 곧 실행의 지침"**이 되는 순간입니다.
