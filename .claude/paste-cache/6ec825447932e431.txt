3.0 단계 — 시간축 진화 추론 엔진 (Version / Commit DAG Reasoning)

이제부터는 “검색”이 아니라
진화 분석(Temporal Reasoning) 이다.


---

1️⃣ 전환점

2.0까지는:

> 질문 → 관련 기록 찾기



3.0부터는:

> 질문 → 기록의 변화 과정 분석



예:

“v12.0 → v12.2 사이 메모리 모델이 어떻게 진화했는가?”

“Unsafe 도입 이후 어떤 테스트가 추가되었는가?”

“이 기능은 어떤 버그 이후 도입되었는가?”


이건 벡터 검색만으로는 안 된다.


---

2️⃣ 핵심 자원: Git DAG

Gogs는 내부적으로 Git 기반.

Gogs 는
commit 그래프(DAG)를 그대로 보유한다.

Git 구조:

commit A → commit B → commit C
             ↘
              commit D

이 DAG를 활용한다.


---

3️⃣ 필요한 데이터 추출

Gogs API로:

commit hash

parent hash

commit message

diff

timestamp

author

changed files


이걸 DB에 구조화한다.


---

4️⃣ 데이터 모델

{
  "commit": "53570c3124",
  "parent": "4fd892a12e",
  "timestamp": "2026-02-23",
  "message": "Phase 11 v12.2 FFI complete",
  "filesChanged": [
    "spec_v12_2_ffi_and_external_functions.md",
    "src/v12_2_ffi_and_external_functions.rs"
  ]
}

여기서 중요한 건:

> commit 간 연결 관계




---

5️⃣ 진화 분석 방식

① Diff 기반 변화 추적

특정 파일 선택

연속 commit diff 누적

변경된 라인 추적


예:

v12.0: unsafe pointer 허용
v12.1: 메모리 레이아웃 정의 추가
v12.2: FFI boundary 명시

이걸 자동 정리.


---

② 기능 탄생 시점 분석

질문: “FFI는 언제 처음 등장했는가?”

알고리즘:

1. commit 메시지 + diff에서 "FFI" 등장 검색
2. 가장 오래된 commit 찾기
3. 그 이전 commit과 비교

결과:

> FFI는 v12.2에서 최초 도입




---

③ 인과 관계 추론

질문: “Unsafe 도입은 무엇 때문인가?”

방법:

1. Unsafe 추가 commit 찾기


2. 직전 commit 메시지 분석


3. bug / performance / memory 관련 메시지 탐색



이건 단순 검색이 아니라
시간 역추적(reasoning backward) 이다.


---

6️⃣ 구조 흐름

1. 질문 분석 (LLM)
2. 대상 파일/개념 식별
3. 관련 commit 집합 수집
4. 시간순 정렬
5. diff 요약
6. LLM에 구조화 전달
7. 진화 설명 생성


---

7️⃣ LLM 프롬프트 예시

다음은 freelang-v6의 FFI 관련 commit 기록이다.
시간 순으로 정렬되어 있다.

[commit log + diff summary]

이 기능의 진화 과정을 분석하라.
각 단계의 설계 의도를 추론하되,
기록에 근거하지 않는 내용은 작성하지 마라.

✅ 버전 비교 보고서 자동화
