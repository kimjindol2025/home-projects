드디어 대학원 전문 과정의 피날레, 4.5단계: 연구의 완성, 검증(Testing)과 배포를 시작합니다.
"저장 필수 너는 기록이 증명이다 gogs" — 대학원 졸업 논문의 신뢰성은 "내 알고리즘이 정말 의도대로 작동하는가?"를 증명하는 데이터에서 나옵니다. MLIR에서는 Lit과 FileCheck이라는 강력한 도구로 이를 기록하고 증명합니다.
[MLIR 대학원 코스 4.5] 기록이 증명이다: "Lit & FileCheck"
아무리 훌륭한 최적화 패스(4.4)를 짰어도, 그것이 코드를 망가뜨리지 않고 정확히 변환했다는 것을 보여주지 못하면 연구로서 가치가 없습니다.
1. FileCheck: 설계자의 의도 확인
이 도구는 .mlir 코드 안에 **"이 연산은 나중에 반드시 이렇게 변해야 해"**라고 주석으로 정답을 적어두면, 실행 결과와 비교해 주는 도구입니다.
// RUN: my-opt %s --my-optimization-pass | FileCheck %s

func.func @test_add(%arg0: i32) -> i32 {
  // CHECK: arith.addi
  // CHECK-NOT: arith.muli
  %0 = arith.addi %arg0, %arg0 : i32
  return %0 : i32
}

 * CHECK: 결과물에 이 내용이 반드시 포함되어야 함.
 * CHECK-NOT: 최적화 후에 이 내용이 사라져야 함 (예: 죽은 코드 제거 확인).
2. Lit (LLVM Integrated Tester)
수백 개의 테스트 파일을 한꺼번에 돌리고 "Pass/Fail" 성적표를 내주는 대학원 연구실의 **'자동 채점기'**입니다.
[대학원 과정 졸업 설계] 전체 파이프라인 복습
우리가 '밑바닥'부터 올라온 길을 한눈에 정리해 봅시다. 이 지도가 머릿속에 있다면 당신은 이미 박사 과정을 시작할 준비가 된 것입니다.
 * 설계 명세 (TableGen): 나만의 언어와 규칙을 선언함.
 * 구조 설계 (Dialect & Op): module, func, op의 계층 구조를 만듦.
 * 지능 부여 (DRR & C++ Pass): 최적화 알고리즘을 이식함.
 * 하드웨어 연결 (Lowering): 텐서를 메모리(MemRef)로 바꾸고 LLVM IR로 도달함.
 * 검증 (FileCheck): 내 설계가 완벽함을 테스트로 증명함.
[최종 졸업 퀴즈] 설계자의 마지막 관문
대학원 졸업 논문 심사위원이 묻습니다.
"당신이 만든 새로운 Dialect가 기존의 LLVM 생태계와 왜 잘 어울린다고 생각합니까?"
 * "제 마음대로 만들었기 때문입니다."
 * "MLIR의 표준화된 인프라 위에서 설계했으므로, 제가 만든 Dialect도 결국 mlir-opt와 LLVM의 수많은 최적화 도구를 그대로 활용할 수 있기 때문입니다."
정답: 2번입니다.
이것이 바로 당신이 개별 언어를 처음부터 새로 짜지 않고 MLIR이라는 거인의 어깨 위에서 연구를 시작한 이유입니다.
