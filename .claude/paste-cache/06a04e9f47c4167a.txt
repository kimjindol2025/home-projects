🔥 Design by Failure — Week 2 재설계

목표:

> 실패가 정책을 바꾸도록 만든다.




---

1️⃣ 실패 정의부터 시작

먼저 “실패”를 명확히 정의해야 한다.

Failure Class A — Latency Explosion

P99 > SLO

Scheduler queue 폭증


Failure Class B — Memory Collapse

Allocator fragmentation 급증

GC pause > threshold


Failure Class C — I/O Backpressure

queue depth 초과

syscall 대기 시간 급증


이 3개만.

확장 금지.


---

2️⃣ Trace를 ‘관찰’에서 ‘원인 체인’으로

지금 Trace는 이벤트 기록.

Week 2 목표:

> Causality Graph 생성



예:

GC pause (120ms)
  ↓
Scheduler backlog
  ↓
P99 latency 340ms
  ↓
SLO violation

이걸 자동 생성해야 한다.

단순 타임라인이 아니라:

> 원인 그래프




---

3️⃣ Metric → Policy 연결

여기서 Control Authority가 생긴다.

예시 설계:

Rule 1

IF P99 > threshold 3회 연속
→ GC aggressive mode ON

Rule 2

IF memory_pressure > 80%
→ prealloc + scheduling weight 조정

Rule 3

IF I/O queue depth > N
→ I/O bound task 우선순위 변경

중요:

정책 변경은 로그에 기록되어야 한다.

> “왜 바꿨는지”가 남아야 PostMortem 가능




---

4️⃣ PostMortem 자동화

이게 핵심이다.

단순 리포트가 아니라:

Root Cause Engine v0.1

알고리즘 구조:

1. SLO violation 감지


2. 직전 60초 Trace 수집


3. Event frequency clustering


4. 가장 높은 영향도 이벤트 계산


5. “Most Probable Cause” 출력



출력 예:

Primary Cause: GC pause spike (confidence 0.72)
Secondary Cause: I/O contention (confidence 0.21)
Policy Adjustment: Increase incremental GC frequency

여기서 처음으로

> 시스템이 자기 진단을 시작한다.




---

5️⃣ Week 2 성공 기준 (엄격)

다음 4개 통과해야 “production-hardened” 1단계 인정:

1. 실제 커널 이벤트 trace 캡처


2. Metric 기반 정책 변경 실행


3. 정책 변경 로그 기록


4. SLO 위반 시 Root Cause 자동 출력



하나라도 수동이면 실패.


---

6️⃣ 냉정한 경고

Design by Failure는 위험하다.

왜냐면:

지금 구조가 무너지기 시작할 것이다.

스케줄러/GC와 진짜로 충돌할 것이다.

Metric이 흔들릴 것이다.


하지만 그게 깊이다.


---

7️⃣ 진짜 질문

Control Authority를 어디까지 줄 것인가?

옵션:

A. Soft Control
→ 로그만 남김

B. Limited Control
→ GC / Scheduler parameter만 조정

C. Full Adaptive
→ 정책 자체 교체 가능

지금은 B가 적절하다.

C는 아직 위험하다.
