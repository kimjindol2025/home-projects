마침내 Zig 전공 301 과정의 최종장에 도달하신 것을 진심으로 축하드립니다! 부트 로더부터 시스템 호출까지, 당신은 맨바닥에서 하나의 우주(OS)를 창조하는 모든 과정을 섭렵하셨습니다.
이제 마지막 단계는 이 모든 구성 요소들을 유기적으로 결합하여, 확장성과 안정성을 극대화한 마이크로커널(Microkernel) 아키텍처로 완성하는 것입니다.
저장 필수 너는 기록이 증명이다 gogs. 301 과정의 마지막 연구는 단순한 코드가 아닌, 살아 움직이는 '생태계'의 완성입니다.
🎓 Zig 전공 301: 3-8. 최종 프로젝트 - 마이크로커널 아키텍처 완성
모놀리식 커널(Monolithic Kernel)이 모든 것을 한 몸에 담는다면, 마이크로커널은 커널의 기능을 최소화(스케줄링, IPC, 메모리 관리만 담당)하고 나머지는 유저 서비스로 밀어내는 고도의 설계 방식입니다.
1. 프로세스 간 통신 (IPC: Inter-Process Communication) 설계
마이크로커널의 심장은 **메시지 패싱(Message Passing)**입니다. 파일 시스템이나 네트워크 드라이버가 유저 공간에서 동작하므로, 프로세스끼리 데이터를 주고받는 통로가 매우 빨라야 합니다.
 * 동기식 통신: 메시지를 보낸 쪽이 응답을 받을 때까지 기다리는 방식.
 * 비동기식 통신: 메시지만 던져두고 자기 할 일을 계속하는 방식.
2. 서비스 서버(Service Servers) 구축
이제 당신의 커널 위에서 돌아갈 핵심 유저 프로세스들을 정의합니다.
 * VFS Server: 파일 접근 요청을 받아 디스크 드라이버와 통신합니다.
 * Window Server: 화면 그리기 요청을 받아 VGA 드라이버와 통신합니다.
 * Network Stack: 패킷을 처리하여 소켓 API를 제공합니다.
3. 최종 통합: 커널 부팅 시퀀스 (The Final Chain)
 * PMM/Paging 초기화: 메모리 영토 확보.
 * GDT/IDT/TSS 로드: CPU 권한 및 이벤트 체계 확립.
 * 드라이버 로드: 최소한의 하드웨어(타이머, 키보드) 제어.
 * 첫 번째 유저 프로세스(init) 실행: 시스템의 모든 서비스를 깨우는 신호탄.
🔬 전공 심화: 커널의 안정성 증명 (Verification)
마이크로커널은 서비스 하나가 죽어도 커널 전체가 멈추지 않습니다.
 * 설계: 드라이버가 크래시(Crash) 나면 커널이 이를 감지하고 해당 프로세스만 다시 시작(Restart)하게 만듭니다. 이것이 **"기록이 증명하는 회복 탄력성"**입니다.
📦 배포: 배포 가능한 디스크 이미지(ISO) 제작
Zig의 build.zig를 확장하여, 컴파일된 커널과 init 프로세스, 그리고 파일 시스템 데이터를 하나의 .iso 파일로 묶는 자동화 공정을 구축합니다. 이제 이 이미지는 QEMU 같은 에뮬레이터나 실제 PC에서 부팅될 준비가 끝납니다.
// build.zig의 예시 개념
const iso_cmd = b.addSystemCommand(&.{
    "grub-mkrescue", "-o", "myos.iso", "isodir",
});

📝 최종 졸업 과제 (The Grand Finale)
 * IPC 구현: 두 유저 프로세스가 시스템 호출을 통해 메시지를 주고받는 기능을 완성하세요.
 * 커스텀 쉘(Shell): 사용자가 명령어를 입력하면 파일을 실행하거나 화면에 글자를 출력하는 최소한의 쉘 프로세스를 만드세요.
 * 멀티태스킹 증명: 배경에서 타이머가 돌아가고, 전경에서는 쉘이 돌아가는 '동시성'을 구현하세요.
 * 최종 기록: 이 과정을 통해 당신이 정의한 **"나만의 OS 철학"**을 한 페이지 분량으로 정리하여 기록하세요.
