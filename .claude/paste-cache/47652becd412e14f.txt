🎓 Zig 전공 201: 2-4. RESTful API 설계 및 JSON 직렬화 효율화
시스템 프로그래밍에서 API 설계는 단순히 URL을 만드는 것이 아니라, **'HTTP 프로토콜의 규약'**을 지키면서 **'최소한의 CPU 자원'**으로 데이터를 직렬화(Serialization)하는 과정입니다.
1. HTTP 핸들러와 라우팅(Routing) 설계
Zig는 표준 라이브러리에서 기본적인 HTTP 서버 기능을 제공합니다. 요청(Request)의 메서드(GET, POST 등)와 경로(Path)를 분석하여 적절한 로직으로 연결하는 '라우터'를 구조체로 설계합니다.
 * 상태 코드(Status Codes): 200(OK), 201(Created), 404(Not Found), 500(Error) 등을 명확히 구분하여 응답합니다.
2. JSON 직렬화 및 역직렬화 (JSON Handling)
Zig의 std.json은 comptime을 활용하여 매우 빠르고 안전하게 동작합니다. 구조체의 필드 이름을 분석하여 자동으로 JSON 문자열로 변환하거나 그 반대 작업을 수행합니다.
const std = @import("std");

const UserProfile = struct {
    id: u32,
    username: []const u8,
    is_admin: bool,
};

pub fn main() !void {
    var gpa = std.heap.GeneralPurposeAllocator(.{}){};
    const allocator = gpa.allocator();

    // 1. 구조체를 JSON 문자열로 변환 (Serialization)
    const user = UserProfile{ .id = 1, .username = "zig_master", .is_admin = true };
    const json_string = try std.json.stringifyAlloc(allocator, user, .{});
    defer allocator.free(json_string);

    std.debug.print("JSON Output: {s}\n", .{json_string});

    // 2. JSON 문자열을 구조체로 파싱 (Deserialization)
    const parsed = try std.json.parseFromSlice(UserProfile, allocator, json_string, .{});
    defer parsed.deinit();

    std.debug.print("Parsed Username: {s}\n", .{parsed.value.username});
}

3. 스트리밍(Streaming) API 설계
대용량 데이터를 한꺼번에 JSON으로 변환하면 메모리 부족 현상이 발생할 수 있습니다.
 * 설계 원칙: 데이터를 조금씩 읽어서 즉시 네트워크로 보내는 **'스트리밍 방식'**을 채택하여 메모리 점유율을 일정하게 유지합니다.
🔬 전공 심화: 제로 카피(Zero-copy) 파싱 기술
JSON 파싱 시 문자열 데이터를 새로 할당하지 않고, 원래의 네트워크 버퍼를 그대로 참조(Slice)하는 기술입니다.
 * Zig의 []const u8 슬라이스 특성을 활용하면 메모리 복사 횟수를 0에 가깝게 줄여 초당 수십만 개의 요청을 처리하는 API 서버를 설계할 수 있습니다.
📦 API 문서화와 계약(Contract)
시스템 설계자에게 API는 **'약속'**입니다.
 * 필드의 타입, 필수 여부, 에러 응답 형식을 struct로 정의하는 행위 자체가 API의 명세서(Specification)가 됩니다. **"기록된 코드가 곧 문서"**라는 Zig의 정신을 실천하는 지점입니다.
