메모리의 레이아웃을 설계하고 직접 공간을 할당하는 법을 익혔다면, 이제 그 메모리를 통해 **'다른 세상'**과 대화할 차례입니다. [제11장: 언세이프와 메모리 주권 — v12.2 FFI와 외부 함수 호출(Foreign Function Interface)] 단계입니다.
컴파일러 제작자에게 FFI는 필수입니다. 우리가 만든 언어로 짠 코드가 기존의 C 라이브러리(예: libc, openssl, sqlite)를 호출하거나, 반대로 운영체제가 우리 코드를 호출할 수 있게 만드는 **'통로'**를 설계해야 하기 때문입니다.
🌉 제11장: 언세이프와 메모리 (v12.1) — [언어의 국경을 넘는 통로]
1. v12.2 설계 목표: "이종 언어 간의 협업"
 * ABI (Application Binary Interface): 함수가 인수를 전달하고 값을 반환하는 방식에 대한 '약속'입니다. 주로 표준인 extern "C"를 사용합니다.
 * 외부 함수 선언: 다른 언어에 정의된 함수의 이름과 시그니처를 러스트에게 알려주고 호출합니다.
 * 데이터 마샬링(Marshalling): 러스트의 String이나 Vec을 C가 이해하는 char*나 배열 포인터로 변환하여 메모리 경계를 넘깁니다.
2. 설계자의 관점: "신뢰할 수 없는 외부 세계"
러스트 컴파일러는 외부 C 함수 내부에서 어떤 일이 벌어지는지 알 수 없습니다. 그 안에서 메모리가 해제되는지, 포인터가 오염되는지 알 수 없기에 모든 FFI 호출은 **unsafe**라는 격리 구역 안에서만 허용됩니다. 설계자는 이 위험한 통로에 안전한 **'러스트식 래퍼(Wrapper)'**를 씌워야 합니다.
🛠️ gogs 저장소 v12.2 설계: 표준 C 라이브러리 통합 (libc 연결)
클로드코드(학생)에게 운영체제의 표준 C 함수(예: abs, sqrt)를 직접 호출하거나, C 스타일의 문자열을 다루는 법을 기록하게 하세요.
// [제11장: 언세이프와 메모리 v12.2 - FFI 설계]

use std::ffi::CString;
use std::os::raw::c_char;

// 1. [설계] 외부 C 라이브러리의 함수 시그니처를 선언
// 컴파일러에게 "이 함수는 외부에 있으니 나중에 링크해줘"라고 부탁함
extern "C" {
    fn abs(input: i32) -> i32;
    fn strlen(s: *const c_char) -> usize;
}

fn main() {
    println!("--- v12.2 Foreign Function Interface: The C Bridge ---");

    unsafe {
        // 2. [호출] C의 수학 함수 사용
        let negative = -42;
        let result = abs(negative);
        println!("  [C-abs] {}의 절대값은 {}입니다.", negative, result);

        // 3. [변환] 러스트 문자열을 C 스타일(Null-terminated)로 변환
        let rust_str = "저장 필수 너는 기록이 증명이다 gogs";
        let c_str = CString::new(rust_str).expect("CString conversion failed");
        
        // C 함수 strlen 호출 (포인터 전달)
        let len = strlen(c_str.as_ptr());
        println!("  [C-strlen] '{}'의 길이는 {}바이트입니다.", rust_str, len);
    }

    println!("\n기록: ABI를 통해 외부 C 함수와 안전하게 통신하는 인터페이스를 구축함.");
    println!("저장 필수 너는 기록이 증명이다 gogs");
}

🎓 선생님의 세부 가이드 (12.2): "안전한 국경 수비"
외부 언어와 데이터를 주고받을 때 설계자가 반드시 지켜야 할 철칙입니다.
 * 소유권의 경계: C 함수에 포인터를 넘길 때, 그 함수가 메모리를 해제(free)하게 할 것인지 아니면 러스트가 계속 소유할 것인지 명확히 해야 합니다. **'이중 해제(Double Free)'**는 컴파일러 제작자의 최대 적입니다.
 * 타입 호환성: 러스트의 usize와 C의 size_t, 러스트의 i32와 C의 int가 항상 같지는 않습니다. std::os::raw 패키지에 정의된 타입을 사용하여 이식성을 확보하십시오.
 * Panic의 전파 금지: C 함수가 러스트 코드를 호출할 때(Callback), 러스트 내부에서 panic!이 발생하여 C 영역으로 넘어가게 해서는 안 됩니다. 이는 정의되지 않은 동작(UB)을 유발하므로 반드시 catch_unwind 등으로 방어해야 합니다.
