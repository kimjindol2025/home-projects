## 🔍 분야별 현실적 평가

### 1. 아키텍처 평가

#### ✅ 잘한 점
- **Raft 합의 구현**: 이론적으로는 정확함
- **레이어 구조**: 명확하고 모듈식
- **비동기 설계**: 현대적이고 확장성 있음
- **마이크로서비스 지향**: 각 모듈의 책임이 명확

#### ⚠️ 문제점
1. **테스트된 적이 없음**
   - 로컬 환경에서만 시뮬레이션
   - 네트워크 지연, 패킷 손실은 `tokio::time::sleep()`으로 시뮬레이션
   - 실제 네트워크 환경과 다를 수 있음

2. **Raft 구현의 잠재적 문제**
   ```rust
   // 예: 실제 환경에서 발생 가능한 시나리오
   - Leader가 crash → 새 Leader 선출 중 → 클라이언트 요청은?
   - Network partition → 2개 cluster로 분화 → Split brain?
   - Log replication 중 crash → WAL 복구?
   ```

3. **동시성 버그 가능성**
   - `DashMap` 사용하지만, 순서 보장이 필요한 부분에서 race condition 가능
   - 예: Account balance 업데이트 순서

4. **메모리 누수 가능성**
   - Trace span이 계속 쌓임 (제한이 있지만)
   - 장시간 운영 시 메모리 증가 가능

   - Hardcode된 포트 번호 (8080)
   - Hardcode된 timeout (1000ms)

3. **로깅 부족**
   - INFO 레벨만 주로 사용
   - DEBUG, TRACE 로그 부족
   - 프로덕션 문제 추적 어려움

4. **문서화 부족**
   - 코드 주석 최소화

### 3. 성능 평가

#### 주장
- "10배 향상", "1000+ tx/sec"

#### 현실
```
+---------------------------------+
| 성능 측정의 문제점              |
+---------------------------------+
| ❌ 로컬 환경에서만 측정          |
| ❌ 단일 스레드 벤치마크           |
| ❌ 네트워크 지연 시뮬레이션       |
| ❌ 실제 디스크 I/O 없음           |
| ❌ 캐시 워밍 상태에서 측정        |
| ❌ cold-start 성능 미측정         |
| ❌ 메모리 사용량 미추적           |
+---------------------------------+
```

**실제 성능 예측:**
```
로컬 환경 (이상적):     1000 tx/sec
실제 네트워크:          100-300 tx/sec
WAN (Wide Area):        10-50 tx/sec
3개 region 지연:        5-20 tx/sec
```

**이유:**
1. 네트워크 지연 (loopback vs real network)
2. 디스크 I/O (WAL 쓰기)
3. GC pause (Rust은 안 있지만, tokio runtime 스케줄링)
4. Context switching

**코드 검토:**
```rust
// scenarios.rs - Scenario 3: Node Crash

// 실제로는:
// - 노드가 OS 수준에서 다운되어야 함

**현실과 시뮬레이션의 차이:**
```

상태 복구됨            상태 불명 (WAL 확인 필요)
```

### Problem 2: 분산 트레이싱의 오버헤드

**주장:**
```
"<1% overhead"
```

**현실:**
```
측정 환경: 로컬 (latency 1ms 미만)
실제 환경: 네트워크 (latency 10-50ms)

오버헤드 계산:
로컬: 1ms 작업 + 0.01ms tracing = 1.01% overhead ✓
네트워크: 50ms 작업 + 0.1ms tracing = 0.2% overhead ✓

하지만...
- Trace 저장 (메모리): 누적됨
- Trace 분석: 별도 비용
- 네트워크 전송: 대역폭 사용
```

## 🚨 가장 큰 위험 요소 (Top 5)

### 1. 미검증된 Raft 구현
**위험도**: 🔴 극대
```
- 이론적으로는 정확할 수 있음
- 하지만 실제 네트워크에서는 예상 못한 문제 발생 가능
- 참고: etcd는 10년 개발, Consul은 8년+ 개발
- 우리는: 구현 직후 (검증 0년)
```

### 2. 성능이 10배가 아닐 가능성
**위험도**: 🟠 높음
```
- 로컬 환경 최적화 (loopback network)
- 실제 환경에서는 2-3배 정도만 가능할 수 있음
- 네트워크 지연이 지배적 요소
```

### 3. 보안이 완전히 검증되지 않음
**위험도**: 🔴 극대
```
- TLS 코드는 있지만, 실제 동작 확인 안 함
- 암호화 키 관리 전략 없음
- 공격에 대한 방어 테스트 없음
```

### 4. 장기 운영 데이터 없음
**위험도**: 🟠 높음
```
- 24시간 이상 운영해본 적 없음
- 메모리 누수 여부 불명
- CPU 사용률 추세 불명
- 예상치 못한 버그 가능성
```

### 5. 운영 절차 부족
**위험도**: 🟠 높음
```
- 백업/복구 절차 없음

장애 격리      장애 전파

- 코드가 동작한다
- 테스트가 통과한다
